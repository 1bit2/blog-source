# 进程内核结构关系

> 基于 Linux 5.15.78，梳理 `task_struct` 与内核各子系统的关联结构。
> 配套 drawio 图：`docs/mini-docker/process-kernel-structures.drawio`

---

## 一、task_struct 核心关联字段

`struct task_struct`（`include/linux/sched.h:723`）是进程描述符，通过指针关联内核各子系统：

| 字段 | 类型 | 行号 | 关联子系统 |
|------|------|------|-----------|
| `mm` | `struct mm_struct *` | :857 | 虚拟内存管理 |
| `fs` | `struct fs_struct *` | :1067 | 根目录/当前工作目录 |
| `files` | `struct files_struct *` | :1070 | 文件描述符表 |
| `nsproxy` | `struct nsproxy *` | :1077 | 命名空间代理 |
| `signal` | `struct signal_struct *` | :1080 | 信号/资源限制 |
| `cgroups` | `struct css_set __rcu *` | :1205 | Cgroup 关联集 |

---

## 二、各子结构详解

### 2.1 mm_struct — 虚拟内存

`include/linux/mm_types.h`，管理进程的虚拟地址空间：

```
mm_struct
  ├── mmap (struct vm_area_struct *)  — VMA 链表头 :1067
  ├── mm_rb (struct rb_root)          — VMA 红黑树 :1080
  ├── mmap_lock (struct rw_semaphore) — 保护 VMA
  ├── pgd (pgd_t *)                  — 页全局目录
  └── start_code/end_code/start_brk/brk/start_stack — 地址空间边界
```

每个 `vm_area_struct` 描述一段连续的虚拟地址映射（代码段、数据段、堆、栈、mmap 区域等）。

### 2.2 fs_struct — 文件系统根目录

`include/linux/fs_struct.h:9`：

```c
struct fs_struct {
    int users;
    spinlock_t lock;
    seqcount_spinlock_t seq;
    int umask;
    int in_exec;
    struct path root, pwd;  // 根目录和当前工作目录
};
```

`struct path`（`include/linux/path.h:8`）= `vfsmount *mnt` + `dentry *dentry`。
`pivot_root()` 修改的就是 `fs->root`。

### 2.3 files_struct — 文件描述符表

`include/linux/fdtable.h:49`：

```
files_struct
  └── fdt (struct fdtable __rcu *)
        └── fd[] (struct file __rcu **)  — fd[0]=stdin, fd[1]=stdout, ...
              └── struct file
                    ├── f_path (struct path)
                    ├── f_inode (struct inode *)
                    ├── f_op (const struct file_operations *)
                    └── f_pos (loff_t)
```

### 2.4 nsproxy — 命名空间代理

`include/linux/nsproxy.h:31`：

```c
struct nsproxy {
    atomic_t count;
    struct uts_namespace  *uts_ns;
    struct ipc_namespace  *ipc_ns;
    struct mnt_namespace  *mnt_ns;
    struct pid_namespace  *pid_ns_for_children;
    struct net            *net_ns;
    struct time_namespace *time_ns;
    struct time_namespace *time_ns_for_children;
    struct cgroup_namespace *cgroup_ns;
};
```

nsproxy 是代理模式：不同隔离域的进程持有不同的 nsproxy 实例，每次系统调用时内核通过 `current->nsproxy` 确定进程所在的"世界"。

### 2.5 css_set — Cgroup 关联集

`include/linux/cgroup-defs.h:199`：

```
css_set
  └── subsys[CGROUP_SUBSYS_COUNT] (struct cgroup_subsys_state *[])
        ├── [memory]  — 内存限制
        ├── [cpu]     — CPU 时间配额
        ├── [pids]    — 进程数限制
        ├── [devices] — 设备访问白名单
        ├── [cpuset]  — CPU/内存节点绑定
        └── ...
```

## 三、与容器 (mini-docker) 的对应关系

| 容器操作 | 内核系统调用 | 影响的 task_struct 字段 |
|---------|------------|----------------------|
| `clone(CLONE_NEW*)` | `copy_namespaces()` | `nsproxy` — 创建新的命名空间代理 |
| `setup_cgroup()` | 写 `/sys/fs/cgroup/` | `cgroups` — 关联新的 css_set |
| `pivot_root()` | `SYS_pivot_root` | `fs` — 修改进程的 root/pwd |
| `execvp()` | `do_execve()` | `mm` — 替换为新程序的地址空间（保留 nsproxy/cgroups/fs） |

---

## 四、完整结构关系树

```
task_struct (include/linux/sched.h:723)
│
├── mm → mm_struct
│       └── mmap / mm_rb → vm_area_struct → vm_file → struct file
│
├── fs → fs_struct
│       └── root, pwd → struct path (vfsmount + dentry)
│
├── files → files_struct
│           └── fdt → fdtable → fd[0,1,2...] → struct file
│
├── nsproxy → nsproxy
│             ├── uts_ns      — 独立 hostname
│             ├── ipc_ns      — 独立 IPC 对象
│             ├── mnt_ns      — 独立挂载表
│             ├── pid_ns_for_children — 独立 PID 空间
│             ├── net_ns      — 独立网络栈
│             ├── time_ns     — 独立时钟
│             └── cgroup_ns   — 独立 cgroup 视图
│
├── cgroups → css_set
│             └── subsys[] → memory / cpu / pids / devices / cpuset ...
│
└── signal → signal_struct
              └── rlim[] — 资源限制 / tty — 控制终端
```
