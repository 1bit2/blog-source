## 零、30 秒理解调度器

**一句话**：调度器决定"哪个进程占用 CPU"——它不是一直在运行的守护进程，而是被特定事件**触发**时才执行的一段代码。

**类比**：想象一个只有一个窗口的银行（= 一个 CPU 核心）。客户（= 进程）排队等候。柜员（= CPU）正在服务当前客户。调度器就是那个**叫号系统**——它不会随时打断柜员，而是在以下时刻才响：

1. 当前客户主动说"我要等取款机回复"（**主动睡眠**）
2. 计时器响了：当前客户已占用太久（**时间片到期**）
3. VIP 客户走进来（**高优先级唤醒**）
4. 柜员处理完一个中断事务后（**中断返回**）

**为什么这样设计？** 直接在中断上下文中调用 `schedule()` 会带来死锁风险（持锁时被抢占）、栈嵌套溢出等问题。标记-检查分离让内核可以在**安全点**（`preempt_count == 0`、返回用户态边界）才执行调度，同时保证响应延迟可控。

## 二、唯一的切换入口：__schedule()

整个内核中，**真正做上下文切换的只有 `__schedule()` 一个函数**，所有路径最终都汇聚于此。

```c
// kernel/sched/core.c:6291
static void __sched notrace __schedule(unsigned int sched_mode)
{
    cpu = smp_processor_id();
    rq = cpu_rq(cpu);
    prev = rq->curr;

    local_irq_disable();
    rq_lock(rq, &rf);
    update_rq_clock(rq);

    // 非抢占 + 任务要睡 → 出队
    prev_state = READ_ONCE(prev->__state);
    if (!(sched_mode & SM_MASK_PREEMPT) && prev_state) {
        if (signal_pending_state(prev_state, prev)) {
            WRITE_ONCE(prev->__state, TASK_RUNNING);
        } else {
            deactivate_task(rq, prev, DEQUEUE_SLEEP | DEQUEUE_NOCLOCK);
        }
    }

    next = pick_next_task(rq, prev, &rf);
    clear_tsk_need_resched(prev);
    clear_preempt_need_resched();

    if (likely(prev != next)) {
        rq = context_switch(rq, prev, next, &rf);
    }
}
```

### sched_mode 语义

| 模式 | 值 | 含义 |
|------|-----|------|
| `SM_NONE` | 0x0 | 主动睡眠：若 `prev->__state != 0` 则出队 |
| `SM_PREEMPT` | 0x1 | 抢占：不出队当前任务（仍 RUNNING） |
| `SM_RTLOCK_WAIT` | 0x2 | RT 锁等待 |

### schedule() 包装

```c
// kernel/sched/core.c:6489
asmlinkage __visible void __sched schedule(void)
{
    struct task_struct *tsk = current;
    sched_submit_work(tsk);      // 处理 workqueue/blk plug
    do {
        preempt_disable();
        __schedule(SM_NONE);
        sched_preempt_enable_no_resched();
    } while (need_resched());    // 循环直到标志清除
    sched_update_worker(tsk);
}
```

### A. 主动调度：进程自己调用 schedule()

进程因资源不可用而主动放弃 CPU。

| 场景 | 典型代码 |
|------|----------|
| mutex 获取失败 | `set_current_state(TASK_UNINTERRUPTIBLE); schedule();` |
| 等待 I/O | `io_schedule()` |
| 定时睡眠 | `schedule_timeout()` |
| 让出 CPU | `sched_yield()` → `do_sched_yield()` → `schedule()` |
| 线程退出 | `do_task_dead()` → `__schedule(SM_NONE)` |

```c
// kernel/sched/core.c:8267
static void do_sched_yield(void)
{
    current->sched_class->yield_task(rq);
    schedule();
}
```

### B. 内核抢占：preempt_enable() / spin_unlock()

当 `preempt_count` 递减归零且 `need_resched` 已设置时，触发抢占。

```c
// include/linux/preempt.h:196
#define preempt_enable() \
do { \
    barrier(); \
    if (unlikely(preempt_count_dec_and_test())) \
        __preempt_schedule(); \
} while (0)
```

`__preempt_schedule()` 最终调用：

```c
// kernel/sched/core.c:6571
static void __sched preempt_schedule_common(void)
{
    do {
        preempt_disable_notrace();
        __schedule(SM_PREEMPT);
        preempt_enable_no_resched_notrace();
    } while (need_resched());
}
```

### C. 中断返回内核态（CONFIG_PREEMPTION=y）

中断处理完毕，返回内核态时检查：

```c
// kernel/entry/common.c:383
void irqentry_exit_cond_resched(void)
{
    if (!preempt_count()) {
        rcu_irq_exit_check_preempt();
        if (need_resched())
            preempt_schedule_irq();
    }
}
```

```c
// kernel/sched/core.c:6805
asmlinkage __visible void __sched preempt_schedule_irq(void)
{
    do {
        preempt_disable();
        local_irq_enable();
        __schedule(SM_PREEMPT);
        local_irq_disable();
        sched_preempt_enable_no_resched();
    } while (need_resched());
}
```

### D. 返回用户态（系统调用/中断返回）

无论内核是否配置 `CONFIG_PREEMPTION`，返回用户态时都会检查：

```c
// kernel/entry/common.c:151
static unsigned long exit_to_user_mode_loop(struct pt_regs *regs,
                        unsigned long ti_work)
{
    while (ti_work & EXIT_TO_USER_MODE_WORK) {
        local_irq_enable_exit_to_user(ti_work);

        if (ti_work & _TIF_NEED_RESCHED)
            schedule();

        // signal、uprobe、livepatch 等处理 ...
        ti_work = READ_ONCE(current_thread_info()->flags);
    }
    return ti_work;
}
```

调用路径：
```
syscall_exit_to_user_mode() → exit_to_user_mode_prepare()
  → exit_to_user_mode_loop()

irqentry_exit() → irqentry_exit_to_user_mode()
  → exit_to_user_mode_prepare()
    → exit_to_user_mode_loop()
```

### E. 协作式调度：cond_resched()

非抢占内核（`CONFIG_PREEMPTION=n`）中，长循环必须调用 `cond_resched()` 给其他任务执行机会：

```c
// kernel/sched/core.c:8299
int __sched __cond_resched(void)
{
    if (should_resched(0)) {
        preempt_schedule_common();
        return 1;
    }
    return 0;
}
```

## 四、need_resched 双层标志机制

### 层1：TIF_NEED_RESCHED（thread_info.flags）

通用标志，所有架构共用：

```c
// include/linux/sched.h:2001
static inline void set_tsk_need_resched(struct task_struct *tsk)
{
    set_tsk_thread_flag(tsk, TIF_NEED_RESCHED);
}

// include/linux/sched.h:2113
static __always_inline bool need_resched(void)
{
    return unlikely(tif_need_resched());
}
```

### 层2：折叠进 preempt_count（x86 优化）

x86 将 `NEED_RESCHED` 反向编码到 `preempt_count` 中，使 `preempt_enable()` 递减后能**单条指令**同时检测"计数归零"+"需要调度"：

```c
// arch/x86/include/asm/preempt.h:50
static __always_inline void set_preempt_need_resched(void)
{
    raw_cpu_and_4(__preempt_count, ~PREEMPT_NEED_RESCHED);
}

static __always_inline bool should_resched(int preempt_offset)
{
    return unlikely(raw_cpu_read_4(__preempt_count) == preempt_offset);
}
```

当 `PREEMPT_NEED_RESCHED` 位被清零（注意是反向逻辑），且 `preempt_count` 正好等于 `preempt_offset`（通常为 0），则 `should_resched` 为 true。

### 核心设置函数：resched_curr()

```c
// kernel/sched/core.c:962
void resched_curr(struct rq *rq)
{
    struct task_struct *curr = rq->curr;

    if (test_tsk_need_resched(curr))
        return;

    if (cpu == smp_processor_id()) {
        set_tsk_need_resched(curr);
        set_preempt_need_resched();
        return;
    }

    // 远程 CPU：设标志 + 发 IPI
    if (set_nr_and_not_polling(curr))
        smp_send_reschedule(cpu);
}
```

| 场景 | 行为 |
|------|------|
| 本 CPU | 设 `TIF_NEED_RESCHED` + 折叠到 `preempt_count` |
| 远程 CPU（非 idle polling） | 发 reschedule IPI |
| 远程 CPU（idle polling） | 只设标志，idle 循环自己检查 |

### 清除时机

在 `__schedule()` 中选出 `next` 后立即清除：
```c
// kernel/sched/core.c:6378
clear_tsk_need_resched(prev);
clear_preempt_need_resched();
```

### 触发链

```
时钟中断
  → tick_periodic() / tick_sched_handle()
    → update_process_times()           [kernel/time/timer.c:1779]
      → scheduler_tick()               [kernel/sched/core.c:5320]
```

### scheduler_tick() 本体

```c
// kernel/sched/core.c:5320
void scheduler_tick(void)
{
    int cpu = smp_processor_id();
    struct rq *rq = cpu_rq(cpu);
    struct task_struct *curr = rq->curr;

    rq_lock(rq, &rf);
    update_rq_clock(rq);
    curr->sched_class->task_tick(rq, curr, 0);  // 调度类回调
    calc_global_load_tick(rq);
    rq_unlock(rq, &rf);

    trigger_load_balance(rq);  // SMP 负载均衡检查
}
```

**关键：`scheduler_tick` 从不直接调用 `schedule()`**。它只通过调度类回调间接设置 `TIF_NEED_RESCHED`。

### CFS 回调链：task_tick_fair → check_preempt_tick

```c
// kernel/sched/fair.c:4609
static void entity_tick(struct cfs_rq *cfs_rq, struct sched_entity *curr, int queued)
{
    update_curr(cfs_rq);
    if (cfs_rq->nr_running > 1)
        check_preempt_tick(cfs_rq, curr);
}
```

```c
// kernel/sched/fair.c:4452
static void check_preempt_tick(struct cfs_rq *cfs_rq, struct sched_entity *curr)
{
    unsigned long ideal_runtime = sched_slice(cfs_rq, curr);
    unsigned long delta_exec = curr->sum_exec_runtime - curr->prev_sum_exec_runtime;

    if (delta_exec > ideal_runtime) {
        resched_curr(rq_of(cfs_rq));  // 时间片耗尽 → 标记需要调度
        return;
    }

    // 最小粒度保护：即使份额未用完，若 vruntime 差值过大也抢占
    struct sched_entity *se = __pick_first_entity(cfs_rq);
    s64 delta = curr->vruntime - se->vruntime;
    if (delta > ideal_runtime)
        resched_curr(rq_of(cfs_rq));
}
```

设计要点：
- `sched_slice()` 根据权重和队列中的实体数动态计算每个任务的"理想运行时间"
- 不是固定时间片，而是**按比例公平分配**
- 设置标志后，切换发生在中断返回（路径 C/D）

## 六、唤醒抢占：高优先级任务如何抢占

### 唤醒路径

```
wake_up_process()
  → try_to_wake_up()
    → ttwu_do_wakeup()
      → check_preempt_curr(rq, p, wake_flags)
        → rq->curr->sched_class->check_preempt_curr(rq, p, wake_flags)
```

### CFS 唤醒抢占检查

```c
// kernel/sched/fair.c:7155
static void check_preempt_wakeup(struct rq *rq, struct task_struct *p, int wake_flags)
{
    struct sched_entity *se = &curr->se, *pse = &p->se;

    // 如果 buddy 提示 + 同组 → 跳过抢占
    if (wakeup_preempt_entity(se, pse) == 1) {
        goto preempt;
    }
    return;

preempt:
    resched_curr(rq);
}
```

`wakeup_preempt_entity()` 比较两个调度实体的 vruntime，差值超过一个调度粒度则允许抢占。

### 跨核唤醒

如果被唤醒任务应该抢占远程 CPU 上的当前任务：
1. `resched_curr(rq)` 检测到 `cpu != smp_processor_id()`
2. 发送 reschedule IPI
3. 远程 CPU 收到 IPI → 中断返回时检查 `need_resched` → `preempt_schedule_irq()`

## 八、设计思考

### 1. 为什么 scheduler_tick 不直接 schedule()？

- **scheduler_tick 在中断上下文中执行**，此时当前任务可能持有各种锁
- 直接 schedule 会导致持锁进程被切走 → 死锁
- 标记-延迟模型保证切换只发生在安全点

### 2. 为什么 x86 要折叠 NEED_RESCHED 到 preempt_count？

传统检查需要：
```c
preempt_count--;          // 指令1
if (preempt_count == 0)   // 指令2
    if (need_resched())   // 指令3：读另一个变量
```

折叠后：
```c
// 单次递减+比较 = 一条 dec+jz 指令
if (--preempt_count == 0)  // 同时满足"计数归零"+"需要调度"
```

在抢占频繁的热路径（spin_unlock 等），节省的指令累积可观。

### 3. idle 进程的特殊处理

idle 进程在 `cpu_idle_loop()` 中 polling `need_resched`：

```c
// kernel/sched/idle.c:282
while (!need_resched()) {
    // cpuidle / halt / poll ...
}
schedule_idle();
```

`resched_curr()` 对 polling idle 不发 IPI（`set_nr_and_not_polling` 返回 false），因为 idle 自己会快速检测到标志变化，省去一次跨核中断。

### 4. PREEMPT_NONE vs PREEMPT_VOLUNTARY vs PREEMPT

| 配置 | 调度延迟 | 吞吐量 | 切换点 |
|------|----------|--------|--------|
| `PREEMPT_NONE` | 高（ms 级） | 最高 | 仅返回用户态 |
| `PREEMPT_VOLUNTARY` | 中等 | 较高 | 返回用户态 + cond_resched() |
| `PREEMPT` | 低（μs 级） | 较低 | 任何 preempt_count==0 的点 |

服务器通常用 `PREEMPT_NONE`/`VOLUNTARY`（重吞吐），桌面/嵌入式用 `PREEMPT`（重延迟）。
