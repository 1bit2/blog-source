# 系统调用机制

> 基于 Linux 5.15.78，分析用户态程序如何进入内核态执行特权操作：从 `syscall` 指令到 `sys_call_table` 分派，以及返回时的调度检查点。
>
> 前置阅读：[上下文切换实现原理](上下文切换实现原理.md) — 内核栈与寄存器保存恢复

---

## 一、两种系统调用方式

### 1.1 老方式：`int 0x80`（软件中断）

早期 x86 32 位通过软件中断进入内核：

```asm
; 用户态程序
mov eax, 1        ; 系统调用号 (1 = write)
mov ebx, 1        ; 参数1: fd = stdout
mov ecx, msg      ; 参数2: 缓冲区地址
mov edx, len      ; 参数3: 长度
int 0x80          ; 触发软中断 → CPU 查 IDT → 跳转到内核
```

CPU 收到 `int 0x80` 后执行完整的中断处理流程：查中断描述符表（IDT）→ 保存寄存器到内核栈 → 跳转到 `entry_INT80_compat`。整个过程约 **400 个 CPU 周期**。

### 1.2 新方式：`syscall`（专用指令）

x86_64 使用 CPU 提供的快速通道：

```asm
; 用户态程序 (x86_64 参数通过寄存器传递)
mov rax, 1        ; 系统调用号
mov rdi, 1        ; 参数1: fd
mov rsi, msg      ; 参数2: buf
mov rdx, len      ; 参数3: len
syscall           ; 专用指令 → CPU 直接跳转到 MSR_LSTAR 指向的地址
```

`syscall` 指令由 CPU 硬件直接处理，不查 IDT，不走中断流程，约 **100 个周期**。

### 1.3 对比

| | `int 0x80`（老） | `syscall`（新） |
|---|---|---|
| **本质** | 软件中断 | 专用 CPU 指令 |
| **速度** | ~400 周期 | ~100 周期 |
| **入口查找** | 查 IDT 表 | 读 MSR_LSTAR 寄存器 |
| **切栈** | IDT 中描述的内核栈 | TSS 中预设的内核栈 |
| **返回指令** | `iret` | `sysret` |
| **参数传递** | 栈 / eax+ebx+ecx+edx | rax + rdi,rsi,rdx,r10,r8,r9 |
| **x86_64 默认** | 否（兼容保留） | 是 |

## 二、内核初始化：注册 syscall 入口

内核启动时，将 `entry_SYSCALL_64` 的地址写入 CPU 的 `MSR_LSTAR` 寄存器。此后每次用户态执行 `syscall` 指令，CPU 就跳转到这个地址：

```c
// arch/x86/kernel/cpu/common.c:1874
wrmsrl(MSR_LSTAR, (unsigned long)entry_SYSCALL_64);
```

同时启用 EFER.SCE 位（System Call Enable），允许 `syscall`/`sysret` 指令执行：

```asm
// arch/x86/kernel/head_64.S (启动代码)
movl    $MSR_EFER, %ecx
rdmsr
btsl    $_EFER_SCE, %eax      ; 启用 syscall/sysret
wrmsr
```

## 三、完整流程：从 syscall 到 sysret

### 3.1 调用链总览

```
用户态: syscall 指令
  ↓ CPU 硬件自动完成:
  │  1. 读 MSR_LSTAR → 得到 entry_SYSCALL_64 地址
  │  2. 保存用户态 RIP 到 RCX，RFLAGS 到 R11
  │  3. 从 TSS 加载内核栈指针到 RSP
  │  4. 加载内核代码段/数据段
  ↓
entry_SYSCALL_64()             [arch/x86/entry/entry_64.S:87]
  │  swapgs                    ; 交换 GS 基址(用户→内核 per-cpu)
  │  保存用户态 RSP 到 per-cpu 暂存区
  │  SWITCH_TO_KERNEL_CR3      ; 切换页表(KPTI 场景)
  │  加载内核栈顶到 RSP
  │  构造 struct pt_regs (压入所有寄存器)
  ↓
do_syscall_64(regs, nr)        [arch/x86/entry/common.c:73]
  │  syscall_enter_from_user_mode()  ; 审计/seccomp/ptrace 钩子
  │  do_syscall_x64(regs, nr)        ; 查 sys_call_table[nr] → 执行
  ↓
sys_xxx()                      ; 具体的系统调用实现 (如 sys_write)
  ↓
syscall_exit_to_user_mode()    ; 返回前的检查
  ↓
exit_to_user_mode_loop()       [kernel/entry/common.c:151]
  │  检查 _TIF_NEED_RESCHED → schedule()   ← 调度检查点!
  │  检查 _TIF_SIGPENDING → 处理信号
  │  检查 _TIF_NOTIFY_RESUME → 回调通知
  ↓
sysretq                        ; CPU 硬件: 恢复用户态 RCX/R11/RFLAGS
  ↓
用户态: 继续执行 syscall 下一条指令
```

### 3.2 entry_SYSCALL_64 汇编入口

CPU 执行 `syscall` 指令后，硬件自动跳转到 `MSR_LSTAR` 指向的地址：

```asm
// arch/x86/entry/entry_64.S:87
SYM_CODE_START(entry_SYSCALL_64)
    swapgs                              ; 交换 GS 段寄存器 (用户 per-cpu → 内核 per-cpu)
    movq %rsp, PER_CPU_VAR(cpu_tss_rw + TSS_sp2)  ; 暂存用户栈
    SWITCH_TO_KERNEL_CR3 scratch_reg=%rsp           ; KPTI: 切换到内核页表
    movq PER_CPU_VAR(cpu_current_top_of_stack), %rsp ; 加载内核栈顶

    ; 在内核栈上构造 struct pt_regs (保存所有用户态寄存器)
    pushq $__USER_DS                    ; ss
    pushq PER_CPU_VAR(cpu_tss_rw + TSS_sp2)  ; sp (用户栈)
    pushq %r11                          ; flags (syscall 自动把 RFLAGS 存入 R11)
    pushq $__USER_CS                    ; cs
    pushq %rcx                          ; ip (syscall 自动把返回地址存入 RCX)
    pushq %rax                          ; orig_ax (系统调用号)

    PUSH_AND_CLEAR_REGS rax=$-ENOSYS   ; 保存并清零所有通用寄存器

    movq %rsp, %rdi                     ; 参数1: struct pt_regs *
    movslq %eax, %rsi                   ; 参数2: 系统调用号 (符号扩展)

    call do_syscall_64                  ; 进入 C 代码
```

### 3.3 do_syscall_64：分派到具体实现

```c
// arch/x86/entry/common.c:73
__visible noinstr void do_syscall_64(struct pt_regs *regs, int nr)
{
    add_random_kstack_offset();    // KASLR: 随机化内核栈偏移 (防栈喷射)
    nr = syscall_enter_from_user_mode(regs, nr);  // 审计/seccomp/ptrace 检查

    instrumentation_begin();

    // 先尝试 x86_64 原生系统调用表，再尝试 x32 兼容表
    if (!do_syscall_x64(regs, nr) && !do_syscall_x32(regs, nr) && nr != -1) {
        regs->ax = __x64_sys_ni_syscall(regs);  // 无效调用号 → 返回 -ENOSYS
    }

    instrumentation_end();
    syscall_exit_to_user_mode(regs);  // 返回用户态前的检查
}
```

`do_syscall_x64` 内部通过系统调用号查表：

```c
// 简化逻辑
static __always_inline bool do_syscall_x64(struct pt_regs *regs, int nr)
{
    if (likely((unsigned long)nr < NR_syscalls)) {
        regs->ax = sys_call_table[nr](regs);  // 查表调用
        return true;
    }
    return false;
}
```

`sys_call_table` 是一个函数指针数组，每个元素指向一个 `sys_xxx()` 实现：

```
sys_call_table[0]  = sys_read
sys_call_table[1]  = sys_write
sys_call_table[2]  = sys_open
sys_call_table[57] = sys_fork
sys_call_table[62] = sys_kill
...
```

### 3.4 返回：exit_to_user_mode_loop

系统调用执行完毕后，`syscall_exit_to_user_mode` 在返回用户态前检查所有待处理工作：

```c
// kernel/entry/common.c:151
static unsigned long exit_to_user_mode_loop(struct pt_regs *regs,
                                            unsigned long ti_work)
{
    while (ti_work & EXIT_TO_USER_MODE_WORK) {

        if (ti_work & _TIF_NEED_RESCHED)
            schedule();                    // ← 调度检查点!

        if (ti_work & _TIF_SIGPENDING | _TIF_NOTIFY_SIGNAL)
            handle_signal_work(regs);     // 处理信号

        if (ti_work & _TIF_NOTIFY_RESUME)
            tracehook_notify_resume(regs); // 通知回调

        // ... 其他工作项

        ti_work = read_thread_flags();     // 重新读取 (可能在处理期间被设置)
    }
    return ti_work;
}
```

**关键**：这是调度器五大触发路径中的 **路径 D（返回用户态）**。时钟中断在 `scheduler_tick` 中设置的 `TIF_NEED_RESCHED` 标志，就是在这里被检查并执行 `schedule()` 的。

## 四、参数传递约定

x86_64 系统调用通过寄存器传递参数（不是栈）：

| 寄存器 | 用途 |
|--------|------|
| `rax` | 系统调用号 → 返回值 |
| `rdi` | 参数 1 |
| `rsi` | 参数 2 |
| `rdx` | 参数 3 |
| `r10` | 参数 4（注意：不是 rcx，因为 syscall 指令占用了 rcx） |
| `r8` | 参数 5 |
| `r9` | 参数 6 |

为什么参数 4 用 `r10` 而不是 `rcx`？因为 `syscall` 指令自动将返回地址存入 `RCX`，`RFLAGS` 存入 `R11`，这两个寄存器被硬件占用了。

## 五、int 0x80 的兼容保留

x86_64 内核仍然支持 `int 0x80`，用于运行 32 位兼容程序（IA32 emulation）：

```
int 0x80 → IDT[0x80] → entry_INT80_compat()
  → 构造 pt_regs
  → do_int80_compat(regs)
    → 查 ia32_sys_call_table[nr]  (32位系统调用表)
    → 执行 __ia32_sys_xxx()
```

注意 32 位和 64 位使用**不同的系统调用表**（`ia32_sys_call_table` vs `sys_call_table`），系统调用号也不完全相同。

## 六、struct pt_regs：内核看到的用户态快照

`entry_SYSCALL_64` 在内核栈上构造的 `pt_regs` 结构保存了用户态的完整快照：

```c
// arch/x86/include/asm/ptrace.h
struct pt_regs {
    /* 由 PUSH_AND_CLEAR_REGS 保存 */
    unsigned long r15, r14, r13, r12;
    unsigned long bp, bx;
    unsigned long r11, r10, r9, r8;
    unsigned long ax, cx, dx, si, di;

    /* 由 entry_SYSCALL_64 手工压入 */
    unsigned long orig_ax;    // 系统调用号
    unsigned long ip;         // 用户态 RIP (返回地址)
    unsigned long cs;         // 用户态代码段
    unsigned long flags;      // 用户态 RFLAGS
    unsigned long sp;         // 用户态 RSP
    unsigned long ss;         // 用户态数据段
};
```

内核通过这个结构访问用户态的所有寄存器值，系统调用实现函数接收 `struct pt_regs *regs` 作为参数。

## 七、设计思考

### 为什么 syscall 比 int 0x80 快？

- `int 0x80` 走完整的中断流程：查 IDT → 读取门描述符 → 段切换 → 栈切换 → 跳转。每一步都要访问内存（IDT 表项在内存中）
- `syscall` 由 CPU 内部微码直接处理：入口地址在 MSR 寄存器中（不需要查表），段切换由 CPU 硬编码（不需要读 GDT），栈切换从 TSS 直接读取

### 为什么返回时要循环检查？

`exit_to_user_mode_loop` 用 `while` 循环反复检查，因为在处理一个工作项（如信号）的过程中，另一个工作项（如调度请求）可能被设置。必须确保所有待处理工作都完成后才能安全返回用户态。

### syscall 返回优先用 sysret 而非 iret

`entry_SYSCALL_64` 返回时会检查是否满足 `sysret` 的条件（CS/SS 匹配、RIP 规范地址等）。满足则用 `sysretq`（快速路径），不满足则走 `swapgs_restore_regs_and_return_to_usermode` 用 `iret`（慢速路径）。大多数正常系统调用走 `sysret` 快路径。
