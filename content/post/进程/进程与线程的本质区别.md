## 1. 核心结论

**Linux 内核中没有独立的"线程"数据结构**——进程和线程都用同一个 `task_struct` 表示，走同一个 `copy_process()` 创建路径。**唯一的区别是 clone flags**，决定了各类资源是"共享引用计数+1"还是"复制一份独立副本"。

```
fork()           → clone_flags ≈ SIGCHLD（所有资源独立复制）
pthread_create() → clone_flags = CLONE_VM | CLONE_FS | CLONE_FILES | 
                                 CLONE_SIGHAND | CLONE_THREAD | ...（资源共享）
```

Linus Torvalds 的设计哲学：**不区分进程和线程，只有"任务"(task) 和"共享程度"的差异**。

## 2. task_struct 中的关键共享字段

```c
// include/linux/sched.h

struct task_struct {
    pid_t               pid;            // 内核级唯一标识（每个线程不同）
    pid_t               tgid;           // 线程组 ID（同进程的线程 tgid 相同）
    struct task_struct   *group_leader;  // 线程组主线程指针

    struct mm_struct     *mm;            // 地址空间 ← CLONE_VM 控制
    struct files_struct  *files;         // 文件描述符表 ← CLONE_FILES 控制
    struct fs_struct     *fs;            // 文件系统信息 ← CLONE_FS 控制
    struct sighand_struct *sighand;      // 信号处理程序 ← CLONE_SIGHAND 控制
    struct signal_struct *signal;        // 信号队列/资源限制 ← CLONE_THREAD 控制

    void                *stack;          // 内核栈（永远独立，不可共享）
    struct sched_entity  se;             // 调度实体（永远独立）
};
```

关键洞察：**这些字段都是指针**。共享时多个 `task_struct` 的指针指向同一个对象（引用计数+1）；独立时各自指向自己的副本。

## 3. clone flags 依赖链

```c
// kernel/fork.c:2271-2286 — copy_process() 入口校验

if ((clone_flags & CLONE_THREAD) && !(clone_flags & CLONE_SIGHAND))
    return ERR_PTR(-EINVAL);

if ((clone_flags & CLONE_SIGHAND) && !(clone_flags & CLONE_VM))
    return ERR_PTR(-EINVAL);
```

内核强制的依赖关系：

```
CLONE_THREAD → CLONE_SIGHAND → CLONE_VM
（线程组）      （共享信号处理）   （共享地址空间）
```

逻辑：线程组成员必须共享信号处理程序（否则信号投递语义不一致）；共享信号处理程序必须共享地址空间（信号处理函数的代码和栈在地址空间中）。

## 4. 各资源的共享 vs 复制源码分析

`copy_process()` 中依次调用以下函数（`kernel/fork.c:2546-2599`）：

```
copy_files()   → copy_fs()    → copy_sighand() → copy_signal() → copy_mm()
```

### 4.1 地址空间 mm_struct（最核心的区别）

```c
// kernel/fork.c:1690-1734

static int copy_mm(unsigned long clone_flags, struct task_struct *tsk)
{
    struct mm_struct *mm, *oldmm;
    tsk->mm = NULL;
    tsk->active_mm = NULL;

    oldmm = current->mm;
    if (!oldmm)
        return 0;  // 内核线程无用户空间 mm

    if (clone_flags & CLONE_VM) {
        // ========== 线程路径 ==========
        // 只增加引用计数，共享同一个 mm_struct
        mmget(oldmm);
        mm = oldmm;
    } else {
        // ========== 进程路径 ==========
        // dup_mm → allocate_mm + memcpy + mm_init + dup_mmap
        // dup_mmap 内部调用 copy_page_range 实现 COW
        mm = dup_mm(tsk, current->mm);
        if (!mm)
            return -ENOMEM;
    }

    tsk->mm = mm;
    tsk->active_mm = mm;
    return 0;
}
```

**进程**：`dup_mm()` 分配新的 `mm_struct`，遍历父进程所有 VMA 创建副本，调用 `copy_page_range()` 设置写时复制（COW）。这是 `fork()` 最重的操作。

**线程**：仅 `mmget(oldmm)` 增加引用计数（一条原子指令），几乎零开销。

### 4.2 文件描述符表 files_struct

```c
// kernel/fork.c:1784-1811

static int copy_files(unsigned long clone_flags, struct task_struct *tsk)
{
    struct files_struct *oldf, *newf;

    oldf = current->files;
    if (clone_flags & CLONE_FILES) {
        // 线程：共享 fd 表，引用计数+1
        atomic_inc(&oldf->count);
        goto out;
    }
    // 进程：dup_fd 复制整个 fd 表
    newf = dup_fd(oldf, NR_OPEN_MAX, &error);
    tsk->files = newf;
    ...
}
```

**影响**：线程共享 fd 表意味着一个线程 `close(fd)` 会影响同进程所有线程。

### 4.3 文件系统信息 fs_struct

```c
// kernel/fork.c:1748-1767

static int copy_fs(unsigned long clone_flags, struct task_struct *tsk)
{
    struct fs_struct *fs = current->fs;
    if (clone_flags & CLONE_FS) {
        // 线程：共享，引用计数+1
        fs->users++;
        return 0;
    }
    // 进程：copy_fs_struct 复制 cwd、root、umask
    tsk->fs = copy_fs_struct(fs);
    ...
}
```

**影响**：线程共享 `fs_struct` 意味着一个线程 `chdir()` 会影响同进程所有线程的工作目录。

### 4.4 信号处理程序 sighand_struct

```c
// kernel/fork.c:1854-1881

static int copy_sighand(unsigned long clone_flags, struct task_struct *tsk)
{
    if (clone_flags & CLONE_SIGHAND) {
        // 线程：共享信号处理程序表
        refcount_inc(&current->sighand->count);
        return 0;
    }
    // 进程：分配新的 sighand_struct 并复制 action 数组
    sig = kmem_cache_alloc(sighand_cachep, GFP_KERNEL);
    RCU_INIT_POINTER(tsk->sighand, sig);
    ...
}
```

### 4.5 信号队列与资源限制 signal_struct

```c
// kernel/fork.c:1925-1931

static int copy_signal(unsigned long clone_flags, struct task_struct *tsk)
{
    if (clone_flags & CLONE_THREAD)
        // 线程：复用父线程的 signal_struct（已在 dup_task_struct 中复制指针）
        return 0;
    // 进程：分配新的 signal_struct
    ...
}
```

`signal_struct` 包含：pending 信号队列、rlimit 资源限制、CPU 时间统计、线程组存活计数等。同一进程的所有线程共享这些信息。

### 4.6 汇总对照表

| 资源 | 数据结构 | 控制 flag | fork（进程） | pthread_create（线程） | 共享含义 |
|------|---------|-----------|-------------|---------------------|---------|
| 地址空间 | `mm_struct` | `CLONE_VM` | COW 复制 | **共享** | 全局变量、堆、代码段互通 |
| 文件描述符表 | `files_struct` | `CLONE_FILES` | 复制 | **共享** | close/dup 互相影响 |
| 文件系统信息 | `fs_struct` | `CLONE_FS` | 复制 | **共享** | chdir/chroot 互相影响 |
| 信号处理 | `sighand_struct` | `CLONE_SIGHAND` | 复制 | **共享** | signal handler 统一 |
| 信号队列/rlimit | `signal_struct` | `CLONE_THREAD` | 新建 | **共享** | 资源限制统一 |
| **内核栈** | — | — | **独立** | **独立** | 永远独立 |
| **调度实体** | `sched_entity` | — | **独立** | **独立** | 独立调度 |

## 5. 共享地址空间却拥有私有栈——线程栈的实现

理解了第 4 节的资源共享后，一个自然的问题出现了：线程共享 `mm_struct`（同一个虚拟地址空间），那每个线程怎么拥有自己的栈？答案是——**栈的"私有"不靠内存隔离，靠的是每个线程的 RSP 寄存器指向地址空间中不同的区域**。

### 5.1 用户栈：glibc mmap 分配 + 内核设置 RSP

流程分两步：glibc 在用户空间分配栈内存 → 内核在 `copy_thread` 中设置新线程的栈指针。

**第一步**：`pthread_create()` 在调 `clone()` 之前，先 `mmap` 一块匿名内存作为栈：

```c
// glibc 伪代码
stack = mmap(NULL, 8MB, PROT_READ|PROT_WRITE,
             MAP_PRIVATE|MAP_ANONYMOUS|MAP_STACK, -1, 0);
clone(CLONE_VM|CLONE_THREAD|..., stack + 8MB, ...);
//                                ^^^^^^^^^^^ 栈顶（高地址端）
```

这段 mmap 区域成为共享 `mm_struct` 中的一个新 VMA。所有线程都**能看到**这块内存（同一地址空间），但只有该线程的 RSP 指向它。

**第二步**：`clone()` 把栈地址传入内核，最终到达 `copy_thread()`：

```c
// kernel/fork.c:2619-2633
retval = copy_thread(clone_flags, args->stack, args->stack_size, p, args->tls);
```

```c
// arch/x86/kernel/process.c:275-299

*childregs = *current_pt_regs();       // 复制父线程全部寄存器
childregs->ax = 0;                     // 子线程返回值 = 0

if (sp)
    childregs->sp = sp;                // 关键：用新栈地址覆盖 RSP
```

`fork()` 时 `sp = 0`，子进程继承父进程的 RSP（COW 页表保证安全）。`pthread_create()` 时 `sp = 栈顶`，新线程指向独立的 mmap 区域。

### 5.2 内核栈：每个线程永远独立

```c
// kernel/fork.c:1044-1087 — dup_task_struct
stack = alloc_thread_stack_node(tsk, node);  // vmalloc 分配 16KB
tsk->stack = stack;
```

内核栈在 vmalloc 空间中（不在用户 `mm_struct` 里），每个线程永远独立分配。

### 5.3 fork_frame 与首次调度

```c
// arch/x86/kernel/process.c:194-221
childregs = task_pt_regs(p);                     // 内核栈顶的 pt_regs
fork_frame = container_of(childregs, struct fork_frame, regs);
frame->ret_addr = (unsigned long) ret_from_fork;  // 首次调度跳转到这里
p->thread.sp = (unsigned long) fork_frame;         // 子线程内核 RSP
```

### 5.4 地址空间布局

```
             共享的 mm_struct（同一个虚拟地址空间）
             ┌──────────────────────────────────────┐
高地址       │  主线程栈 (8MB VMA)                    │ ← 主线程 RSP
             ├──────────────────────────────────────┤
             │  线程1栈 (8MB VMA, mmap分配)           │ ← 线程1 RSP
             ├──────────────────────────────────────┤
             │  线程2栈 (8MB VMA, mmap分配)           │ ← 线程2 RSP
             ├──────────────────────────────────────┤
             │  堆 / 共享库 / 代码段                  │ ← 所有线程共享
低地址       └──────────────────────────────────────┘

内核空间 vmalloc 区域（不在用户 mm 中）:
             ┌────────────────────┐
             │ 主线程内核栈 16KB   │
             │ 线程1内核栈 16KB    │  ← 每个线程 task_struct.stack
             │ 线程2内核栈 16KB    │
             └────────────────────┘
```

**设计本质**：所有线程共享地址空间，栈只是地址空间中的一段普通 VMA，通过 RSP 寄存器的不同指向实现"私有"。这也意味着线程间可以传递栈上变量的指针（同一地址空间，合法但危险）。

## 6. PID 与 TGID 的关系

```c
// kernel/fork.c:2715-2734

p->pid = pid_nr(pid);  // 分配唯一的内核 PID

if (clone_flags & CLONE_THREAD) {
    // 线程：继承父线程的 group_leader 和 tgid
    p->group_leader = current->group_leader;
    p->tgid = current->tgid;
} else {
    // 进程：自己就是 group_leader，tgid = 自己的 pid
    p->group_leader = p;
    p->tgid = p->pid;
}
```

```
进程 A (pid=100, tgid=100, group_leader=自己)
  ├── 线程 1 (pid=101, tgid=100, group_leader=A)
  ├── 线程 2 (pid=102, tgid=100, group_leader=A)
  └── 线程 3 (pid=103, tgid=100, group_leader=A)
```

- `getpid()` → 返回 `tgid`（同进程所有线程看到相同的 "PID"=100）
- `gettid()` → 返回 `pid`（每个线程各不相同：100, 101, 102, 103）
- `ps -ef` 只显示 `tgid`（主线程）；`ps -eLf` 显示所有 LWP（轻量级进程）

线程加入线程组的链表操作：

```c
// kernel/fork.c:2875-2903

current->signal->nr_threads++;               // 线程组线程数+1
atomic_inc(&current->signal->live);          // 存活线程数+1
refcount_inc(&current->signal->sigcnt);      // 信号结构引用+1

list_add_tail_rcu(&p->thread_group,
                  &p->group_leader->thread_group);  // 加入线程组链表
list_add_tail_rcu(&p->thread_node,
                  &p->signal->thread_head);         // 加入信号线程链表
```

## 7. 大量创建线程的影响

### 7.1 内核栈消耗（不可 swap）

每个线程拥有独立的内核栈，x86-64 下：

```c
// arch/x86/include/asm/page_64_types.h:15-16

#define THREAD_SIZE_ORDER  (2 + KASAN_STACK_ORDER)  // 通常为 2
#define THREAD_SIZE  (PAGE_SIZE << THREAD_SIZE_ORDER) // = 4096 << 2 = 16384 (16KB)
```

**1000 个线程 = 16MB 纯内核栈**，且这部分内存是 `vmalloc` 分配、不可 swap 的物理内存。10000 个线程 = 160MB。

### 7.2 task_struct 与调度开销

每个线程都是一个独立的 `sched_entity`，加入 CFS 红黑树。线程越多：

```
 ┌──────────────────────────────────────────────────┐
 │           CFS 红黑树                              │
 │                                                   │
 │     1000 个线程 = 1000 个红黑树节点                │
 │     每次 pick_next_task 代价: O(1) 取最左节点      │
 │     每次 enqueue/dequeue:    O(log N)              │
 │     但 N=1000 时上下文切换频率急剧上升             │
 │     CPU 时间大量消耗在调度器本身                    │
 └──────────────────────────────────────────────────┘
```

- 时间片被切分得更细，每个线程实际运行时间缩短
- 上下文切换频率上升 → cache/TLB 污染加剧
- `schedule()` 本身的锁竞争增大（`rq->lock`）

### 7.3 PID 资源消耗

```c
// include/linux/threads.h:28-35

#define PID_MAX_DEFAULT  0x8000              // 32768
#define PID_MAX_LIMIT    (CONFIG_BASE_SMALL ? PAGE_SIZE * 8 : \
	(sizeof(long) > 4 ? 4 * 1024 * 1024 : PID_MAX_DEFAULT))
// 64 位系统: 上限 4,194,304
```

每个线程消耗一个内核 PID（即使用户空间通过 `getpid()` 看到的是相同的 tgid）。系统全局共享 PID 空间，大量线程会挤压其他进程的 PID 分配。

### 7.4 信号处理锁竞争

同一线程组共享 `signal_struct`，信号投递需要持有 `siglock`：

```
信号投递路径:
  complete_signal() → lock_task_sighand() → spin_lock(siglock) → ...

线程越多 → signal_struct 上的 siglock 竞争越激烈
         → 进程级信号投递（如 SIGTERM）需要遍历线程组所有成员
```

### 7.5 VMA 膨胀

虽然线程共享 `mm_struct`，但每个线程的用户态栈通过 `mmap` 分配独立区域（默认 8MB）：

```
1000 个线程 = 1000 个栈 VMA = mm_struct 中 1000+ 个 vm_area_struct
→ find_vma() 红黑树查找 O(log N) 变慢
→ /proc/<pid>/maps 极长
→ fork 此进程时 dup_mmap 需要复制 1000+ 个 VMA
```

### 7.6 影响量化

| 资源 | 每线程开销 | 1000 线程 | 10000 线程 |
|------|-----------|----------|-----------|
| 内核栈 | 16KB (不可 swap) | 16MB | 160MB |
| task_struct | ~6KB (slab) | 6MB | 60MB |
| 内核 PID | 1 个 | 1000 | 10000 |
| 用户栈 VMA | 8MB 虚拟地址 | 8GB 虚拟 | 80GB 虚拟 |
| 调度延迟 | O(log N) enqueue | 可感知 | 严重退化 |

## 8. 内核对线程数的限制机制

### 8.1 全局线程数上限 max_threads

```c
// kernel/fork.c:259,900-918

static int max_threads;  // 全局线程数上限

static void set_max_threads(unsigned int max_threads_suggested)
{
    u64 threads;
    unsigned long nr_pages = totalram_pages();

    // 计算方式：总内存 / (THREAD_SIZE * 8)
    // 即内核栈最多占用总内存的 1/8
    threads = div64_u64((u64) nr_pages * (u64) PAGE_SIZE,
                        (u64) THREAD_SIZE * 8UL);

    max_threads = clamp_t(u64, threads, MIN_THREADS, MAX_THREADS);
}
```

在 `copy_process()` 中强制检查：

```c
// kernel/fork.c:2417

if (data_race(nr_threads >= max_threads))
    goto bad_fork_cleanup_count;  // 返回 -EAGAIN
```

### 8.2 per-user 进程数限制 RLIMIT_NPROC

```c
// kernel/fork.c:964-967 — 初始化时设为 max_threads/2

init_task.signal->rlim[RLIMIT_NPROC].rlim_cur = max_threads/2;
init_task.signal->rlim[RLIMIT_NPROC].rlim_max = max_threads/2;
```

用户空间可通过 `ulimit -u` 查看/修改。

### 8.3 PID 空间耗尽

```c
// kernel/pid.c:234

nr = idr_alloc_cyclic(&tmp->idr, NULL, pid_min, pid_max, GFP_ATOMIC);
// 分配失败时返回 -ENOSPC → copy_process 返回 -EAGAIN
```

## 9. 进程 vs 线程全景对比

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      fork() 创建进程                                     │
│                                                                         │
│   父进程                              子进程                             │
│  ┌──────────┐                        ┌──────────┐                       │
│  │task_struct│                        │task_struct│                       │
│  ├──────────┤                        ├──────────┤                       │
│  │ mm ──────┼───→ mm_struct A        │ mm ──────┼───→ mm_struct B (COW)│
│  │ files ───┼───→ files_struct A     │ files ───┼───→ files_struct B   │
│  │ fs ──────┼───→ fs_struct A        │ fs ──────┼───→ fs_struct B      │
│  │ signal ──┼───→ signal_struct A    │ signal ──┼───→ signal_struct B  │
│  │ stack ───┼───→ 内核栈 A            │ stack ───┼───→ 内核栈 B         │
│  │ pid=100  │                        │ pid=200  │                       │
│  │ tgid=100 │                        │ tgid=200 │                       │
│  └──────────┘                        └──────────┘                       │
│  每个资源独立复制（或 COW 延迟复制）                                       │
├─────────────────────────────────────────────────────────────────────────┤
│                  pthread_create() 创建线程                               │
│                                                                         │
│   主线程                              新线程                             │
│  ┌──────────┐                        ┌──────────┐                       │
│  │task_struct│                        │task_struct│                       │
│  ├──────────┤                        ├──────────┤                       │
```

## 附录：相关内核参数速查

| 参数 | 路径/命令 | 说明 |
|------|----------|------|
| 系统全局线程上限 | `/proc/sys/kernel/threads-max` | 对应 `max_threads` |
| PID 上限 | `/proc/sys/kernel/pid_max` | 默认 32768，64 位可调到 4M |
| per-user 进程上限 | `ulimit -u` / `RLIMIT_NPROC` | 默认 `max_threads/2` |
| 当前线程总数 | 第一列 `/proc/loadavg` 的分母 | 对应 `nr_threads` |
| 某进程线程数 | `/proc/<pid>/status` 中 `Threads:` | |
| 内核栈大小 | 固定 16KB (x86-64) | `THREAD_SIZE = PAGE_SIZE << 2` |
| 用户栈默认大小 | `ulimit -s` | 默认 8MB |
