# 共享库加载与内存共享机制

> Linux 5.15.78 下共享库（.so）从文件到进程虚拟地址空间的完整加载路径：ELF 段解析 → VMA 创建 → 缺页按需加载 → 物理页共享与 COW 隔离。所有流程均来自内核源码。
>
> 前置阅读：[内存管理完全指南](内存管理完全指南.md) — VMA、缺页处理、伙伴系统的基本流程。
> 相关文档：[x86_64 虚拟内存布局](x86_64虚拟内存布局.md) — 进程地址空间中各段（.text/.data/mmap）的位置。

## 零、三个常见疑问的直觉回答

### Q1：内核是一次性把整个 .so 读进内存，还是按需加载？

**不是整个 .so。** 内核只做两件事，其余全靠缺页中断按需加载：

```
步骤 1: 读文件头（几百字节）
  elf_read(file, &elf_ex, sizeof(elf_ex), 0)     → 64 字节 ELF header
  elf_read(file, phdata, size, elf_ex.e_phoff)   → N × 56 字节 program headers

  此时内核已知：
    PT_LOAD #1: offset=0x0000, filesz=0x1234, flags=R+X  ← .text + .rodata
    PT_LOAD #2: offset=0x2000, filesz=0x0456, flags=RW   ← .data + .bss
    PT_NOTE:    offset=0x1500  ← 跳过，不处理
    PT_DYNAMIC: offset=0x1800  ← 跳过，不处理

步骤 2: 对每个 PT_LOAD 调 mmap()，只建 VMA，不读文件
  elf_map() → vm_mmap(file, addr, size, prot, MAP_PRIVATE, offset)
  → 创建 VMA: 虚拟地址 → (inode, 文件偏移) 的映射关系
  → 物理内存：空。页表项：空。

步骤 3: 进程首次访问时才触发 page fault，从磁盘读入那一页（4KB）
```

**结论**：如果一个 .so 有 100 页代码但进程只调了 1 个函数，可能只有 1-2 页被加载到物理内存。`.symtab`、`.strtab`、`.note` 等非 PT_LOAD 段永远不会被映射。

### Q2：.text/.rodata 在同一个 PT_LOAD 段里，所有进程共享这个物理页吗？怎么实现的？

**共享。** 机制是 page cache — 一个以 `(inode, 页偏移)` 为 key 的全局物理页缓存：

```
.so 文件 (inode=12345)            page cache (全局)
┌──────────────┐                 ┌──────────────────────────────────┐
│ offset 0x0000│──首次缺页加载──→│ key=(12345, 0) → 物理页 A (.text)│
│ (.text 前4KB)│                 │ key=(12345, 1) → 物理页 B (.text)│
│ offset 0x1000│──首次缺页加载──→│ key=(12345, 2) → 物理页 C (.rodata)│
│ (.rodata)    │                 └──────────┬──────────┬────────────┘
└──────────────┘                            │          │
                              ┌─────────────┘          └──────────────┐
                              ▼                                       ▼
                    进程 A 的页表                              进程 B 的页表
                    PTE: VA_A → 物理页 A (只读)               PTE: VA_B → 物理页 A (只读)
                    PTE: VA_A+4K → 物理页 C (只读)            PTE: VA_B+4K → 物理页 C (只读)
                              同一物理页                                   同一物理页
```

**进程 A 先 `dlopen`**：首次执行 `.text` 中的代码 → page fault → 内核查 page cache 未命中 → 从磁盘读入 4KB → 存入 page cache → 建立 A 的页表项。

**进程 B 后 `dlopen` 同一个 .so**：首次执行同一函数 → page fault → 内核查 page cache **命中**（`find_get_page(mapping, offset)` 返回同一个 `struct page`）→ 直接建立 B 的页表项，指向同一物理页。零磁盘 IO。

**为什么 .text/.rodata 永远共享，而 .data 不会？** `.text`/`.rodata` 的 VMA 权限是 `R+X`（不可写），永远不会触发写保护缺页（`do_wp_page`），所以页表永远指向 page cache 中的同一物理页。`.data` 的 VMA 权限是 `RW`（可写），初始也共享 page cache 页，但首次写入时 `do_wp_page()` → `wp_page_copy()` 分配新物理页并复制，此后各进程独立。

### Q3：全局变量和常量落在进程的哪块内存区域？

```
进程虚拟地址空间:

0x0000_0000_0000 ┌──────────────────────┐
                 │  主程序 .text/.data   │
                 ├──────────────────────┤
                 │  堆 (brk/mmap)       │
                 │        ↓              │
                 ├──────────────────────┤
                 │  .so: R+X (PT_LOAD#1)│ ← .text + .rodata → 多进程共享物理页
                 │  .so: RW  (PT_LOAD#2)│ ← .data + .bss → 写入时 COW 私有
                 ├──────────────────────┤
                 │  栈 / [vdso]         │
0xFFFF_8000_0000 ├──────────────────────┤
                 │  内核空间             │
```

**不在主程序的堆上**。全局变量（`.data`/`.bss`）和常量（`.rodata`）都在 .so 的 mmap 映射区。通过 `dlsym()` 拿到的指针，指向的就是这个 mmap 区域内的虚拟地址。

### Q4：page cache 是物理内存吗？COW 是复制整个段还是单个页？

**page cache 就是物理内存（RAM）。** 它和进程的堆/栈使用的匿名页是同一种东西 — 都是物理页帧，区别只在于内容和回收策略：

```
物理内存 (RAM):
┌────────────────────────────────────────────────────────┐
│  page cache 页                    匿名页               │
│  ┌──────────────────┐    ┌──────────────────────┐      │
│  │ libfoo.so .text   │    │ 进程 A 的栈          │      │
│  │ libfoo.so .rodata │    │ 进程 A 的堆          │      │
│  │ libfoo.so .data   │    │ A 的 COW 私有副本    │      │
│  └──────────────────┘    └──────────────────────┘      │
│  内容来自磁盘文件              内容来自进程运行          │
│  按 (inode, offset) 索引     按 (anon_vma, VA) 索引    │
│  内存压力时直接丢弃            内存压力时交换到 swap     │
│  (磁盘有副本,需要时再读回)     (无磁盘副本,必须持久化)   │
└────────────────────────────────────────────────────────┘
```

**COW 是按页粒度（4KB），不是整个段一次性复制。** 如果 `.data` 段占 10 个页，进程只写了其中 1 个页的全局变量，内核只为那 1 页分配新物理页并复制，其余 9 页仍然与 page cache 共享：

```
初始状态 — A、B 共享 .data 段的 3 个 page cache 物理页:

  A 页表                   page cache (物理内存)          B 页表
  ┌─────────┐             ┌──────────┐                ┌─────────┐
  │ 页 0 ───┼────────────→│ 物理页 Y0 │←───────────────┼── 页 0  │  共享
  │ 页 1 ───┼────────────→│ 物理页 Y1 │←───────────────┼── 页 1  │  共享
  │ 页 2 ───┼────────────→│ 物理页 Y2 │←───────────────┼── 页 2  │  共享
  └─────────┘             └──────────┘                └─────────┘

A 写 page 1 中的全局变量 g_count = 99:
  → do_wp_page() → wp_page_copy()
  → 只复制 Y1 为 Z1，Y0 和 Y2 仍然共享

  A 页表                   page cache (物理内存)          B 页表
  ┌─────────┐             ┌──────────┐                ┌─────────┐
  │ 页 0 ───┼────────────→│ 物理页 Y0 │←───────────────┼── 页 0  │  仍共享
  │ 页 1 ───┼──┐          │ 物理页 Y1 │  ←─────────────┼── 页 1  │  仍共享
  └─────────┘  │          │(原始数据) │                └─────────┘
               │          └──────────┘
               └─────────→┌──────────┐
                           │ 物理页 Z1 │  ← A 的匿名私有页
                           │ g_count=99│     B 看不到这次修改
                           └──────────┘
```

**page cache 中的原始页永远不会被 COW 修改** — `wp_page_copy()` 分配一个**新的匿名物理页**，将原始内容复制过去，然后更新该进程的 PTE 指向新页。page cache 中的 Y1 保持不变，B 仍然读到原始值 `g_count = 42`。

**进程退出后 page cache 页不消失。** page cache 是内核的全局资源，按 `(inode, 页偏移)` 索引，不属于任何进程。加载 .so 的进程退出后，page cache 中的物理页仍然保留（直到内存压力触发回收）。后续进程 dlopen 同一 .so 时仍然命中 page cache，无需重新读磁盘。

## 一、SO 文件的 ELF 段结构

`.so` 是位置无关的 ELF 共享目标文件（`ET_DYN`），由若干 **PT_LOAD 段**组成。每个段描述一块需要加载到内存的内容（`include/uapi/linux/elf.h:256`）：

```c
// include/uapi/linux/elf.h:256
typedef struct elf64_phdr {
    Elf64_Word  p_type;     // 段类型: PT_LOAD = 1
    Elf64_Word  p_flags;    // 权限: PF_R(4) | PF_W(2) | PF_X(1)
    Elf64_Off   p_offset;   // 段在文件中的字节偏移
    Elf64_Addr  p_vaddr;    // 段链接时的虚拟地址 (相对值)
    Elf64_Addr  p_paddr;    // 物理地址 (不使用, 为 0)
    Elf64_Xword p_filesz;   // 段在文件中占的字节数
    Elf64_Xword p_memsz;    // 段在内存中占的字节数 (memsz > filesz 部分为 .bss)
    Elf64_Xword p_align;    // 对齐粒度 (通常 0x1000 = 4KB)
} Elf64_Phdr;
```

典型的 `libfoo.so` 有 2 个 PT_LOAD 段：

```
  readelf -l libfoo.so:

  Type  Offset   VirtAddr         PhysAddr  FileSiz  MemSiz   Flg  Align
  LOAD  0x000000 0x000000000000   0x...     0x1234   0x1234   R E  0x1000
        └─ 代码段: .text + .rodata + .plt     filesz == memsz (无 BSS)

  LOAD  0x002000 0x000000003000   0x...     0x0100   0x0200   RW   0x1000
        └─ 数据段: .data + .got + .bss        memsz > filesz → 多出 0x100 是 .bss
```

**设计思考**：`.so` 不指定绝对加载地址，`p_vaddr` 是相对偏移。这使得同一个 `.so` 可以被加载到任意虚拟地址（ASLR 随机化），是位置无关代码（PIC, `-fPIC`）的基础。

## 二、加载触发：谁调用 mmap

```
① 程序启动时 (静态依赖)
   execve() → load_elf_binary()
     → 读取 PT_INTERP 段，发现需要 /lib64/ld-linux-x86-64.so.2
     → load_elf_interp() → elf_map() → vm_mmap()

② 运行时 dlopen() (动态依赖)
   dlopen("libfoo.so")
     → ld-linux.so 解析 ELF → open() + mmap() 系统调用
     → do_mmap() → mmap_region()
```

无论哪种方式，最终都走到同一条内核路径。

## 三、阶段一：elf_map() — 参数转换

内核遍历所有 PT_LOAD 段，对每段调用 `elf_map()`（`fs/binfmt_elf.c:409`）：

```c
// fs/binfmt_elf.c:409
static unsigned long elf_map(struct file *filep, unsigned long addr,
        const struct elf_phdr *eppnt, int prot, int type,
        unsigned long total_size)
{
    unsigned long size = eppnt->p_filesz + ELF_PAGEOFFSET(eppnt->p_vaddr);
    unsigned long off  = eppnt->p_offset - ELF_PAGEOFFSET(eppnt->p_vaddr);

    addr = ELF_PAGESTART(addr);   // 页对齐地址 (向下到 4KB)
    size = ELF_PAGEALIGN(size);   // 页对齐大小 (向上到 4KB)

    if (total_size) {
        // ET_DYN 首个段: 先映射整个 .so 映像大小，再 munmap 多余部分
        // 目的: 预留完整地址空间，防止 ASLR 导致段之间被其他映射插入
        total_size = ELF_PAGEALIGN(total_size);
        map_addr = vm_mmap(filep, addr, total_size, prot, type, off);
        if (!BAD_ADDR(map_addr))
            vm_munmap(map_addr + size, total_size - size);
    } else {
        // 后续段: 按实际大小映射
        map_addr = vm_mmap(filep, addr, size, prot, type, off);
    }
    return map_addr;
}
```

**参数转换示例**（代码段 PT_LOAD#1）：

```
输入:
  p_offset  = 0x0000     p_vaddr  = 0x0000
  p_filesz  = 0x1234     p_flags  = PF_R | PF_X

转换:
  size = 0x1234 + (0x0000 & 0xFFF) = 0x1234
       → ELF_PAGEALIGN(0x1234) = 0x2000 (2 页)
  off  = 0x0000 - 0 = 0x0000
  prot = PROT_READ | PROT_EXEC  (PF_R|PF_X → PROT 标志)
  type = MAP_PRIVATE             (私有映射: 写时复制)
```

**ET_DYN 首次映射的 total_size 策略**：`total_mapping_size()`（`fs/binfmt_elf.c:443`）计算所有 PT_LOAD 段覆盖的总虚拟地址范围。先 `vm_mmap` 整个范围，再 `vm_munmap` 多余部分，确保段间间隙不会被其他映射抢占。

## 四、阶段二：vm_mmap → do_mmap — 参数校验与地址选择

```c
// mm/util.c:561
unsigned long vm_mmap(struct file *file, unsigned long addr,
        unsigned long len, unsigned long prot, unsigned long flags,
        unsigned long pgoff)
{
    // 包装 do_mmap: 加锁 mmap_sem → 调用 do_mmap → 解锁
    return do_mmap(file, addr, len, prot, flags, pgoff, ...);
}
```

```c
// mm/mmap.c:2137
unsigned long do_mmap(struct file *file, unsigned long addr,
        unsigned long len, unsigned long prot,
        unsigned long flags, unsigned long pgoff, ...)
{
    struct mm_struct *mm = current->mm;  // 当前进程的内存描述符

    // 1. 长度对齐到页 (4KB 整数倍)
    //    MMU 以页为最小粒度管理内存，不可能映射半个页
    len = PAGE_ALIGN(len);

    // 2. 非 MAP_FIXED 时，确保地址不低于 mmap_min_addr (通常 0x10000)
    //    低地址映射容易被 NULL 指针解引用利用
    if (!(flags & MAP_FIXED))
        addr = round_hint_to_min(addr);

    // 3. 将用户态标志转为内核 vm_flags:
    //    PROT_READ|PROT_EXEC + MAP_PRIVATE
    //    → VM_READ | VM_EXEC | VM_MAYREAD | VM_MAYWRITE | VM_MAYEXEC
    vm_flags = calc_vm_prot_bits(prot, pkey) | calc_vm_flag_bits(flags);

    // 4. 在进程的虚拟地址空间中找一个空闲区域
    addr = get_unmapped_area(file, addr, len, pgoff, flags);

    // 5. 调用 mmap_region 创建 VMA
    return mmap_region(file, addr, len, vm_flags, pgoff, uf);
}
```

## 五、阶段三：mmap_region() — 创建 VMA

```c
// mm/mmap.c:2932
unsigned long mmap_region(struct file *file, unsigned long addr,
        unsigned long len, vm_flags_t vm_flags, unsigned long pgoff, ...)
{
    struct mm_struct *mm = current->mm;

    // 步骤1: 检查虚拟地址空间是否超限 (RLIMIT_AS, ulimit -v)
    if (!may_expand_vm(mm, vm_flags, len >> PAGE_SHIFT))
        return -ENOMEM;

    // 步骤2: 清理目标区域的旧映射 (MAP_FIXED 时覆盖已有 VMA)
    //        并查找新 VMA 在红黑树中的插入位置
    munmap_vma_range(mm, addr, len, &prev, &rb_link, &rb_parent, uf);

    // 步骤3: 内存配额检查 (overcommit accounting)
    //        只有"私有可写"映射才需要记账 (可能消耗物理内存)
    if (accountable_mapping(file, vm_flags)) {
        charged = len >> PAGE_SHIFT;
        if (security_vm_enough_memory_mm(mm, charged))
            return -ENOMEM;
        vm_flags |= VM_ACCOUNT;
    }

    // 步骤4: 尝试与相邻 VMA 合并 (相同属性且地址连续 → 节省结构体)
    vma = vma_merge(mm, prev, addr, addr + len, vm_flags,
                    NULL, file, pgoff, ...);
    if (vma)
        goto out;  // 合并成功, 无需新建

    // 步骤5: 分配 vm_area_struct (从 vm_area_cachep SLUB 缓存分配)
    vma = vm_area_alloc(mm);

    // 步骤6: 填充 VMA 核心字段
    vma->vm_start     = addr;           // 起始虚拟地址
    vma->vm_end       = addr + len;     // 结束虚拟地址 (不包含)
    vma->vm_flags     = vm_flags;       // VM_READ|VM_EXEC|...
    vma->vm_page_prot = vm_get_page_prot(vm_flags);  // 转换为硬件 PTE 保护位
    vma->vm_pgoff     = pgoff;          // 文件偏移 (页为单位)

    // 步骤7: 文件映射 — 关联 struct file 并设置 vm_ops
    vma->vm_file = get_file(file);       // 增加文件引用计数 (f_count++)
    error = call_mmap(file, vma);        // 调用 file->f_op->mmap(file, vma)
                                         // → 设置 vma->vm_ops (含 fault 回调)

    // 步骤8: 将 VMA 插入进程的 VMA 红黑树 + 链表
    vma_link(mm, vma, prev, rb_link, rb_parent);

    return addr;
}
```

### call_mmap() 做了什么

`call_mmap()` 调用文件系统的 `mmap` 方法（`mm/mmap.c:3116`）。对于 ext4 等普通文件系统上的 `.so` 文件，最终设置：

```
vma->vm_ops = &generic_file_vm_ops
  其中 vm_ops->fault = filemap_fault    ← 缺页时从文件读取页面
```

**关键：此时只创建了 VMA 数据结构，页表项为空，没有分配物理页。**

## 六、阶段四：缺页中断 — 按需加载物理页

当进程首次访问 `.so` 中的代码或数据时，MMU 发现 PTE 为空（Present bit = 0），触发 Page Fault：

```
CPU 执行 call 0x7f1a00200100 (libfoo.so 中的函数)
  → MMU 查页表: PTE Present=0
  → CPU 触发 #PF 异常
  → entry_64.S → do_page_fault() → handle_mm_fault()
  → do_fault()                      [mm/memory.c:4484]
```

### 6.1 缺页入口：do_fault()

```c
// mm/memory.c:4484
static vm_fault_t do_fault(struct vm_fault *vmf)
{
    struct vm_area_struct *vma = vmf->vma;

    if (!(vmf->flags & FAULT_FLAG_WRITE))
        ret = do_read_fault(vmf);     // 读缺页: .text 只读段走这里
    else if (!(vma->vm_flags & VM_SHARED))
        ret = do_cow_fault(vmf);      // 写缺页 + MAP_PRIVATE: .data 走这里 (COW)
    else
        ret = do_shared_fault(vmf);   // 写缺页 + MAP_SHARED
}
```

### 6.2 读缺页：do_read_fault() — 加载 .text 页面

```c
// mm/memory.c:4374
static vm_fault_t do_read_fault(struct vm_fault *vmf)
{
    // 1. fault_around: 尝试一次性映射附近多个页 (减少缺页次数)
    if (vma->vm_ops->map_pages)
        ret = do_fault_around(vmf);

    // 2. 调用 vma->vm_ops->fault → filemap_fault()
    ret = __do_fault(vmf);

    // 3. 将物理页填入页表项 (PTE)
    ret |= finish_fault(vmf);    // → do_set_pte(): 设置 PTE 指向物理页帧
                                 //   设置保护位 (R-X)，Present=1

    unlock_page(vmf->page);
    return ret;
}
```

### 6.3 filemap_fault() — 从 page cache 或磁盘获取页面

```c
// mm/filemap.c:3042
vm_fault_t filemap_fault(struct vm_fault *vmf)
{
    struct file *file = vmf->vma->vm_file;
    struct address_space *mapping = file->f_mapping;
    pgoff_t offset = vmf->pgoff;     // 要读取的页在文件中的偏移 (页单位)

    // 1. 查 page cache: 这个文件页是否已在物理内存中?
    page = find_get_page(mapping, offset);

    if (page) {
        // ★ 命中! 页已在 page cache 中
        // 另一个进程之前加载过同一 .so → 直接共享这个物理页帧
        if (!(vmf->flags & FAULT_FLAG_TRIED))
            fpin = do_async_mmap_readahead(vmf, page);  // 异步预读相邻页
        return page;
    } else {
        // 未命中: 需要从磁盘读取 (主缺页)
        count_vm_event(PGMAJFAULT);    // /proc/vmstat 中 pgmajfault +1

        // 2. 同步预读: 不仅读当前页，还预读相邻页
        fpin = do_sync_mmap_readahead(vmf);

        // 3. 分配物理页并从文件读入内容
        page = pagecache_get_page(mapping, offset,
                                  FGP_CREAT | FGP_FOR_MMAP, gfp_mask);
        // → 分配物理页帧
        // → 调用文件系统的 readpage 从磁盘读取
        // → 插入 page cache (后续进程直接命中)

        return page;
    }
}
```

**两个进程共享物理页的关键**：page cache 以 `(inode, offset)` 为键索引。两个进程 `mmap` 同一个 `.so` 文件时，`find_get_page()` 找到的是同一个 `struct page`，因此两个进程的 PTE 指向**同一个物理页帧**。

### 6.4 写缺页：do_cow_fault() — .data 段的 Copy-on-Write

```c
// mm/memory.c:4403
static vm_fault_t do_cow_fault(struct vm_fault *vmf)
{
    // 1. 准备匿名 VMA 反向映射结构
    anon_vma_prepare(vma);

    // 2. 分配一个新的私有物理页 (COW 的 "Copy" 目标)
    vmf->cow_page = alloc_page_vma(GFP_HIGHUSER_MOVABLE, vma, vmf->address);

    // 3. 调用 vma->vm_ops->fault 从文件读取原始内容
    ret = __do_fault(vmf);      // vmf->page = 文件中的原始页

    // 4. ★ 核心: 将原始内容复制到新分配的私有页
    copy_user_highpage(vmf->cow_page, vmf->page, vmf->address, vma);
    //            ↑ 新私有页          ↑ 文件原始页 (page cache 中)
    __SetPageUptodate(vmf->cow_page);

    // 5. 将私有页填入 PTE (可读写, Present=1)
    ret |= finish_fault(vmf);

    // 6. 释放对原始页的引用 (page cache 中的页不受影响)
    unlock_page(vmf->page);
    put_page(vmf->page);

    return ret;
}
```

### 6.5 已映射页面的写保护：do_wp_page()

页面首次以只读方式映射后（`.data` 段初始也是只读映射），当进程首次写入时触发写保护缺页：

```c
// mm/memory.c:3420
static vm_fault_t do_wp_page(struct vm_fault *vmf)
{
    vmf->page = vm_normal_page(vma, vmf->address, vmf->orig_pte);

    if (PageAnon(vmf->page)) {
        // 匿名页: 检查是否独占
        if (page_mapcount(page) == 1 && page_count(page) == 1)
            return wp_page_reuse(vmf);   // 独占 → 直接改为可写 (无需复制)
        goto copy;                        // 共享 → 复制
    }

copy:
    // 文件页: 分配新物理页，复制内容，更新 PTE 指向新页
    return wp_page_copy(vmf);
    //   alloc_page → copy_user_highpage → ptep_clear_flush → set_pte_at
}
```

## 七、两个进程的物理内存分布

```
  进程 A 页表                     物理内存                      进程 B 页表
  ┌────────────────┐            ┌──────────────────┐          ┌────────────────┐
  │ VA 0x7f1a..200 │───────────→│ page frame 0x1234 │←─────────│ VA 0x7f3b..200 │
  │ PTE: R-X       │  .text     │ (page cache 中)   │  共享    │ PTE: R-X       │
  │ P=1            │  只读      │ 来自 libfoo.so     │          │ P=1            │
  └────────────────┘            │ 文件偏移 0x0000   │          └────────────────┘
                                ├──────────────────┤
  ┌────────────────┐            │ page frame 0x5678 │          ┌────────────────┐
  │ VA 0x7f1a..300 │── 写入前 ──→│ (page cache 中)   │── 写入前 ─│ VA 0x7f3b..300 │
  │ PTE: R--       │  .data     │ 初始共享           │          │ PTE: R--       │
  │ P=1, COW       │            │                    │          │ P=1, COW       │
  └────────────────┘            └──────────────────┘          └────────────────┘

  进程 A 首次写入 .data 后 (do_wp_page → wp_page_copy):

  ┌────────────────┐            ┌──────────────────┐          ┌────────────────┐
  │ VA 0x7f1a..300 │───────────→│ page frame 0xAAAA │          │ VA 0x7f3b..300 │
  │ PTE: RW-       │  .data     │ (A 的私有副本)     │          │ PTE: R--       │
  │ P=1            │  进程A独有 │ copy_user_highpage │          │ P=1            │
  └────────────────┘            └──────────────────┘          │ → 0x5678 共享  │
                                                              └────────────────┘
```

| 段 | 权限 | 物理页行为 | 内核路径 |
|----|------|-----------|----------|
| `.text` | R-X | 始终共享 | `do_read_fault` → `filemap_fault` → `find_get_page` 命中 page cache |
| `.rodata` | R-- | 始终共享 | 同上 |
| `.data` | RW- | 初始共享，写入时 COW 分离 | 首次写: `do_wp_page` → `wp_page_copy` → 分配新页 + 复制 |
| `.bss` | RW- | 初始共享，写入时 COW 分离 | 同上 (`.bss` 与 `.data` 在同一个 PT_LOAD 段中) |

**设计本质**：只读段天然共享（无需分离）；可写段用 COW 延迟复制，兼顾共享效率和隔离正确性。

## 九、内存存储数据与地址的本质

内存本身不区分内容和含义 — 它只是一堆字节。是 CPU 的**指令**决定了这些字节被当作什么解读：

```
  物理内存中的连续 8 字节:  0x00 0x00 0x7f 0x1a 0x00 0x20 0x00 0x00

  mov rax, [addr]    → 当作数据读取，rax = 0x7f1a00200000
  call *[addr]       → 当作地址跳转，跳转到 0x7f1a00200000
  add  rax, [addr]   → 当作数值参与运算
```

`.got.plt`（全局偏移表）正是利用了这个特性 — 内存中存的是函数地址，`call` 指令把它当跳转目标：

```
  .got.plt 中的内容 (每个条目 8 字节):
  ┌──────────────────────────────────────┐
  │  got[3] = 0x7f1a00201040  ← printf  │  存的是虚拟地址
  │  got[4] = 0x7f1a00201050  ← malloc  │  动态链接器在加载时填入
  └──────────────────────────────────────┘

  调用 printf 时:
  call *got[3]     ; CPU 把 got[3] 中存的值当地址，跳转到 printf
```

这是冯·诺依曼架构的本质：**指令和数据存在同一片内存中**，区别仅在于 CPU 当前如何解读。
