# x86_64 虚拟内存布局

> Linux 5.15.78 x86_64 架构下的虚拟地址空间完整布局：从用户空间到内核空间的每一个区域，所有地址和大小均来自源码定义。
>
> 前置阅读：[内存管理完全指南](内存管理完全指南.md) — mm_struct、VMA、进程地址空间的基本概念。
> 相关文档：[CPU 页表与缓存机制](CPU页表与缓存机制.md) — 4 级/5 级页表结构的详细遍历。

---

## 一、总体架构

x86_64 使用 48 位虚拟地址（4 级页表）或 57 位虚拟地址（5 级页表），地址空间分为**用户空间**和**内核空间**，中间是不可访问的**非规范地址空洞**。

```
  高地址
  ┌─────────────────────────────────────────────────────────┐
  │  ffffffffffffffff  │  内核空间                         │
  │        ...         │  所有进程共享                       │
  │  ffff800000000000  │                                    │
  ├─────────────────────────────────────────────────────────┤
  │  ffff7fffffffffff  │  非规范地址空洞                    │
  │        ...         │  访问触发 #GP 异常                  │
  │  0000800000000000  │                                    │
  ├─────────────────────────────────────────────────────────┤
  │  00007fffffffffff  │  用户空间                          │
  │        ...         │  每个进程独立                       │
  │  0000000000000000  │                                    │
  └─────────────────────────────────────────────────────────┘
  低地址
```

源码定义（`arch/x86/include/asm/page_64_types.h`）：

```c
// 4 级页表：47 位用户虚拟地址 → 128 TB
#define __VIRTUAL_MASK_SHIFT    47

// 5 级页表：56 位用户虚拟地址 → 64 PB
// #define __VIRTUAL_MASK_SHIFT  56

#define task_size_max()     ((_AC(1,UL) << __VIRTUAL_MASK_SHIFT) - PAGE_SIZE)
#define TASK_SIZE_MAX       task_size_max()

// 内核空间起始：符号扩展后的最高位区域
#define __PAGE_OFFSET_BASE_L4   _AC(0xffff888000000000, UL)  // 直接映射基址
#define __START_KERNEL_map      _AC(0xffffffff80000000, UL)  // 内核代码基址
```

非规范地址空洞的存在是因为 x86_64 的地址是**符号扩展**的：48 位地址的第 47 位被复制到高位 63~47，使得用户地址的高位全为 0，内核地址的高位全为 1，中间的值无法表示。

## 二、用户空间布局 (0 ~ 128 TB)

每个进程拥有独立的用户空间（`mm_struct` 管理），典型布局如下：

```
  高地址 (TASK_SIZE_MAX = 0x00007fffffffffff)
  ┌──────────────────────────────────────────┐
  │  stack (向下增长)                         │ ← STACK_TOP_MAX
  │    ├─ ASLR 随机偏移 (最大 ~8 MB)          │   47位: 0x00007ffffffff000
  │    └─ rlimit 限制                         │
  ├──────────────────────────────────────────┤
  │  mmap 区域 (自顶向下增长)                 │ ← mm->mmap_base
  │    ├─ 共享库 (libc.so 等)                 │   = TASK_SIZE - gap - random
  │    ├─ 匿名映射 (mmap)                     │
  │    └─ 与 stack 间隔 ≥128 MB              │
  ├──────────────────────────────────────────┤
  │  heap (brk, 向上增长)                     │ ← mm->brk
  ├──────────────────────────────────────────┤
  │  BSS 段 (未初始化全局变量)                │
  ├──────────────────────────────────────────┤
  │  data 段 (已初始化全局变量)               │
  ├──────────────────────────────────────────┤
  │  text 段 (代码, 可执行)                   │
  ├──────────────────────────────────────────┤
  │  ELF 解释器 (ld-linux) / VDSO / VVAR     │
  └──────────────────────────────────────────┘
  低地址 (0x0)
```

### 2.1 mmap 基址计算

`mmap_base` 决定了共享库和匿名映射的起始位置，从地址空间顶部向下计算（`arch/x86/mm/mmap.c:82`）：

```c
static unsigned long mmap_base(unsigned long rnd, unsigned long task_size,
                               struct rlimit *rlim_stack)
{
    unsigned long gap = rlim_stack->rlim_cur;
    unsigned long pad = stack_maxrandom_size(task_size) + stack_guard_gap;

    if (gap + pad > gap)
        gap += pad;

    /* gap 限制在 [128MB, task_size*5/6] 范围内
     * 最小 128MB：保证 mmap 与 stack 有足够间隔
     * 最大 5/6：防止 stack 限制过大导致 mmap 空间被挤压 */
    gap_min = SIZE_128M;
    gap_max = (task_size / 6) * 5;

    if (gap < gap_min)
        gap = gap_min;
    else if (gap > gap_max)
        gap = gap_max;

    return PAGE_ALIGN(task_size - gap - rnd);
}
```

**设计思考**：mmap 从顶向下分配，与 stack 保持 128 MB 最小间隔，这是为了在 stack ASLR 随机化后仍有安全距离。`5/6` 的上限保证 mmap 区域至少有 `task_size/6` 的空间可用。

### 2.2 data 段与 BSS 段的本质区别

ELF 文件中 `.data` 和 `.bss` 的区别源于**存储效率**：

| 段 | ELF 类型 | 磁盘存储 | 加载行为 | 典型内容 |
|----|---------|---------|---------|---------|
| `.data` | `PROGBITS` | 存储初始值字节 | 直接映射文件内容 | `int x = 42;` |
| `.bss` | `NOBITS` | 仅记录大小 | 分配内存并清零 | `int buf[10000];` |

C 标准保证：未显式初始化的全局/静态变量自动为零。既然值一定为零，就没有必要在文件中存储一大片零 —— `.bss` 只需在 ELF 头中记录"大小 = N 字节"，加载时由内核清零即可。这使得 `char buf[65536]` 不会让可执行文件膨胀 64 KB。

## 四、核心区域详解

### 4.1 物理内存直接映射 (64 TB)

```c
// arch/x86/include/asm/page_64_types.h:42
#define __PAGE_OFFSET_BASE_L4   _AC(0xffff888000000000, UL)
// 运行时变量（KASLR 时随机偏移）
#define __PAGE_OFFSET           page_offset_base
```

**这是内核最重要的映射区域**：所有物理内存被线性映射到一段连续的虚拟地址空间。

```
虚拟地址 = 物理地址 + PAGE_OFFSET
```

- `kmalloc()` 分配的内存就在此区域
- `phys_to_virt()` / `virt_to_phys()` 通过简单加减 `PAGE_OFFSET` 互相转换
- 映射大小 64 TB，对应 `MAX_PHYSMEM_BITS = 46`（`arch/x86/include/asm/sparsemem.h:27`）

```c
// arch/x86/include/asm/sparsemem.h:27
# define MAX_PHYSMEM_BITS   (pgtable_l5_enabled() ? 52 : 46)
// 4 级页表: 2^46 = 64 TB 物理内存上限
// 5 级页表: 2^52 = 4 PB 物理内存上限
```

**设计思考**：直接映射使内核访问物理内存只需一次加法，无需查页表。`PAGE_OFFSET` 选择 `0xffff888000000000` 而非 `0xffff800000000000` 是为了在前方留出 8 TB 保护空洞（给 Xen 等虚拟化使用）和 0.5 TB LDT 映射空间。

### 4.2 vmalloc/ioremap (32 TB)

```c
// arch/x86/include/asm/pgtable_64_types.h:123-126,142
#define __VMALLOC_BASE_L4   0xffffc90000000000UL
#define VMALLOC_SIZE_TB_L4  32UL
#define VMALLOC_END         (VMALLOC_START + (VMALLOC_SIZE_TB << 40) - 1)
```

与直接映射的关键区别：**物理页不要求连续**。

| 特性 | 直接映射 | vmalloc |
|------|---------|---------|
| 物理页连续性 | 天然连续 | 不连续 |
| 虚拟地址连续 | 是 | 是 |
| 用途 | `kmalloc`、普通内核访问 | `vmalloc`、`ioremap`、模块加载 |
| 映射建立时机 | 启动时一次性建立 | 按需逐页建立 |

- `vmalloc()` 分配大块虚拟连续、物理不连续的内存
- `ioremap()` 将设备 MMIO 物理地址映射到此区域
- 模块加载（`module_alloc`）也使用此区域的特殊映射

### 4.3 vmemmap — 虚拟内存映射 (1 TB)

```c
// arch/x86/include/asm/pgtable_64_types.h:129
#define __VMEMMAP_BASE_L4   0xffffea0000000000UL
```

每个物理页帧对应一个 `struct page`（64 字节）。vmemmap 将所有 `struct page` 映射为虚拟连续：

```
64 TB 物理内存 / 4KB 页大小 = 2^34 个页帧
每个页帧一个 struct page (64 字节)
总大小 = 2^34 × 64 = 2^40 = 1 TB
```

**设计思考**：`virt_to_page()` 利用 vmemmap 的线性关系，通过一次减法 + 移位即可得到 `struct page*`，无需查表：

```c
// 简化逻辑：
struct page *virt_to_page(void *addr) {
    unsigned long pfn = __pa(addr) >> PAGE_SHIFT;
    return vmemmap_base + pfn;  // 虚拟地址连续，直接索引
}
```

### 4.4 内核代码映射 (512 MB / 1 GB)

```c
// arch/x86/include/asm/page_64_types.h:50
#define __START_KERNEL_map  _AC(0xffffffff80000000, UL)

// arch/x86/include/asm/page_64_types.h:84-98
/*
 * KERNEL_IMAGE_SIZE 是内核代码虚拟地址的上限（不是实际大小限制）。
 * KASLR 启用时用 1 GB，为模块留 1 GB；
 * KASLR 禁用时用 512 MB，模块空间增大到 1.5 GB。
 */
#ifdef CONFIG_RANDOMIZE_BASE
#define KERNEL_IMAGE_SIZE   (1024 * 1024 * 1024)  // 1 GB
#else
#define KERNEL_IMAGE_SIZE   (512 * 1024 * 1024)   // 512 MB
#endif
```

- 映射到物理地址 0 开始的内核映像（vmlinux）
- 包含内核 .text / .data / .bss / .rodata 段
- KASLR 启用时代码基址在 1 GB 范围内随机偏移

### 4.5 模块映射空间 (1520 MB)

```c
// arch/x86/include/asm/pgtable_64_types.h:144-151
#define MODULES_VADDR   (__START_KERNEL_map + KERNEL_IMAGE_SIZE)
// 无 KASLR: 0xffffffff80000000 + 512MB = 0xffffffffa0000000
// 有 KASLR: 0xffffffff80000000 + 1GB  = 0xffffffffc0000000

#define MODULES_END     _AC(0xffffffffff000000, UL)
#define MODULES_LEN     (MODULES_END - MODULES_VADDR)
// 无 KASLR: 1520 MB; 有 KASLR: ~512 MB
```

- 内核模块（`.ko`）加载到此区域
- 与内核代码区相邻，使得模块可以直接调用内核函数（相对跳转 / PLT）

**设计思考**：模块空间紧跟内核代码之后，是因为 x86_64 的 `CALL`/`JMP` 指令使用 32 位相对偏移（±2 GB），模块必须在内核代码 ±2 GB 范围内才能使用直接跳转，否则需要 PLT 间接跳转（性能更差）。

### 4.6 fixmap — 固定映射

```c
// arch/x86/include/asm/fixmap.h:150-151
#define FIXADDR_SIZE    (__end_of_permanent_fixed_addresses << PAGE_SHIFT)
#define FIXADDR_START   (FIXADDR_TOP - FIXADDR_SIZE)
```

- 编译时确定虚拟地址，运行时才绑定物理页
- 用于 APIC、ACPI 表、early_ioremap 等需要固定地址的硬件映射
- `kmap_atomic()` 使用 fixmap 临时映射高端内存页
- `FIXADDR_TOP` 紧跟 vsyscall 页面下方

### 4.7 cpu_entry_area (512 GB)

```c
// arch/x86/include/asm/pgtable_64_types.h:156-157
#define CPU_ENTRY_AREA_PGD  _AC(-4, UL)
#define CPU_ENTRY_AREA_BASE (CPU_ENTRY_AREA_PGD << P4D_SHIFT)
// = -4 × 512GB = -2TB → 0xfffffe0000000000
```

- 每个 CPU 的异常栈、IST（Interrupt Stack Table）
- GDT、IDT 等 CPU 私有数据
- KPTI（Kernel Page Table Isolation）的用户态 trampoline 页面
- 占用整个 PGD 条目（512 GB），确保与其他区域隔离

### 4.8 KASAN 影子内存 (16 TB)

```c
// arch/x86/include/asm/kasan.h:7,15-24
#define KASAN_SHADOW_SCALE_SHIFT  3
// 影子大小 = 内核地址空间 / 8
// 47 位地址: (1 << 47) / 8 = 16 TB
// 56 位地址: (1 << 56) / 8 = 2^53 = 8 PB

#define KASAN_SHADOW_START  (KASAN_SHADOW_OFFSET + \
        ((-1UL << __VIRTUAL_MASK_SHIFT) >> KASAN_SHADOW_SCALE_SHIFT))
```

- 每 8 字节内核内存对应 1 字节影子内存
- 用于检测 use-after-free、buffer-overflow 等内存错误
- KASAN 启用时 KASLR 被禁用（地址冲突）

### 4.9 vsyscall (4 KB)

```c
// arch/x86/include/uapi/asm/vsyscall.h:11
#define VSYSCALL_ADDR   (-10UL << 20)  // 0xffffffffff600000
```

- 遗留的 ABI 兼容页面，提供 `gettimeofday` 等快速系统调用
- 现代系统使用 vDSO 替代，vsyscall 仅作兼容
- 固定地址，不可重定位

## 五、KASLR 对布局的影响

启用 `CONFIG_RANDOMIZE_BASE` 和 `CONFIG_RANDOMIZE_MEMORY` 时，关键区域的基址在启动早期随机偏移：

| 区域 | 随机化 | 说明 |
|------|--------|------|
| 内核代码 | 在 `KERNEL_IMAGE_SIZE`(1 GB) 内偏移 | `__START_KERNEL_map` + random |
| 直接映射 | `PAGE_OFFSET` 随机偏移 | `page_offset_base` 运行时确定 |
| vmalloc | 随机偏移 | `vmalloc_base` 运行时确定 |
| vmemmap | 随机偏移 | `vmemmap_base` 运行时确定 |

```c
// arch/x86/include/asm/page_64_types.h:44-48
#ifdef CONFIG_DYNAMIC_MEMORY_LAYOUT
#define __PAGE_OFFSET       page_offset_base   // 运行时变量
#else
#define __PAGE_OFFSET       __PAGE_OFFSET_BASE_L4  // 编译时常量
#endif

// arch/x86/include/asm/pgtable_64_types.h:132-139
#ifdef CONFIG_DYNAMIC_MEMORY_LAYOUT
# define VMALLOC_START      vmalloc_base       // 运行时变量
# define VMEMMAP_START      vmemmap_base       // 运行时变量
#else
# define VMALLOC_START      __VMALLOC_BASE_L4
# define VMEMMAP_START      __VMEMMAP_BASE_L4
#endif
```

**设计思考**：三个区域（直接映射、vmalloc、vmemmap）的相对顺序保持不变，只是整体基址随机偏移。这使得区域间的相对偏移计算仍然有效，同时增加了攻击者猜测内核地址的难度。

## 六、5 级页表扩展 (57 位虚拟地址)

启用 `CONFIG_X86_5LEVEL` 时，增加一级 P4D 页表，虚拟地址从 48 位扩展到 57 位：

```c
// arch/x86/include/asm/page_64_types.h:41
#define __PAGE_OFFSET_BASE_L5   _AC(0xff11000000000000, UL)

// arch/x86/include/asm/pgtable_64_types.h:124,127,130
#define __VMALLOC_BASE_L5       0xffa0000000000000UL
#define VMALLOC_SIZE_TB_L5      12800UL    // 12.5 PB
#define __VMEMMAP_BASE_L5       0xffd4000000000000UL
```

| 区域 | 4 级 (48 位) | 5 级 (57 位) |
|------|-------------|-------------|
| 用户空间 | 128 TB | **64 PB** (×512) |
| 保护空洞 | 8 TB | 4 PB |
| 直接映射 | 64 TB | **32 PB** |
| vmalloc | 32 TB | **12.5 PB** |
| vmemmap | 1 TB | **0.5 PB** |
| KASAN 影子 | 16 TB | **~8 PB** |
| 内核代码+模块 | 不变 | **不变** |

**从 `0xfffffc0000000000` (-4 TB) 往上，4 级和 5 级的布局完全相同。** 这是因为内核代码、模块、fixmap 等区域的大小不随物理内存规模变化，不需要扩展。

## 七、地址转换关系总结

```
用户空间虚拟地址
    │
    │  页表查找 (每进程独立 PGD, CR3 指向)
    ▼
物理地址 ──┬── 直接映射区: phys + PAGE_OFFSET
           │    特点: 一次加法，无需查表
           │    用途: kmalloc、内核普通访问
           │
           ├── vmalloc 区: 虚拟连续、物理不连续
           │    特点: 每页独立映射
           │    用途: vmalloc、ioremap、模块
           │
           └── vmemmap 区: struct page 数组
                特点: 虚拟连续，按 pfn 索引
                用途: virt_to_page() 直接计算
```

内核内部地址转换：

```c
// 直接映射区：虚拟地址 ↔ 物理地址（无需页表查找）
#define __va(x)  ((void *)((unsigned long)(x) + PAGE_OFFSET))
#define __pa(x)  ((unsigned long)(x) - PAGE_OFFSET)

// 直接映射区：虚拟地址 → struct page（利用 vmemmap 线性关系）
#define virt_to_page(kaddr) \
    (vmemmap + ((unsigned long)(kaddr) - PAGE_OFFSET) >> PAGE_SHIFT)
```
