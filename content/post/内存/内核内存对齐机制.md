## 一、对齐的三个维度

内核中的对齐不仅仅是"让访问快一点"，它服务于三个完全不同的目标：

| 维度 | 核心诉求 | 不满足的后果 |
|------|----------|--------------|
| **正确性** | DMA buffer 满足 cache line 边界；原子指令满足操作数对齐 | 数据损坏、CPU 异常 |
| **性能** | 热数据独占 cache line；消除 false sharing | 多核场景性能下降 10x |
| **可移植性** | 满足架构最小对齐约束（如 ARM 不支持未对齐访问） | 运行时 fault |

### 澄清：不对齐导致的是"多访问一条 cache line"，不是 cache 失效

一个常见误解是"字节不对齐会导致 cache 失效"。**错。** Cache 失效（Invalidate）只由 MESI 协议的一致性消息、容量驱逐、或软件显式刷新触发，与对齐无关。

不对齐的真正代价是**跨 cache line 访问**：

```
对齐访问（8 字节 long 在地址 0x40）：
┌──────────────── Cache Line 1 (0x40~0x7F) ────────────────┐
│  ... │ [long 在这，完整落在一条 line 内] │ ...             │
└──────────────────────────────────────────────────────────┘
  → 一次 cache line 访问

不对齐访问（8 字节 long 在地址 0x3C，跨越 0x3F/0x40 边界）：
┌────── Cache Line 0 (0x00~0x3F) ──────┐┌────── Cache Line 1 (0x40~0x7F) ───────┐
│                       │ [前4字节]      ││ [后4字节] │                            │
└────────────────────────────────────────┘└───────────────────────────────────────┘
  → 两次 cache line 访问 + CPU 内部拼接
```

| 场景 | 代价 |
|------|------|
| 跨 line 访问（x86） | 延迟翻倍（如果第二条 line miss 则更糟）；占用两个 cache line 空间 |
| 跨 line 访问（ARM/MIPS） | 部分架构直接触发对齐异常，由软件模拟（极慢）或 fault |

Cache 失效（Invalidate）的真正触发条件：

| 触发条件 | 机制 |
|----------|------|
| 其他核写了同一物理地址 | MESI: Shared → 收到 Invalidate → Invalid |
| 容量/冲突驱逐 | LRU 淘汰旧 line |
| 软件显式刷新 | `clflush`/`wbinvd`/`invd` 指令 |
| DMA 一致性操作 | `dma_sync_for_cpu` 等 |

## 二、宏定义体系

### 2.1 Cache Line 对齐宏

```c
// include/linux/cache.h:40
#define ____cacheline_aligned __attribute__((__aligned__(SMP_CACHE_BYTES)))

#ifdef CONFIG_SMP
#define ____cacheline_aligned_in_smp ____cacheline_aligned
#else
#define ____cacheline_aligned_in_smp   // UP 下为空，不浪费空间
#endif

#define __cacheline_aligned \
  __attribute__((__aligned__(SMP_CACHE_BYTES), \
                 __section__(".data..cacheline_aligned")))
```

设计细节：
- `____cacheline_aligned_in_smp`：SMP 内核下按 cache line 对齐，单处理器内核下为空（因为没有 false sharing 问题）
- `__cacheline_aligned`：额外放入专用段 `.data..cacheline_aligned`，便于链接器统一布局

### 2.2 架构 Cache Line 大小

**ARM64**：L1 = 64B，DMA 最小对齐 = 128B

```c
// arch/arm64/include/asm/cache.h:31
#define L1_CACHE_SHIFT      (6)
#define L1_CACHE_BYTES      (1 << L1_CACHE_SHIFT)   // 64
#define ARCH_DMA_MINALIGN   (128)
```

**x86**：L1 大小由 Kconfig 决定

```c
// arch/x86/include/asm/cache.h:7
#define L1_CACHE_SHIFT  (CONFIG_X86_L1_CACHE_SHIFT)
#define L1_CACHE_BYTES  (1 << L1_CACHE_SHIFT)
```

### 2.3 Slab 最小对齐保证

```c
// include/linux/slab.h:202
#if defined(ARCH_DMA_MINALIGN) && ARCH_DMA_MINALIGN > 8
#define ARCH_KMALLOC_MINALIGN ARCH_DMA_MINALIGN      // ARM64: 128
#define KMALLOC_MIN_SIZE ARCH_DMA_MINALIGN
#else
#define ARCH_KMALLOC_MINALIGN __alignof__(unsigned long long)  // x86: 8
#endif

#ifndef ARCH_SLAB_MINALIGN
#define ARCH_SLAB_MINALIGN __alignof__(unsigned long long)     // 8
#endif
```

API 保证：
- `kmalloc()` 返回至少 `ARCH_KMALLOC_MINALIGN` 对齐的指针
- 2 的幂大小额外保证**自然对齐**（size 本身即为对齐边界）

## 三、Slab/SLUB 分配器中的对齐处理

### 3.1 calculate_alignment()：对齐计算核心逻辑

```c
// mm/slab_common.c:146
static unsigned int calculate_alignment(slab_flags_t flags,
        unsigned int align, unsigned int size)
{
    if (flags & SLAB_HWCACHE_ALIGN) {
        unsigned int ralign = cache_line_size();
        // 小对象减半对齐：16B 对象不需要占满整条 64B cache line
        while (size <= ralign / 2)
            ralign /= 2;
        align = max(align, ralign);
    }

    if (align < ARCH_SLAB_MINALIGN)
        align = ARCH_SLAB_MINALIGN;

    return ALIGN(align, sizeof(void *));
}
```

规则链：
1. 用户传入 `align` 为下限
2. `SLAB_HWCACHE_ALIGN` 提升到 cache line 大小（小对象减半）
3. 不低于 `ARCH_SLAB_MINALIGN`
4. 最终对齐到指针大小

### 3.2 SLUB 对象布局

SLUB 通过将对象 stride 对齐到 `s->align` 来保证每个对象满足对齐要求：

```c
// mm/slub.c:4101
/*
 * SLUB stores one object immediately after another beginning from
 * offset 0. In order to align the objects we have to simply size
 * each object to conform to the alignment.
 */
size = ALIGN(size, s->align);
s->size = size;
```

对象连续排列，stride 是 `align` 的倍数 → 每个对象地址自动满足对齐。

### 3.3 KMEM_CACHE 宏：结构体对齐属性自动传递

```c
// include/linux/slab.h:158
#define KMEM_CACHE(__struct, __flags) \
    kmem_cache_create(#__struct, sizeof(struct __struct), \
        __alignof__(struct __struct), (__flags), NULL)
```

结构体定义中使用 `____cacheline_aligned_in_smp` → `__alignof__` 反映 cache line 对齐 → `kmem_cache_create` 的 `align` 参数自动满足。

### 3.4 kmalloc cache：2 的幂自然对齐

```c
// mm/slab_common.c:655
unsigned int align = ARCH_KMALLOC_MINALIGN;
if (is_power_of_2(size))
    align = max(align, size);
s->align = calculate_alignment(flags, align, size);
```

`kmalloc-64` 返回 64 字节对齐的地址，`kmalloc-128` 返回 128 字节对齐——这是 DMA 安全使用 `kmalloc` 缓冲区的基础。

### 3.5 Cache 合并时的对齐兼容检查

```c
// mm/slab_common.c:222
if ((s->size & ~(align - 1)) != s->size)
    continue;  // stride 不是新对齐的倍数 → 不允许合并
```

## 四、DMA 对齐：不是性能问题，是正确性问题

### 问题本质

设备 DMA 直接写物理内存，不经过 CPU cache。如果 DMA buffer 与其他数据共享一条 cache line：

```
Cache Line [64 bytes]
┌──────────────────────────────────────┬──────────────────┐
│  DMA buffer (设备正在写入)            │  其他变量 X      │
└──────────────────────────────────────┴──────────────────┘
                                              ↑
                                     CPU 读 X，把整条 line 拉入 cache
                                     设备写完 DMA buffer
                                     CPU 写回 cache line → 覆盖 DMA 数据！
```

### ARM64 的处理

```c
// arch/arm64/include/asm/cache.h:43
/*
 * Memory returned by kmalloc() may be used for DMA, so we must make
 * sure that all such allocations are cache aligned. Otherwise,
 * unrelated code may cause parts of the buffer to be read into the
 * cache before the transfer is done, causing old data to be seen by
 * the CPU.
 */
#define ARCH_DMA_MINALIGN   (128)
```

ARM64 的 CTR_EL0.CWG（Cache Writeback Granule）可能为 128 字节，因此 `ARCH_DMA_MINALIGN` = 128。启动时校验：

```c
// arch/arm64/mm/dma-mapping.c:43
WARN_TAINT(!coherent && cls > ARCH_DMA_MINALIGN,
    TAINT_CPU_OUT_OF_SPEC,
    "ARCH_DMA_MINALIGN smaller than CTR_EL0.CWG (%d < %d)");
```

### USB 子系统的依赖

```c
// drivers/usb/core/buffer.c:35
/*
 * The pool_max values must never be smaller than ARCH_KMALLOC_MINALIGN.
 */
if (ARCH_KMALLOC_MINALIGN <= 32)
    ;
else if (ARCH_KMALLOC_MINALIGN <= 64)
    pool_max[0] = 64;
else if (ARCH_KMALLOC_MINALIGN <= 128)
    pool_max[0] = 0;  // ARM64: 最小 pool 已经 >= 128，不需要小 pool
```

## 五、原子操作的硬件对齐要求

### struct page 的 16 字节对齐

SLUB 快速路径使用双字 cmpxchg 原子更新 `freelist` + `counters`，x86 的 `CMPXCHG16B` 要求操作数 16 字节对齐：

```c
// include/linux/mm_types.h:67
#ifdef CONFIG_HAVE_ALIGNED_STRUCT_PAGE
#define _struct_page_alignment  __aligned(2 * sizeof(unsigned long))
#endif
```

使用处：
```c
// mm/slub.c:463
if (cmpxchg_double(&page->freelist, &page->counters,
                   freelist_old, counters_old,
                   freelist_new, counters_new))
```

### per-CPU slab 数据的双字对齐

```c
// mm/slub.c:3854
/*
 * Must align to double word boundary for the double cmpxchg
 * instructions to work; see __pcpu_double_call_return_bool().
 */
s->cpu_slab = __alloc_percpu(sizeof(struct kmem_cache_cpu),
                             2 * sizeof(void *));
```

不满足对齐 → x86 上触发 #GP（General Protection fault），ARM 上结果不确定。

## 六、False Sharing 防护实战

> **视角分工**：本节只讲**内核侧**的 false sharing 防护实例（XDP 环形队列、TCP TIME_WAIT 计数器、BPF map 引用计数、HTB 流控、网络哈希表锁）。
>
> 关于 MESI 状态机层面的原理、用户态矩阵遍历案例、`perf c2c` 定位方法等**软件视角**内容，详见 [CPU 页表与缓存机制 §19](CPU页表与缓存机制.md)。

False sharing：不同 CPU 修改同一 cache line 内的不同变量，导致 cache line 在核间反复失效，实际独立的操作变成了串行。

### 为什么有了自然对齐还需要手动写对齐？

编译器的自然对齐（N 字节类型对齐到 N 的倍数）解决的是**单个变量不跨 cache line**；手动对齐解决的是**不同核写的变量不在同一条 cache line**。两者目标完全不同：

```
自然对齐（编译器自动保证）:
  ┌──────────── 64 字节 Cache Line ─────────────┐
  │ [int_a] [int_b] [int_c] ... [int_d]         │
  └──────────────────────────────────────────────┘
  Core 0 写 int_a → 整条 line Invalidate → Core 1 读 int_d 被波及
  
  ✓ 单个 int 不会跨 line（自然对齐的功劳）
  ✗ 但多个变量挤在同一 line 内（false sharing）

手动 cache line 对齐（程序员显式声明）:
  ┌── Cache Line 0 ──┐  ┌── Cache Line 1 ──┐
  │ int_a + 60B pad   │  │ int_d + 60B pad   │
  └───────────────────┘  └───────────────────┘
       Core 0 独占             Core 1 独占
       
  ✓ 各核的写操作互不触发 Invalidate
```

总结：

| | 自然对齐（编译器自动） | 手动 cache line 对齐 |
|---|---|---|
| **对齐到** | `sizeof(type)` 字节（int→4, long→8） | 64 字节（整条 cache line） |
| **保证** | 一次访问不跨两条 line | 不同核的写不在同一条 line |
| **代价** | 极小（结构体内部 padding 几字节） | 较大（每个字段浪费最多 60 字节） |
| **自动 vs 手动** | 编译器自动完成 | 程序员显式声明 `____cacheline_aligned_in_smp` |
| **解决的问题类别** | 正确性+单核性能 | 多核并发性能 |

关键洞察：4 字节的 `u32` 自然对齐到 4 的倍数——它永远不可能"跨" 64 字节边界（4 < 64），但它完全可能和另一个高频写变量**共处一条** cache line。手动对齐解决的是后者。

### "对齐"在 false sharing 语境下的含义

内核中"对齐"用于两个不同目的，容易混淆：

| 目的 | 手段 | 解决的问题 |
|------|------|-----------|
| **自然对齐** | 变量起始地址对齐到 `sizeof(type)` 边界 | 避免一次访问跨两条 cache line（多一次加载） |
| **独占对齐** | 变量占满/对齐到整条 cache line（64B） | 避免与其他核的写变量共享 line（MESI Invalidate） |

`____cacheline_aligned_in_smp` 做的是**后者**——它不是怕变量"跨 line"（4 字节 `u32` 不可能跨 64B 边界），而是要让这个变量**独占**一整条 line，使不同核的写操作落在不同 line 上。

### MESI 状态转换：false sharing 的微观过程

假设 `producer` 和 `consumer` **没有**对齐（同在一条 cache line）：

```
┌────────────── 同一条 Cache Line ──────────────┐
│  producer (Core 0 写)  │  consumer (Core 1 写) │
└───────────────────────────────────────────────┘

t1: Core 0 写 producer
    → 该 line 在 Core 0 上变为 Modified (M)
    → MESI 发 Invalidate 给 Core 1
    → Core 1 的这条 line 变为 Invalid (I)

t2: Core 1 要写 consumer
    → 发现自己的 line 是 Invalid → 必须从 Core 0 取最新数据
    → Core 0 写回（M→I），Core 1 获得 line（I→M）
    → Core 1 写 consumer
    → 发 Invalidate 给 Core 0 → Core 0 的 line 变 Invalid

t3: Core 0 要写 producer
    → 又是 Invalid → 又要从 Core 1 拿回来...

→ 每次写操作都要等 30-100ns 的跨核传输（cache-to-cache transfer）
→ 本来可以各自 1ns 完成的写，变成串行的 30-100ns
```

对齐后各自独占一条 line：

```
┌──── Cache Line A ────┐  ┌──── Cache Line B ────┐
│ producer + 60B pad    │  │ consumer + 60B pad    │
└───────────────────────┘  └───────────────────────┘
     Core 0: Modified            Core 1: Modified
     
互不干扰：Core 0 写 producer 不触发任何 Invalidate
         Core 1 写 consumer 不触发任何 Invalidate
```

### 案例1：XDP 环形队列——SPSC 的经典设计

```c
// net/xdp/xsk_queue.h:16
struct xdp_ring {
    u32 producer ____cacheline_aligned_in_smp;
    /* Hinder the adjacent cache prefetcher to prefetch the consumer
     * pointer if the producer pointer is touched and vice versa. */
    u32 pad1 ____cacheline_aligned_in_smp;
    u32 consumer ____cacheline_aligned_in_smp;
    u32 pad2 ____cacheline_aligned_in_smp;
    u32 flags;
    u32 pad3 ____cacheline_aligned_in_smp;
};
```

4 字节的 `producer` 独占 64 字节 cache line。为什么？
- 内核写 producer，用户空间写 consumer
- 若共享 cache line：每次一端写入，另一端的 CPU cache 失效 → 重新加载
- 高频路径（数百万包/秒）下，false sharing 可导致吞吐量减半

### 案例2：TCP TIME_WAIT 计数器——手动 padding

```c
// include/net/netns/ipv4.h
struct inet_timewait_death_row {
    atomic_t        tw_count;                              // 高频原子操作
    char            tw_pad[L1_CACHE_BYTES - sizeof(atomic_t)];  // 填满 cache line
    struct inet_hashinfo    *hashinfo;                     // 几乎只读
    int             sysctl_max_tw_buckets;                 // 几乎只读
};
```

`tw_count` 每次 TIME_WAIT 创建/销毁都要原子修改。如果和只读字段共享 cache line，所有读这些字段的 CPU 都会被无谓地 invalidate。

### 案例3：BPF map 引用计数——隔离高频写

```c
// include/linux/bpf.h:186
/* The 3rd and 4th cacheline with misc members to avoid false sharing
 * particularly with refcounting. */
atomic64_t refcnt ____cacheline_aligned;
```

### 案例4：HTB 流控——读写分区布局

```c
// net/sched/sch_htb.c:90
/* interior & leaf nodes; props specific to leaves are marked L:
 * To reduce false sharing, place mostly read fields at beginning,
 * and mostly written ones at the end. */
unsigned int drops ____cacheline_aligned_in_smp;
```

### 案例5：网络哈希表锁——扩展锁数量覆盖 cache line

```c
// net/ipv4/inet_hashtables.c:1430
// 锁数量至少填满 2 个 cache line，减少相邻 bucket 锁的 false sharing
nblocks = max(2U * L1_CACHE_BYTES / locksz, 1U);
nblocks = roundup_pow_of_two(nblocks * num_possible_cpus());
```

不是对齐结构体，而是通过增加锁数量让相邻 hash bucket 的锁自然落在不同 cache line。

## 七、对齐的空间代价与权衡设计

对齐是有代价的——padding 浪费空间。内核做了精细的权衡：

### 1. 小对象减半对齐

`calculate_alignment()` 中：
```c
while (size <= ralign / 2)
    ralign /= 2;
```

16 字节对象不需要 64 字节对齐（浪费 75%），对齐到 16 字节即可——一条 cache line 放 4 个对象，命中率不差，空间利用率高。

### 2. UP 下消除 SMP padding

`____cacheline_aligned_in_smp` 在单处理器内核下为空：没有多核就没有 false sharing，不需要浪费空间。

### 3. SLUB 元数据自身 cache line 对齐

```c
// mm/slub.c:4796
create_boot_cache(kmem_cache_node, "kmem_cache_node",
    sizeof(struct kmem_cache_node), SLAB_HWCACHE_ALIGN, 0, 0);
```

分配器自己的管理结构也 cache line 对齐——这些结构在每次分配/释放时都要访问，是最热的路径。

### 4. 合并策略

SLUB 尽量合并相似 cache 以减少碎片。合并条件之一是对齐兼容：

```c
if ((s->size & ~(align - 1)) != s->size)
    continue;  // 对齐不兼容，不合并
```

## 八、设计思考

### 1. 为什么 ARM64 的 ARCH_DMA_MINALIGN 是 128 而不是 64？

ARM64 的 Cache Writeback Granule（CWG）可以大于 L1 cache line。CWG 是 CPU 单次写回内存的最小粒度——如果 DMA buffer 不满足 CWG 对齐，CPU 写回相邻数据时可能覆盖 DMA 正在写入的区域。128 是 ARM big.LITTLE 架构中常见的最大 CWG。

### 2. 为什么 x86 不需要那么大的 DMA 对齐？

x86 有 cache coherent DMA（设备通过 snoop 协议参与 cache 一致性），不存在 CPU 回写覆盖 DMA 数据的问题。因此 `ARCH_KMALLOC_MINALIGN` 只需要 8 字节（`unsigned long long` 的自然对齐）。

### 3. 对齐的三层递进

```
层1：最小正确性对齐（ARCH_SLAB_MINALIGN = 8）
  → 保证基本类型访问不触发异常

层2：DMA 安全对齐（ARCH_KMALLOC_MINALIGN = 128 on ARM64）
  → 保证 kmalloc buffer 可安全用于 DMA

层3：性能优化对齐（SLAB_HWCACHE_ALIGN / ____cacheline_aligned_in_smp）
  → 消除 false sharing，提升多核并发性能
```

### 4. 何时该用对齐

| 场景 | 建议 |
|------|------|
| 多核高频写字段 | `____cacheline_aligned_in_smp` |
| slab 缓存的热路径对象 | `KMEM_CACHE()` + 结构体对齐属性 |
| DMA 缓冲区 | 用 `kmalloc`/`dma_alloc_coherent`，依赖系统保证 |
| SPSC 队列的 producer/consumer | 各自独占 cache line |
| 单核或低频路径 | 不需要额外对齐，保持紧凑 |
