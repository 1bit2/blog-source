## 零、为什么需要虚拟内存

在理解内核实现之前，先明确虚拟内存解决的根本问题：

**物理内存的困境**：如果进程直接使用物理地址，那么——
1. 进程 A 可以通过指针访问进程 B 的数据（无隔离）
2. 内存碎片化后无法分配连续大块（无连续视图）
3. 每个进程必须知道自己代码加载在哪个物理地址（无法重定位）

**虚拟内存的解法**：给每个进程一个独立的、从零开始的地址空间。内核通过页表将虚拟地址映射到物理地址，实现：

| 需求 | 实现方式 |
|------|---------|
| 进程隔离 | 每个进程有自己的页表，看不到其他进程的物理页 |
| 连续视图 | 虚拟地址连续，物理页可以分散在任意位置 |
| 按需分配 | VMA 只记账虚拟范围，首次访问时才通过缺页异常分配物理页 |
| 共享 | 多个进程的页表项指向同一物理页（如 libc.so 的代码段） |

**核心机制链**：VMA（§二）→ mmap/brk（§五）→ 缺页异常（§九）→ 物理页分配（§八）→ 回收/OOM（§十）。其中**缺页异常是串联一切的枢纽**——它把虚拟地址空间的记账、页表翻译、物理页分配全部连接起来。

## 二、虚拟内存核心数据结构

### 2.1 mm_struct — 进程地址空间描述符

每个进程有一个 `mm_struct`（`include/linux/mm_types.h`），是虚拟内存的总管：

```
struct mm_struct
 │
 ├── mmap              VMA链表头(按地址升序)
 ├── mm_rb             VMA红黑树根(用于O(log n)查找)
 ├── map_count         VMA数量
 │
 ├── pgd               顶级页表指针(CR3指向此处)
 ├── task_size          用户空间大小(x86_64: 0x7FFFFFFFF000)
 ├── mmap_base          mmap区域基地址(自顶向下分配时的起始)
 ├── highest_vm_end     最高VMA结束地址
 │
 ├── total_vm           虚拟页总数(所有VMA覆盖的页数)
 ├── locked_vm          mlock锁定页数
 ├── data_vm            数据段页数
 ├── exec_vm            代码段页数
 ├── stack_vm           栈页数
 │
 ├── start_code/end_code       代码段虚拟地址范围
 ├── start_data/end_data       数据段虚拟地址范围
 ├── start_brk/brk             堆起始/当前堆顶
 ├── start_stack               初始栈指针(SP)
 ├── arg_start/arg_end         命令行参数在栈中的位置
 ├── env_start/env_end         环境变量在栈中的位置
 │
 ├── mmap_lock          保护VMA的读写信号量
 ├── mm_users            用户态引用计数(线程数)
 ├── mm_count            内核引用计数
 └── get_unmapped_area   查找空闲虚拟地址的函数指针
```

### 2.2 vm_area_struct — 虚拟内存区域(VMA)

每段连续的虚拟地址范围由一个 VMA 描述：

```
struct vm_area_struct
 │
 ├── vm_start / vm_end         区域范围 [start, end)
 ├── vm_next / vm_prev         VMA链表(按地址排序)
 ├── vm_rb                     VMA红黑树节点
 ├── rb_subtree_gap            子树中最大空闲间隙(用于mmap找空间)
 │
 ├── vm_mm                     所属的mm_struct
 ├── vm_flags                  属性标志(见下表)
 ├── vm_page_prot              页表保护位(构建PTE时用)
 │
 ├── vm_file                   映射的文件(NULL=匿名映射)
 ├── vm_pgoff                  文件内偏移(页为单位)
 ├── vm_ops                    操作方法(fault/map_pages等)
 │
 ├── anon_vma / anon_vma_chain 匿名页反向映射
 └── shared                    文件映射的区间树节点(用于rmap)
```

### 2.3 VM_* 标志（`include/linux/mm.h`）

| 标志 | 值 | 含义 |
|------|------|------|
| VM_READ | 0x01 | 可读 |
| VM_WRITE | 0x02 | 可写 |
| VM_EXEC | 0x04 | 可执行 |
| VM_SHARED | 0x08 | 共享映射(修改写回文件) |
| VM_GROWSDOWN | 0x100 | 向下增长(栈) |
| VM_LOCKED | 0x2000 | 锁定在内存(不被换出) |
| VM_IO | 0x4000 | I/O映射区域 |
| VM_DONTCOPY | 0x20000 | fork时不复制 |
| VM_ACCOUNT | 0x100000 | 计入内存配额 |
| VM_HUGETLB | 0x400000 | 大页映射 |

### 2.4 VMA 的双重索引结构

同一组 VMA 同时存储在**链表**和**红黑树**中：

```
mm_struct
 ├── mmap ──→ VMA_1 ──→ VMA_2 ──→ VMA_3 ──→ NULL   (链表:顺序遍历)
 │            addr=0x400000  0x600000  0x7fff...
 │
 └── mm_rb                                          (红黑树:快速查找)
              ┌───────────┐
              │   VMA_2   │
              │ 0x600000  │
              ├─────┬─────┤
              │     │     │
          ┌───▼──┐ ┌▼────┐
          │VMA_1 │ │VMA_3│
          │0x4000│ │0x7ff│
          └──────┘ └─────┘

 find_vma(addr) 先查 vmacache(per-thread缓存)，
   miss 时走红黑树 O(log n)
 vma_link() 同时插入链表和红黑树
```

> **阅读指引**：§二介绍了"用什么数据结构管理虚拟地址"。§三展示这些 VMA 在进程地址空间中实际长什么样——代码段、堆、栈、mmap 区域各在哪里。

## 四、进程启动时虚拟内存的建立过程

### 4.1 完整调用链

```
用户: execve("/bin/ls", argv, envp)
  │
  ▼
do_execveat_common()                    [fs/exec.c]
  ├── alloc_bprm()
  │     └── bprm_mm_init()
  │           └── __bprm_mm_init()      ★ 创建第1个VMA: 临时栈
  │
  ├── bprm_execve()
  │     └── exec_binprm()
  │           └── search_binary_handler()
  │                 └── load_elf_binary()  [fs/binfmt_elf.c]
  │
  └── load_elf_binary() 内部流程:
        │
        ├── ①  begin_new_exec()           切换到新mm(丢弃旧地址空间)
        ├── ②  setup_new_exec()           设置mmap_base(ASLR布局)
        ├── ③  setup_arg_pages()          ★ 栈VMA定型: 重定位+扩展
        ├── ④  for each PT_LOAD:          ★ 创建代码/数据段VMA
        │       └── elf_map() → do_mmap()
        ├── ⑤  set_brk()                  ★ 创建BSS VMA + 设置堆起点
        │       └── vm_brk_flags() → do_brk_flags()
        ├── ⑥  load_elf_interp()          ★ 加载ld-linux.so(动态链接器)
        ├── ⑦  ARCH_SETUP_ADDITIONAL_PAGES ★ 映射vDSO
        ├── ⑧  create_elf_tables()        在栈上写入argc/argv/envp/auxv
        └── ⑨  填充 mm->start_code/end_code/start_data/end_data/start_stack
```

### 4.2 虚拟内存变化示意

```
  ① __bprm_mm_init():     创建 1 页临时栈 VMA (STACK_TOP_MAX 附近)

  ② begin_new_exec():     旧 mm 的所有 VMA 销毁，切换到新 mm

  ③ setup_arg_pages():    栈 VMA 重定位到 ASLR 随机地址，预扩展 ~128KB

  ④ PT_LOAD 循环:         每个 PT_LOAD 段 → elf_map() → 代码段/数据段 VMA
                           (文件映射, MAP_PRIVATE)

  ⑤ set_brk():            创建 BSS VMA (匿名映射) + 设置 mm->start_brk/brk

  ⑥ load_elf_interp():    加载 ld-linux.so → 代码/数据/BSS 多个 VMA

  ⑦ ARCH_SETUP_ADDITIONAL_PAGES: 映射 [vvar] + [vdso]

  ⑧⑨ create_elf_tables(): 在栈内写入 argc/argv/envp/auxv, 不创建新 VMA
```

```
  x86_64 典型的动态链接程序 (cat /proc/<pid>/maps):

  0x7FFFFFFFFFFF ┌─────────────────────┐
                 │ [stack] (8MB limit) │ VM_READ|VM_WRITE|VM_GROWSDOWN
                 ├─────────────────────┤
                 │ [vvar] 内核只读数据  │

                 ├─────────────────────┤

                 ├─────────────────────┤ mm->brk (初始 = start_brk)

                 │ TEXT (.text)        │ VM_READ|VM_EXEC  (文件映射)
```

## 五、malloc 完整分配流程

> **关键认知**：malloc/brk/mmap 只创建 VMA（虚拟地址记账），**不分配物理页**。物理页在进程首次访问该地址时，由 CPU 缺页异常触发分配（详见 §七 和 §九）。这是 Linux "按需分配"的核心设计。

### 5.1 用户态到内核态全链路

```
┌─────────────────────────────────────────────────────────────────┐
│                        用户空间 (glibc)                          │
├─────────────────────────────────────────────────────────────────┤
│  malloc(size)                                                    │
│      │                                                           │
│      ├─ size < 128KB:                                            │
│      │     ① 检查 tcache/bins 空闲链表                            │
│      │        有合适空闲块? → 直接返回(不调内核!)                  │
│      │     ② 空闲池不够 → brk(当前brk + N) 批量扩堆              │
│      │                                                           │
│      └─ size >= 128KB:                                           │
│            → mmap(NULL, size, PROT_READ|PROT_WRITE,              │
│                   MAP_PRIVATE|MAP_ANONYMOUS, -1, 0)              │
└──────────────────────────────┬──────────────────────────────────┘
                               │ 系统调用
┌──────────────────────────────┴──────────────────────────────────┐
│                        内核空间                                   │
├─────────────────────────────────────────────────────────────────┤
│  brk 路径:                        mmap 路径:                     │
│  ─────────                        ──────────                     │
│  SYSCALL_DEFINE1(brk)             ksys_mmap_pgoff()              │
│      │                                │                          │
│      └─→ do_brk_flags()               └─→ do_mmap()             │
│              │                                │                  │
│              │                          mmap_region()            │
│              │                                │                  │
│              └────────────────────────────────┘                  │
│                              │                                   │
│                    创建/扩展 VMA (vm_area_struct)                 │
│                    【此时不分配物理页!】                          │
└─────────────────────────────────────────────────────────────────┘
                               │ 首次访问时
┌──────────────────────────────┴──────────────────────────────────┐
│                        缺页异常处理                               │
│  CPU → MMU → PTE不存在 → #PF                                    │
│      → exc_page_fault() → do_user_addr_fault()                  │
│          → find_vma() → handle_mm_fault()                       │
│              → __handle_mm_fault() (建立各级页表)                │
│                  → handle_pte_fault()                            │
│                      → do_anonymous_page() (分配物理页+建PTE)    │
│  返回用户态，重新执行指令                                         │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 brk 与 mmap 对比

| 特性 | brk (堆) | mmap |
|------|---------|------|
| 典型触发 | `malloc(< 128KB)` | `malloc(≥ 128KB)` 或直接 `mmap()` |
| 地址增长 | 连续向上，只能在堆顶扩展 | 离散，可在 mmap 区域任意位置 |
| 系统调用 | `brk(new_addr)` | `mmap(addr, len, prot, flags, fd, off)` |
| 核心函数 | `do_brk_flags()` | `do_mmap()` → `mmap_region()` |
| 映射类型 | 只能匿名私有 | 匿名/文件，私有/共享 |
| 释放方式 | `brk(低地址)` → `__do_munmap()` | `munmap()` → `__do_munmap()` |
| 碎片问题 | 堆顶下方空洞无法归还 | 随时可单独释放，无碎片问题 |
| 适用场景 | 频繁小块分配(堆管理器内部切分) | 大块内存、文件映射、共享内存 |

### 5.3 mmap 申请内存的详细流程

```
mmap(NULL, 256KB, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS, -1, 0)
  │
  ▼
do_mmap()                               [mm/mmap.c]
  │
  ├── 1. 参数校验 (len 页对齐, pgoff 溢出检查, map_count 检查)
  │
  ├── 2. get_unmapped_area()            查找空闲虚拟地址
  │     └── arch_get_unmapped_area_topdown()
  │           自顶向下搜索红黑树，利用 rb_subtree_gap 快速剪枝 O(log n)
  │
  ├── 3. 计算 vm_flags
  │
  └── 4. mmap_region()
        │
        ├── may_expand_vm()             检查 RLIMIT_AS 限制
        ├── security_vm_enough_memory() overcommit 检查
        ├── vma_merge()                 尝试与相邻 VMA 合并
        │     成功 → 直接扩展已有 VMA
        │     失败 → vm_area_alloc() 分配新 VMA
        ├── vma_set_anonymous()         标记为匿名映射
        └── vma_link()                  插入链表 + 红黑树

  ★ 此时没有分配物理页! PTE为空! 首次访问触发缺页异常
```

### 5.4 brk 扩展堆的详细流程

```
brk(new_brk)
  │
  ├── new_brk > mm->brk → 堆扩展
  │     └── do_brk_flags()
  │           ├── get_unmapped_area(NULL, addr, len, 0, MAP_FIXED)
  │           ├── may_expand_vm() + security_vm_enough_memory()
  │           ├── vma_merge()  尝试与已有堆 VMA 合并 (最常见路径)
  │           └── 更新 mm->total_vm, mm->data_vm
  │     更新 mm->brk = new_brk
  │
  └── new_brk < mm->brk → 堆收缩
        └── __do_munmap() 裁剪/删除 VMA

  ★ 同样不分配物理页，等缺页异常
```

### 5.5 VMA 合并条件

```
新 VMA 能与相邻 VMA 合并需全部满足:
  ✓ vm_flags 相同     ✓ vm_file 相同(或都是NULL)
  ✓ vm_pgoff 连续     ✓ 无 vm_ops->close
  ✓ anon_vma 兼容     ✓ NUMA 策略相同
  ✓ 非 VM_SPECIAL     ✓ userfaultfd 上下文相同

合并类型:
  前合并: prev->vm_end == addr → 扩展 prev 的 vm_end
  后合并: addr + len == next->vm_start → 缩小 next 的 vm_start
  前后同时合并: prev 和 next 都能合并 → 合并为一个大 VMA
```

## 七、虚拟地址到物理地址映射

> **详细源码分析已移至姊妹篇**：x86_64 四级页表遍历、CR3、PTE 标志位、TLB 工作原理、PCID、TLB Shootdown 等**硬件 + 软件完整路径**，详见 [CPU 页表与缓存机制](CPU页表与缓存机制.md) 第一部分（§2–§8）。
>
> 本节只给出**内存管理视角**的高层概览：理解 VMA 如何通过页表映射到物理页，以及缺页异常如何串联 VMA → 页表 → 物理页分配。

### 7.1 页表层次（速览）

```
虚拟地址 (48 位有效) 拆成 4 段索引：
  PGD[47:39] → PUD[38:30] → PMD[29:21] → PTE[20:12] → offset[11:0]
                                              │
                                              ▼
                                          PFN + offset = 物理地址

CR3 寄存器 = 当前进程 PGD 的物理首地址（进程切换时由 switch_mm_irqs_off 更新）
```

### 7.2 内存管理视角下的 PTE 关键字段

| 位 | 名称 | 内存管理相关的意义 |
|----|------|------------------|
| 0 | Present | 0 = 缺页，由 `handle_pte_fault()` 分派 |
| 1 | R/W | 0 + 写访问 = COW 或写保护缺页 |
| 5 | Accessed | 页面回收时判断"最近是否访问过"（`pte_young`） |
| 6 | Dirty | 文件页回写判断（`pte_dirty`） |
| [51:12] | PFN | 指向伙伴系统分配的物理页帧 |

### 7.3 缺页异常如何串联三层机制

```
用户态访问虚拟地址
    ↓
CPU MMU 遍历页表（CR3 → PGD → PUD → PMD → PTE）
    ↓ PTE Present=0 或权限不足
触发 #PF → arch/x86/mm/fault.c → do_user_addr_fault()
    ↓
handle_mm_fault() → __handle_mm_fault() 逐级分配页表页（伙伴系统 order=0）
    ↓
handle_pte_fault() 根据 PTE 状态分派：
    ├── 匿名页（无 VMA 文件） → do_anonymous_page() → 分配零页
    ├── 文件页（VMA 映射了文件） → do_fault() → 从 page cache 读
    ├── 写只读页 → do_wp_page() → COW 复制
    └── 已换出 → do_swap_page() → 从 swap 换入
    ↓
填写 PTE（Present=1 + PFN + 权限）→ invlpg 刷新 TLB → 重新执行指令
```

> 缺页异常是内存管理的**核心串联点**：它把 VMA 记账（第 1 层）、页表翻译（基础设施）、物理页分配（第 3 层）、页面回收（第 4 层）四层全部串起来。完整调用链详见 [§九 缺页异常全流程](#九缺页异常全流程)。

## 八、物理内存管理

### 8.1 物理内存组织层次

```
┌─────────────────────────────────────────────────────────┐
│  NUMA 节点 (pg_data_t)                                    │
│  ├── node_zones[MAX_NR_ZONES]                           │
│  │     ├── ZONE_DMA      (0~16MB, ISA DMA)              │
│  │     ├── ZONE_DMA32    (0~4GB, 32位 DMA)              │
│  │     ├── ZONE_NORMAL   (直接映射区，内核正常使用)       │
│  │     ├── ZONE_HIGHMEM  (仅32位：超出内核线性映射范围)   │
│  │     └── ZONE_MOVABLE  (可迁移页，支持内存热插拔)       │
│  ├── kswapd              (每节点一个回收线程)             │
│  └── node_zonelists      (分配回退顺序)                  │
└─────────────────────────────────────────────────────────┘
```

### 8.2 水位线机制

```
每个 zone 的空闲页面量有三条水位线:

  pages_high  ← kswapd 回收到此停止
  pages_low   ← 低于此唤醒 kswapd
  pages_min   ← 紧急底线：低于此只允许 GFP_ATOMIC
  (已耗尽)    ← 直接回收 / OOM killer
```

### 8.3 伙伴系统 (mm/page_alloc.c)

```
zone->free_area[MAX_ORDER] 结构 (MAX_ORDER=11):

  order 0:  1页  (4KB)    free_list → [page] → [page] → ...
  order 1:  2页  (8KB)    free_list → [page] → ...
  ...
  order 10: 1024页(4MB)   free_list → [page] → ...

分配: __rmqueue_smallest() — 逐级向上找，找到后 expand() 拆分
释放: __free_one_page()    — XOR 算伙伴PFN，逐级向上合并
```

### 8.4 SLUB 分配器 (mm/slub.c)

```
struct kmem_cache:
  ├── cpu_slab (per-CPU)
  │     └── freelist → 快速路径: cmpxchg_double 无锁分配 (~20ns)
  ├── node[NUMA_NODE]
  │     └── partial → 慢速路径: 从节点partial链取slab
  └── 全部耗尽 → new_slab() → 伙伴系统分配新页

kmalloc(size) 路由:
  ≤ 8KB → SLUB 缓存 (桶: 8/16/32/64/96/128/.../8K)
  > 8KB → 直接用伙伴系统 alloc_pages()
```

### 8.5 vmalloc vs kmalloc

| 特性 | kmalloc | vmalloc |
|------|---------|---------|
| 物理连续性 | **物理连续** | 物理不连续，虚拟连续 |
| 地址空间 | 内核直接映射区 | VMALLOC_START ~ VMALLOC_END |
| 速度 | 快 (per-CPU 缓存) | 慢 (需要修改页表) |
| 适用场景 | 小型、频繁的内核对象 | 大型缓冲区 (如模块加载) |

## 九、缺页异常全流程

### 9.1 缺页异常触发条件

CPU 访问虚拟地址时，MMU 查页表发现以下任一情况即触发 `#PF`（Page Fault，中断向量 14）：

| 情况 | 页表状态 | 含义 |
|------|---------|------|
| 首次访问 | PTE 为空（全零） | 从未建立映射，需要分配物理页 |
| 被换出 | PTE 非空但 Present=0 | 页面在 swap 中，需读回 |
| 权限不足 | PTE 存在但写只读页 | COW 或真正的权限错误 |
| NUMA 陷阱 | PTE 存在但 protnone | 内核故意清权限用于 NUMA 平衡 |

### 9.2 完整调用链路

```
exc_page_fault()                           ← arch/x86/mm/fault.c, IDT 入口
  │ 读 CR2 获取故障虚拟地址
  │
  ├─ 内核地址 → do_kern_addr_fault()       ← vmalloc fault / oops
  └─ 用户地址 → do_user_addr_fault()       ← 用户态缺页主处理
       │
       ├── 安全检查 (SMAP/异常嵌套/中断上下文)
       ├── mmap_read_lock(mm)               ← 获取 VMA 读锁
       ├── find_vma(mm, addr)               ← 在红黑树中查找 VMA
       │     vma == NULL       → SIGSEGV (SEGV_MAPERR, 地址无映射)
       │     addr < vm_start   → 尝试扩展栈，失败 → SIGSEGV
       │     权限不匹配       → SIGSEGV (SEGV_ACCERR, 无权限)
       │
       └── handle_mm_fault(vma, addr, flags) ← 架构无关主入口
             │
             ├── HugeTLB VMA → hugetlb_fault()  (预留式大页)
             └── 普通 VMA    → __handle_mm_fault()
                                    │
                                    └── handle_pte_fault()
```

### 9.3 缺页处理函数操作的是什么地址？

`__handle_mm_fault` 和 `handle_pte_fault` 并不是"直接操作物理地址"，而是**同时涉及虚拟地址和物理地址**，各有不同用途：

```
┌──────────────────────┬──────────────────────────────────────────┐
│  操作                 │  地址类型                                 │
├──────────────────────┼──────────────────────────────────────────┤
│  vmf->address        │  用户态虚拟地址（CR2寄存器值）             │
│                      │  用于计算各级页表的索引                    │
│                      │  如 PGD索引=[47:39], PTE索引=[20:12]      │
├──────────────────────┼──────────────────────────────────────────┤
│  pgd/pud/pmd/pte 指针│  内核虚拟地址                              │
│                      │  pgd_offset(mm, addr) 返回内核虚拟地址    │
│                      │  pud_alloc() 分配物理页，通过内核线性映射  │
│                      │  (PAGE_OFFSET + 物理地址) 返回内核虚拟地址 │
├──────────────────────┼──────────────────────────────────────────┤
│  页表项内容 [51:12]   │  物理页帧号 (PFN)，给 MMU 硬件看          │
│                      │  p4d_populate() 把 __pa(new) 写入条目     │
│                      │  mk_pte(page, prot) 把 page_to_pfn(page) │
│                      │  打包成 PTE 值写入页表                     │
├──────────────────────┼──────────────────────────────────────────┤
│  alloc_page()        │  分配物理页，返回 struct page *            │
│                      │  struct page 是内核数据结构(内核虚拟地址)  │
│                      │  page_to_pfn(page) 可得到物理页帧号       │
└──────────────────────┴──────────────────────────────────────────┘

核心理解：
  内核代码运行在虚拟地址空间，通过虚拟地址读写页表结构。
  但页表项中填写的值（PFN）是物理地址——这是给 CPU 的 MMU 硬件看的。
  最终效果：建立 "用户虚拟地址 → 物理页帧" 的硬件映射。
```

**do_anonymous_page() 的地址转换全过程**:

```
用户写 *p = 42，触发缺页:
  p 的值（如 0x7ffff000）= 用户虚拟地址 → vmf->address

handle_pte_fault() → do_anonymous_page():
  ① alloc_zeroed_user_highpage_movable(vma, addr)
     → 伙伴系统分配一个物理页帧
     → 返回 struct page *page (内核虚拟地址)

  ② mk_pte(page, vma->vm_page_prot)
     → page_to_pfn(page) 得到物理 PFN (如 0x12345)
     → PFN << 12 | 权限位 → 组装成 64 位 PTE 值

  ③ set_pte_at(mm, addr, vmf->pte, entry)
     → 把 PTE 值写入页表（通过内核虚拟地址 vmf->pte 访问）

  ④ 返回用户态，CPU 重新执行 *p = 42
     → MMU 查页表，从 PTE 读出 PFN=0x12345
     → 物理地址 = 0x12345 << 12 + offset = 0x12345000 + 偏移
     → 写入成功
```

### 9.4 __handle_mm_fault() — 多级页表遍历与大页处理

**核心职责**：沿 PGD → P4D → PUD → PMD 逐级查找/分配页表项，最终调用 `handle_pte_fault()` 处理 PTE 级别的映射。

```
__handle_mm_fault() 执行流程:

  1. pgd_offset(mm, addr)    ← PGD 条目始终存在（进程创建时已分配）
  2. p4d_alloc(mm, pgd, addr) ← P4D（x86_64 四级模式下折叠，不实际分配）
  3. pud_alloc(mm, p4d, addr) ← PUD 不存在？分配一个新的 PUD 页表页
  │    ├── PUD 为空 + THP 启用 → 尝试 1GB 大页 (create_huge_pud)
  │    └── PUD 已是大页 → 直接处理（设 Accessed/COW/NUMA），返回
  │
  4. pmd_alloc(mm, pud, addr) ← PMD 不存在？分配一个新的 PMD 页表页
  │    ├── PMD 为空 + THP 启用 → 尝试 2MB 大页 (create_huge_pmd)
  │    │     分配失败 → VM_FAULT_FALLBACK → 继续用 4KB 普通页
  │    └── PMD 已是大页 → 直接处理，返回
  │
  5. → handle_pte_fault()     ← 到达最底层 PTE
```

#### 为什么大页能提高性能？

CPU 内的 TLB 缓存虚拟→物理地址的翻译结果，但容量很小（典型 x86 CPU 约 64 条数据 TLB 表项）：

| 页大小 | 覆盖 8MB 内存 | 需要 TLB 表项 | miss 时查页表级数 |
|--------|-------------|-------------|----------------|
| 4KB 普通页 | 2048 页 | 2048 条 → 远超容量 → 频繁 miss | 4 级 |
| 2MB 大页 | 4 页 | 4 条 → 完全够用 → 高命中 | 3 级（省 PTE） |
| 1GB 大页 | 1 页 | 1 条 | 2 级（省 PMD+PTE） |

**页越大 → 覆盖同样内存的 TLB 条目越少 → 命中率越高 → 地址翻译越快**。

### 9.5 handle_pte_fault() — PTE 级决策分派

在 `__handle_mm_fault()` 走完多级页表到达 PTE 层后，`handle_pte_fault()` 根据 PTE 当前状态分派到不同的处理函数：

```
handle_pte_fault() 决策树:

    PTE 存在吗？
    │
    ├── 否 (pte == NULL 或 pte_none):
    │     即"首次访问"，从未建立过映射
    │     │
    │     ├── 匿名 VMA → do_anonymous_page()
    │     │     读: 映射到全局共享零页（只读，节省内存）
    │     │     写: 分配物理页 + 清零 + 建立可写 PTE
    │     │
    │     └── 文件 VMA → do_fault()
    │           调 vm_ops->fault 从 page cache/磁盘读
    │
    ├── PTE 非空但 Present=0:
    │     └── do_swap_page()   ← 从 swap 读回内存
    │
    ├── Present=1 但 protnone + VMA 可访问:
    │     └── do_numa_page()   ← NUMA 平衡迁移
    │
    └── Present=1 且有正常权限位:
          加 PTL 锁 → 二次验证 PTE 未变
          │
          ├── 写 + PTE 只读 → do_wp_page()  ← COW 写时复制
          └── 读或其他 → 设 Accessed/Dirty 位即可

二次验证原理（多核并发安全）:
  CPU-A 快照 PTE（无锁）→ 加锁 → 再读 PTE 与快照比较
  如果不一致（CPU-B 已处理完同一缺页）→ 释放锁，直接返回
  这避免了重复处理同一缺页
```

### 9.6 do_anonymous_page() — 匿名页首次访问

```
读访问:
  → PTE 指向全局共享零页 (my_zero_pfn)，标记只读
  → 所有进程的未写匿名页共享同一个零页 (节省内存)
  → 后续写入时再触发 COW → do_wp_page

写访问:
  → pte_alloc()                             分配 PTE 子表（如果需要）
  → anon_vma_prepare()                      建立反向映射基础设施
  → alloc_zeroed_user_highpage_movable()    分配物理页并清零
  → mem_cgroup_charge()                     内存 cgroup 记账
  → page_add_new_anon_rmap()               建立反向映射
  → lru_cache_add_inactive()               加入 LRU 链表（回收用）
  → mk_pte() + set_pte_at()               构建 PTE 并原子写入页表
  → update_mmu_cache()                      通知 TLB
```

### 9.7 do_wp_page() — 写时复制 (COW)

```
fork 后父子共享物理页(只读PTE):

子进程写入 → #PF → do_wp_page()
  ├── 页只有自己映射 (mapcount=0)?
  │     └── wp_page_reuse() → 直接改 PTE 为可写 (省复制，最快路径)
  │
  ├── PageAnon + 引用计数=1 + mapcount=1?
  │     └── wp_page_reuse() → 也可以复用
  │
  └── 多人映射?
        └── wp_page_copy()
              alloc_page_vma() → 分配新物理页
              copy_user_highpage() → 复制内容
              新 PTE 指向新页(可写) → 旧页引用计数减 1
```

### 9.8 do_fault() — 文件映射缺页

```
→ vma->vm_ops->fault() → filemap_fault()
  → find_get_page() 查 page cache
    命中 (minor fault): 直接使用，建 PTE → 返回
    未命中 (major fault): readpage() 从磁盘读入 → 加入 page cache → 建 PTE

三种子路径:
  ├── 读 → do_read_fault()     只读 PTE
  ├── 写私有 → do_cow_fault()  复制一份再写
  └── 写共享 → do_shared_fault() 直接写 + 标脏 + writeback
```

### 9.9 完整缺页处理流程图

```
                       CPU 访问虚拟地址
                             │
                       TLB 命中? ─是→ 直接访问物理内存
                             │否
                       硬件遍历页表 PGD→P4D→PUD→PMD→PTE
                             │
                       PTE 有效? ─是→ 填入TLB，访问内存
                             │否
                       触发 #PF 异常 (中断向量 14)
                             │
                    exc_page_fault()
                    读 CR2 获取故障地址
                             │
                  用户地址?───┤───内核地址?
                       │              │
              do_user_addr_fault()   do_kern_addr_fault()
                       │
                  mmap_read_lock(mm)
                  find_vma(mm, addr)
                       │
              ┌────────┼────────┬──────────┐
              ↓        ↓        ↓          ↓
          vma==NULL  addr<     权限       正常
                    vm_start   错误
              │        │        │          │
              ↓        ↓        ↓          ↓
          SIGSEGV  扩展栈?  SIGSEGV   handle_mm_fault()
         SEGV_MAPERR        SEGV_ACCERR    │
                                            ↓
                                  __handle_mm_fault()
                                  逐级遍历/分配页表:
                                  PGD → P4D → PUD → PMD
                                  (PUD/PMD 层检查大页 THP)
                                            │
                                  handle_pte_fault()
                                  根据 PTE 状态分派:
                                            │
                  ┌─────────┬───────────┬───────────┬───────────┐
                  ↓         ↓           ↓           ↓           ↓
            do_anonymous  do_fault   do_wp_page  do_swap    do_numa
            _page()      (文件映射)   (COW)      _page()    _page()
            (首次匿名)   (首次文件)   (写只读)   (换入)     (NUMA迁移)
                  │         │           │           │           │
                  └─────────┴───────────┴───────────┴───────────┘
                                  │
                          分配物理页 + 填 PTE + 刷 TLB
                                  │
                          返回用户态，重新执行触发指令
```

## 十、页面回收与 OOM Killer

### 10.1 LRU 链表

```
lruvec (每个 NUMA 节点):
  ├── active_anon    活跃匿名页
  ├── inactive_anon  不活跃匿名页 (回收候选)
  ├── active_file    活跃文件页
  ├── inactive_file  不活跃文件页 (最优先回收)
  └── unevictable    不可驱逐页 (mlock 等)

回收优先级:
  1. inactive_file (丢弃干净页缓存，无需写回)
  2. 脏文件页: 写回磁盘后回收
  3. 匿名页: 换出到 swap 后回收
  4. active → inactive: 二次机会算法 (检查 Accessed 位)
```

### 10.2 回收触发路径

```
__alloc_pages_slowpath() (分配器内存不足时)
    │
    ├─→ 唤醒 kswapd (后台异步)
    │     → balance_pgdat() → shrink_node() → shrink_lruvec()
    │
    ├─→ 直接回收 (同步阻塞)
    │     → try_to_free_pages() → shrink_zones()
    │
    └─→ OOM killer (最后手段)
          → out_of_memory() → select_bad_process() → oom_kill_process()
```

### 10.3 OOM Killer 评分

```
oom_badness() 评分:
  base_score = RSS + swap使用量 + 页表占用
  最终得分 = base_score + oom_score_adj × totalpages / 1000

  oom_score_adj 范围 [-1000, 1000]:
    -1000: 永不被杀    0: 默认    1000: 最优先被杀
  /proc/<pid>/oom_score_adj 可调整
```

## 十二、常见内存问题

### 12.1 OOM

```
现象: dmesg 中 "Out of memory: Killed process XXX"
原因: 物理内存 + swap 耗尽，所有回收手段失败
排查: dmesg | grep -i "oom"
      /proc/meminfo, /proc/<pid>/status (VmRSS, VmSwap)
```

### 12.2 内存泄漏

```
用户态: malloc/new 后未 free/delete → RSS 持续增长
  工具: valgrind, AddressSanitizer, /proc/<pid>/smaps

内核态: kmalloc 后未 kfree, get_page 后未 put_page
  工具: kmemleak (CONFIG_DEBUG_KMEMLEAK), /proc/slabinfo
```

### 12.3 内存碎片

```
外部碎片 (伙伴系统): 空闲页分散，大 order 分配失败
  查看: cat /proc/buddyinfo
  对策: 内存压缩 (compaction), 迁移类型隔离

内部碎片 (Slab): 对象大小不是桶大小的整数 → 浪费空间
  查看: slabtop
```

### 12.4 其他问题

| 问题 | 现象 | 关键参数/工具 |
|------|------|--------------|
| 栈溢出 | 用户态 SIGSEGV / 内核态 panic | RLIMIT_STACK, CONFIG_VMAP_STACK |
| UAF/Double-Free | 不确定行为/堆损坏 | ASAN, KASAN, SLUB_DEBUG |
| Swap 抖动 | 系统极慢，si/so 高 | `vmstat 1`, swappiness, min_free_kbytes |
| VMA 数量超限 | mmap 返回 ENOMEM | max_map_count (默认 65530) |

## 十三、vmstat / top 内存统计的内核实现

用户空间的 `vmstat`、`top` 命令并不直接读硬件寄存器，而是读取 `/proc` 下的伪文件，这些文件由内核函数填充。

### 13.1 数据源总览

```
用户命令          读取的 /proc 文件              内核填充函数
──────────       ─────────────────            ──────────────────
vmstat           /proc/meminfo                meminfo_proc_show()
                 /proc/vmstat                 vmstat_show()

top 系统级       /proc/meminfo                meminfo_proc_show()

top 进程级       /proc/<pid>/stat             do_task_stat()
                 /proc/<pid>/statm            proc_pid_statm() → task_statm()
                 /proc/<pid>/status           task_mem()
```

### 13.2 系统级内存统计 — `/proc/meminfo`

**填充函数**: `fs/proc/meminfo.c` → `meminfo_proc_show()`

```
meminfo_proc_show() 核心流程:

  ① si_meminfo(&i)      ← mm/page_alloc.c, 填充 struct sysinfo
  ② si_swapinfo(&i)     ← mm/swapfile.c, 填充 swap 信息
  ③ 逐项输出各计数器
```

**各字段的内核来源**:

| 字段 | 内核来源 | 含义 |
|------|---------|------|
| **MemTotal** | `si_meminfo() → totalram_pages()` | 系统可用物理内存总量（不含保留页） |
| **MemFree** | `si_meminfo() → global_zone_page_state(NR_FREE_PAGES)` | 空闲物理页，各 zone 空闲页之和 |
| **MemAvailable** | `si_mem_available()` | 估算可用内存 = Free + 可回收的 pagecache + 可回收的 slab |
| **Buffers** | `si_meminfo() → nr_blockdev_pages()` | 块设备的页缓存（裸设备 buffer cache） |
| **Cached** | `NR_FILE_PAGES - total_swapcache_pages() - bufferram` | 文件页缓存（减去 swap cache 和 buffer） |
| **SwapTotal** | `si_swapinfo() → total_swap_pages` | swap 空间总量 |
| **SwapFree** | `si_swapinfo() → nr_swap_pages` | swap 空闲量 |
| **Active** | `NR_LRU_BASE + LRU_ACTIVE_ANON + LRU_ACTIVE_FILE` | LRU 活跃页（最近被访问的匿名页+文件页） |
| **Inactive** | `NR_LRU_BASE + LRU_INACTIVE_ANON + LRU_INACTIVE_FILE` | LRU 不活跃页（回收候选） |
| **AnonPages** | `global_node_page_state(NR_ANON_MAPPED)` | 匿名页（堆/栈/匿名mmap） |
| **Mapped** | `global_node_page_state(NR_FILE_MAPPED)` | 已映射到用户空间的文件页 |
| **Shmem** | `si_meminfo() → NR_SHMEM` | 共享内存 + tmpfs |
| **Slab** | `NR_SLAB_RECLAIMABLE_B + NR_SLAB_UNRECLAIMABLE_B` | 内核 slab 分配器占用 |
| **PageTables** | `global_node_page_state(NR_PAGETABLE)` | 页表本身占用的内存 |
| **Dirty** | `global_node_page_state(NR_FILE_DIRTY)` | 脏页（已修改还未写回磁盘的文件页） |

**`top` 命令的 Mem/Swap 行**（读 `/proc/meminfo`）:

```
top 显示:
  Mem:  7912232K total,  2401372K used,  5510860K free,  4472K buffers
  Swap: 1048572K total,       0K used,  1048572K free, 938028K cached

计算方式:
  Mem total  = MemTotal           ← totalram_pages() × PAGE_SIZE
  Mem free   = MemFree            ← NR_FREE_PAGES × PAGE_SIZE
  Mem used   = MemTotal - MemFree - Buffers - Cached
  buffers    = Buffers            ← nr_blockdev_pages()
  cached     = Cached + Slab(SReclaimable)

  Swap total = SwapTotal          ← total_swap_pages × PAGE_SIZE
  Swap free  = SwapFree           ← nr_swap_pages × PAGE_SIZE
  Swap used  = SwapTotal - SwapFree
```

### 13.3 vmstat 命令的内存列

`vmstat` 读取 `/proc/meminfo` 和 `/proc/vmstat`：

```
$ vmstat 1
procs -----------memory---------- ---swap-- -----io----
 r  b   swpd   free   buff  cache   si   so    bi    bo
 1  0      0 551086   4472 938028    0    0     5    10

字段           /proc 来源                     内核计数器
──────        ─────────                      ──────────
swpd          SwapTotal - SwapFree           total_swap_pages - nr_swap_pages
free          MemFree                        NR_FREE_PAGES
buff          Buffers                        nr_blockdev_pages()
cache         Cached + SReclaimable          NR_FILE_PAGES - swap_cache - buffer + SReclaimable

si (swap in)  /proc/vmstat → pswpin          vm_event_states.event[PSWPIN]
so (swap out) /proc/vmstat → pswpout         vm_event_states.event[PSWPOUT]
bi (block in) /proc/vmstat → pgpgin          vm_event_states.event[PGPGIN]
bo (block out)/proc/vmstat → pgpgout         vm_event_states.event[PGPGOUT]
```

`/proc/vmstat` 由 `mm/vmstat.c` → `vmstat_show()` 填充，数据来源是两组全局原子计数器：

```
atomic_long_t vm_zone_stat[NR_VM_ZONE_STAT_ITEMS]  ← zone 级别计数器
atomic_long_t vm_node_stat[NR_VM_NODE_STAT_ITEMS]   ← node 级别计数器

更新路径: 各子系统操作内存时 → 更新 per-CPU 计数器
          → 定期/fold 到全局原子变量
          → 读 /proc/vmstat 时汇总输出
```

### 13.4 进程级内存统计 — top 的 VIRT/RES/SHR

top 进程列表中每个进程的内存信息来自 `/proc/<pid>/statm` 和 `/proc/<pid>/status`。

**核心函数**: `fs/proc/task_mmu.c`

```c
// VIRT (虚拟内存总量)
unsigned long task_vsize(struct mm_struct *mm)
{
    return PAGE_SIZE * mm->total_vm;   // total_vm 是所有 VMA 页数之和
}

// statm 各字段
unsigned long task_statm(struct mm_struct *mm,
    unsigned long *shared, unsigned long *text,
    unsigned long *data, unsigned long *resident)
{
    *shared   = get_mm_counter(mm, MM_FILEPAGES)   // 文件映射页
              + get_mm_counter(mm, MM_SHMEMPAGES);  // 共享内存页
    *text     = (end_code - start_code) >> PAGE_SHIFT;  // 代码段
    *data     = mm->data_vm + mm->stack_vm;              // 数据段+栈
    *resident = *shared + get_mm_counter(mm, MM_ANONPAGES); // RSS
    return mm->total_vm;                                     // VIRT
}
```

**各字段解释**:

| top 列 | /proc 来源 | 内核变量 | 含义 |
|--------|-----------|---------|------|
| **VIRT** | `/proc/<pid>/stat` 第23字段 (vsize) | `mm->total_vm × PAGE_SIZE` | 虚拟内存总量：所有 VMA 的 `[vm_start, vm_end)` 范围之和 |
| **RES** | `/proc/<pid>/statm` 第2字段 | `MM_FILEPAGES + MM_ANONPAGES + MM_SHMEMPAGES` | 常驻内存(RSS)：实际占用物理页数 |
| **SHR** | `/proc/<pid>/statm` 第3字段 | `MM_FILEPAGES + MM_SHMEMPAGES` | 共享内存：文件映射页 + 共享内存页 |
| **%MEM** | RES / MemTotal | `get_mm_rss(mm) / totalram_pages()` | RES 占物理内存百分比 |

### 13.5 VIRT (虚拟内存) 的统计原理

VIRT 是最容易误解的指标。它**不代表实际内存消耗**，只是所有 VMA 虚拟地址范围的总和。

**`mm->total_vm` 的更新时机**（`mm/mmap.c`）:

```
vm_stat_account(mm, flags, npages):
    mm->total_vm += npages;        ← 每次 mmap/brk 增加 VMA 时 +
    if (is_exec_mapping(flags))
        mm->exec_vm += npages;     ← 可执行段(代码)
    else if (is_stack_mapping(flags))
        mm->stack_vm += npages;    ← 栈
    else if (is_data_mapping(flags))
        mm->data_vm += npages;     ← 数据段

调用场景:
  mmap_region()     → vm_stat_account(mm, vm_flags, len >> PAGE_SHIFT)   加
  do_brk_flags()    → mm->total_vm += len >> PAGE_SHIFT                 加
  expand_stack()    → vm_stat_account(mm, vma->vm_flags, grow)           加
  __do_munmap()     → vm_stat_account(mm, vma->vm_flags, -nrpages)      减
```

**VIRT 大不代表实际占用多**：

```
例: 一个进程 mmap 了 1GB 匿名区域，但只写了 1 个字节

  VIRT = 1GB + 代码段 + 堆 + 栈 + 共享库 ≈ 1.1GB     ← 很大
  RES  = 实际触发缺页分配的物理页数 × 4KB ≈ 4KB        ← 很小

原因: mmap 只创建 VMA (mm->total_vm += 页数)，不分配物理页。
      物理页在首次访问时通过缺页异常按需分配。
```

### 13.6 /proc/<pid>/status 完整输出

`fs/proc/task_mmu.c` → `task_mem()` 填充 `/proc/<pid>/status` 中的内存行：

| 字段 | 内核来源 | 含义 |
|------|---------|------|
| VmPeak | `max(mm->hiwater_vm, mm->total_vm)` | 虚拟内存峰值 |
| VmSize | `mm->total_vm` | 当前虚拟内存 |
| VmLck | `mm->locked_vm` | 被 mlock 锁定的页（不可被换出） |
| VmHWM | `max(mm->hiwater_rss, get_mm_rss(mm))` | RSS 峰值 (High Water Mark) |
| VmRSS | `get_mm_rss(mm)` | 当前 RSS |
| RssAnon | `get_mm_counter(mm, MM_ANONPAGES)` | 匿名页 RSS |
| RssFile | `get_mm_counter(mm, MM_FILEPAGES)` | 文件映射 RSS |
| RssShmem | `get_mm_counter(mm, MM_SHMEMPAGES)` | 共享内存 RSS |
| VmData | `mm->data_vm` | 数据段虚拟大小 |
| VmStk | `mm->stack_vm` | 栈虚拟大小 |
| VmExe | `end_code - start_code` | 代码段大小 |
| VmSwap | `get_mm_counter(mm, MM_SWAPENTS)` | 被换出到 swap 的页数 |
| VmPTE | `mm_pgtables_bytes(mm)` | 页表本身占的内存 |

### 13.7 RSS 计数器的更新时机

RSS 是 `mm->rss_stat.count[]` 中三个原子计数器的和，在缺页/释放时实时更新：

```
增加 RSS (缺页分配物理页时):
  do_anonymous_page()  → add_mm_counter(mm, MM_ANONPAGES, 1)
  do_fault()           → add_mm_counter(mm, MM_FILEPAGES, 1)
  do_swap_page()       → add_mm_counter(mm, MM_ANONPAGES, 1)

减少 RSS (释放/换出物理页时):
  unmap_page_range()   → add_mm_counter(mm, MM_ANONPAGES, -1)
  try_to_unmap_one()   → add_mm_counter(mm, MM_ANONPAGES, -1)
  swap_out            → add_mm_counter(mm, MM_SWAPENTS, 1)
                         add_mm_counter(mm, MM_ANONPAGES, -1)
```

### 14.2 核心原理清单

| # | 原理 | 一句话解释 | 对应源码 |
|---|------|-----------|---------|
| 1 | 虚拟地址空间 | 每个进程有独立的虚拟地址空间，通过页表映射到物理内存 | `mm_struct`, `pgd` |
| 2 | 延迟分配 | mmap/brk只创建VMA不分配物理页，首次访问时缺页分配 | `mmap_region()` → `do_anonymous_page()` |
| 3 | 缺页异常 | CPU访问未映射的虚拟地址时触发#PF，内核分配页并填页表 | `exc_page_fault` → `handle_mm_fault` |

| 6 | 伙伴系统 | 物理页按2^n组织，分裂/合并解决外部碎片 | `__alloc_pages` → `__rmqueue_smallest` |

| 8 | 反向映射(rmap) | 从物理页找到所有映射它的PTE（回收时需要） | `anon_vma`, `address_space->i_mmap` |
| 9 | 内存cgroup | 按cgroup限制和统计内存使用 | `mem_cgroup_charge`, `memory.max` |
| 10 | OOM Killer | 所有手段耗尽后杀死内存消耗最大的进程 | `out_of_memory` → `oom_badness` |
