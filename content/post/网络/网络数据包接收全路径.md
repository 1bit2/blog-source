# 网络数据包接收全路径：从网卡到用户 Socket

> 基于 Linux 5.15.78 内核源码，完整分析一个网络数据包从网卡硬件到用户态 `recv()` 的六层处理过程。
> 三次握手参见 [TCP三次握手源码分析](TCP三次握手源码分析.md)，数据就绪与 Epoll 唤醒参见 [TCP数据就绪与Epoll唤醒](TCP数据就绪与Epoll唤醒.md)，发送路径参见 [UDP发送路径](UDP发送路径.md)。

---

## 全局总览

```
网卡硬件 → ① 硬中断+NAPI软中断 → ② 网络核心层(netif_receive_skb)
         → ③ IP层(ip_rcv) → ④ 传输层(tcp_v4_rcv / udp_rcv)
         → ⑤ Socket接收队列(sk_receive_queue) → ⑥ 用户态read/recv
```

## 一、网卡驱动层：硬中断 → NAPI 调度

### 1.0 为什么需要 NAPI

NAPI（**New API**）是 Linux 2.6 引入的网络收包框架，用来解决旧模型的**中断风暴**问题。

**旧模型（纯中断驱动）**——每个包一次硬中断：

```
包1到达 → 硬中断 → 驱动收包 → 协议栈处理 → 中断返回
包2到达 → 硬中断 → 驱动收包 → 协议栈处理 → 中断返回
包3到达 → 硬中断 → 驱动收包 → 协议栈处理 → 中断返回
...（高流量时每秒几十万次硬中断，CPU 时间耗在中断上下文切换上）
```

**NAPI 模型（中断 + 轮询混合）**——一次硬中断 + 批量轮询收包：

```
包1到达 → 硬中断（只做：关中断 + 调度NAPI）
                ↓
         软中断: net_rx_action()
           → 驱动 poll(): 一次性从 Ring Buffer 批量取 N 个包
           → 处理完 → 重新开启硬中断
                ↓
         后续的包到达不再触发硬中断（因为中断已关）
         → 等 poll 循环自动取走
```

核心思想：**高负载时从"每包一次中断"切换到"关中断批量轮询"，低负载时恢复中断模式**。NAPI 的生命周期三步：`napi_schedule()`（硬中断中调度）→ `net_rx_action()`（软中断中 poll）→ `napi_complete_done()`（处理完后重开中断）。

### 1.1 硬中断：最小化处理

网卡收到数据帧后，DMA 将数据写入 ring buffer，触发硬中断。驱动的中断处理函数做的事情极少——**只是调度 NAPI 轮询**，然后立即返回，避免长时间关中断：

```c
// 典型的驱动中断处理（伪代码）
irqreturn_t driver_interrupt(int irq, void *dev_id) {
    napi_schedule(&adapter->napi);
    return IRQ_HANDLED;
}
```

### 1.2 NAPI 调度：从硬中断到软中断

`napi_schedule()` 是 `include/linux/netdevice.h` 中的内联函数：

```c
// include/linux/netdevice.h:472
static inline void napi_schedule(struct napi_struct *n)
{
    if (napi_schedule_prep(n))    // 原子置位 NAPI_STATE_SCHED，防重入
        __napi_schedule(n);
}
```

`__napi_schedule()` → `____napi_schedule()`，核心操作：

```c
// net/core/dev.c:4670
static inline void ____napi_schedule(struct softnet_data *sd,
                     struct napi_struct *napi)
{
    struct task_struct *thread;

    // 线程化NAPI模式：唤醒专用内核线程
    if (test_bit(NAPI_STATE_THREADED, &napi->state)) {
        thread = READ_ONCE(napi->thread);
        if (thread) {
            wake_up_process(thread);
            return;
        }
    }

    // 标准模式：将NAPI挂到per-CPU的poll_list，触发软中断
    list_add_tail(&napi->poll_list, &sd->poll_list);
    __raise_softirq_irqoff(NET_RX_SOFTIRQ);
}
```

**设计要点**：每个 CPU 有独立的 `softnet_data`，`poll_list` 链表管理该 CPU 上待处理的 NAPI。硬中断结束后，`NET_RX_SOFTIRQ` 软中断将在合适时机被调度执行。

### 1.3 `struct napi_struct` 核心结构

```c
// include/linux/netdevice.h:341
struct napi_struct {
    struct list_head    poll_list;       // 链入 softnet_data.poll_list
    unsigned long       state;           // NAPI_STATE_SCHED 等状态位
    int                 (*poll)(struct napi_struct *, int);  // 驱动的poll函数
    struct net_device   *dev;            // 关联的网络设备
    struct gro_list     gro_hash[GRO_HASH_BUCKETS];  // GRO哈希表
    struct sk_buff      *skb;
    struct list_head    rx_list;         // GRO_NORMAL 待处理的 skb
    int                 weight;          // 单次poll的budget（通常64）
    struct task_struct  *thread;         // 可选的专用内核线程（threaded NAPI）
    // ...
};
```

#### napi_struct 的构造与所有权

`napi_struct` **不是动态 malloc 的**，而是嵌在驱动的 adapter/queue 结构体中，由驱动在 probe 阶段通过 `netif_napi_add()` 初始化：

```c
// 单队列网卡（e1000）：adapter 内嵌 1 个 napi_struct
// drivers/net/ethernet/intel/e1000/e1000_main.c:1015
netif_napi_add(netdev, &adapter->napi, e1000_clean, 64);
//                      ^^^^^^^^^^^^^^  ^^^^^^^^^^^  ^^
//                      驱动内嵌的napi   poll回调      weight

// 多队列网卡（mlx4）：每个 RX 完成队列(CQ) 内嵌 1 个 napi_struct
// drivers/net/ethernet/mellanox/mlx4/en_cq.c:156
netif_napi_add(cq->dev, &cq->napi, mlx4_en_poll_rx_cq, 64);
// 网卡有 N 个 RX 队列 → N 个 CQ → N 个独立的 napi_struct
```

`netif_napi_add()`（`net/core/dev.c:7426`）做的核心工作：设置 `poll` 回调和 `weight`、关联 `dev`、初始化 `poll_list` 和 GRO 哈希、将 napi 挂到 `dev->napi_list`。

#### softnet_data：严格的 per-CPU 结构

每个 CPU 有独立的 `softnet_data` 实例，按缓存行对齐以避免 false sharing：

```c
// net/core/dev.c:408
DEFINE_PER_CPU_ALIGNED(struct softnet_data, softnet_data);
```

`softnet_data` 的核心字段：

| 字段 | 说明 |
|------|------|
| `poll_list` | 待 poll 的 NAPI 链表（`net_rx_action` 遍历此链表） |
| `input_pkt_queue` | RPS 跨 CPU 转发的 skb 入队队列 |
| `process_queue` | `process_backlog` 正在处理的 skb 队列 |
| `backlog` | 内置的伪 NAPI（poll = `process_backlog`），用于处理 RPS 转发过来的包 |

#### poll_list 上的多 napi 共存

多队列网卡的多个 `napi_struct` 可以同时出现在**同一个 CPU** 的 `poll_list` 上（当多个队列的 IRQ 绑到同一 CPU 时）。`net_rx_action` 的循环逐个调用每个 napi 的 `poll()`，公平共享 budget。

**但同一个 napi 绝不会同时出现在两个 CPU 的 poll_list 上**——`napi_schedule_prep()` 的 CAS（`cmpxchg` on `NAPI_STATE_SCHED`）保证一旦某 napi 被调度，后续对它的 `napi_schedule_prep()` 返回 `false`。

```
   多队列网卡，队列0和队列2的IRQ都绑到CPU-0：

   CPU-0 的 softnet_data.poll_list:
      napi_0 (RX-queue-0) → napi_2 (RX-queue-2) → ...

   net_rx_action 循环：
      poll(napi_0) → poll(napi_2) → ...  // 逐个调用，共享 budget
```

### 1.4 SKB 分配

驱动在 poll 回调中从 ring buffer 取数据，需要分配 `sk_buff`：

```c
// net/core/skbuff.c:553  (__napi_alloc_skb)
// include/linux/skbuff.h:3136 (napi_alloc_skb 包装)
static inline struct sk_buff *napi_alloc_skb(struct napi_struct *napi,
                         unsigned int length)
{
    return __napi_alloc_skb(napi, length, GFP_ATOMIC);
}
```

`__napi_alloc_skb` 使用 per-CPU 的 `napi_alloc_cache` 和 page_frag 分配器，避免全局锁竞争。

### 1.5 硬中断与软中断的 CPU 亲和性

**规则：硬中断在哪个 CPU 上触发，软中断就在哪个 CPU 上执行。** 原因是 Linux 的软中断是 per-CPU 的，从源码看：

**第一步**：硬中断中 `____napi_schedule` 设置的是**当前 CPU** 的 pending 位：

```c
// net/core/dev.c:4670
list_add_tail(&napi->poll_list, &sd->poll_list);  // sd = this_cpu_ptr
__raise_softirq_irqoff(NET_RX_SOFTIRQ);           // 只标记本CPU的pending
```

**第二步**：硬中断返回时 `irq_exit()` 检查的也是**当前 CPU**：

```c
// kernel/softirq.c:626
static inline void __irq_exit_rcu(void)
{
    preempt_count_sub(HARDIRQ_OFFSET);
    if (!in_interrupt() && local_softirq_pending())  // 检查本CPU
        invoke_softirq();                             // 在本CPU直接执行
}
```

`invoke_softirq()` → `__do_softirq()` → `net_rx_action()`，全程在同一个 CPU 上：

```
              CPU-3（网卡中断绑定的CPU）
         ┌────────────────────────────────────┐
         │  硬中断上下文                        │
         │    驱动ISR → napi_schedule()         │
         │    → __raise_softirq_irqoff() [本CPU] │
         │                                      │
         │  irq_exit()                          │
         │    → local_softirq_pending() == true │
         │    → invoke_softirq()                │
         │                                      │
         │  软中断上下文（还在CPU-3！）           │
         │    → __do_softirq()                  │
         │      → net_rx_action()               │
         │        → 驱动poll() → ip_rcv → ...   │
         └────────────────────────────────────┘
```

如果 `__do_softirq` 超时（`MAX_SOFTIRQ_TIME` = 2ms）或重启次数超限（`MAX_SOFTIRQ_RESTART` = 10），则唤醒本 CPU 的 `ksoftirqd/N` 内核线程接管，**仍然在同一个 CPU 上执行**。唯一的跨 CPU 例外是 RPS 通过 IPI 显式触发远程 CPU 的软中断（参见 2.3 节）。

## 二、网络核心层：GRO + RPS + 协议分发

### 2.1 `NET_RX_SOFTIRQ` 处理函数：`net_rx_action`

软中断在系统初始化时注册：

```c
// net/core/dev.c:12265-12266
open_softirq(NET_TX_SOFTIRQ, net_tx_action);
open_softirq(NET_RX_SOFTIRQ, net_rx_action);
```

`net_rx_action` 从 `poll_list` 中取出 NAPI，逐个调用驱动的 poll 函数：

```c
// net/core/dev.c:7714
static __latent_entropy void net_rx_action(struct softirq_action *h)
{
    struct softnet_data *sd = this_cpu_ptr(&softnet_data);
    unsigned long time_limit = jiffies +
        usecs_to_jiffies(READ_ONCE(netdev_budget_usecs));  // 默认 2ms
    int budget = READ_ONCE(netdev_budget);                  // 默认 300

    // 取出所有待处理的NAPI
    list_splice_init(&sd->poll_list, &list);

    for (;;) {
        struct napi_struct *n;
        if (list_empty(&list)) { /* ... */ break; }
        n = list_first_entry(&list, struct napi_struct, poll_list);
        budget -= napi_poll(n, &repoll);  // 调用驱动的poll函数

        // 两个退出条件：budget用完 或 时间超限
        if (unlikely(budget <= 0 ||
                 time_after_eq(jiffies, time_limit))) {
            sd->time_squeeze++;
            break;
        }
    }

    // 还有未处理的NAPI，再次触发软中断
    if (!list_empty(&sd->poll_list))
        __raise_softirq_irqoff(NET_RX_SOFTIRQ);
}
```

`napi_poll()` → `__napi_poll()` 调用驱动的 poll：

```c
// net/core/dev.c:7549
static int __napi_poll(struct napi_struct *n, bool *repoll)
{
    int work, weight;
    weight = n->weight;           // 通常 64
    work = n->poll(n, weight);    // ★ 调用驱动poll，如e1000_clean
    // ...
}
```

### 2.2 GRO（Generic Receive Offload）

驱动通常调用 `napi_gro_receive()` 而不是直接 `netif_receive_skb()`。GRO 将**连续的同流小包合并成大包**，减少协议栈处理次数：

```c
// net/core/dev.c:6707
gro_result_t napi_gro_receive(struct napi_struct *napi, struct sk_buff *skb)
{
    // ...
    ret = napi_skb_finish(napi, skb, dev_gro_receive(napi, skb));
    // ...
}
```

GRO 流程：

```
napi_gro_receive()
    ↓
dev_gro_receive()     ← 在GRO哈希表中查找可合并的包
    ↓ 合并完成
napi_gro_complete()   ← 完成合并
    ↓
gro_normal_one() / gro_normal_list()
    ↓
netif_receive_skb_list_internal()  ← 送入标准接收路径
```

### 2.3 RPS / RFS 与 IRQ Affinity 的区别

**全称**：

| 缩写 | 全称 | 含义 |
|------|------|------|
| **RPS** | **Receive Packet Steering** | 接收包导向：软件层面将包分发到不同 CPU |
| **RFS** | **Receive Flow Steering** | 接收流导向：在 RPS 基础上将流导向处理该流的应用所在 CPU |
| **IRQ Affinity** | 中断亲和性 | 硬件层面将网卡中断绑定到指定 CPU |

#### 快递分拣类比

把网络包接收想象成快递站分拣：快递站（网卡）收到大量快递（包），需要分给多个快递员（CPU）处理。

**IRQ Affinity — 快递站有多个分拣口，硬件直接分流**：

```
          快递站（网卡多队列）
         /      |       \
      分拣口1  分拣口2  分拣口3      ← 网卡硬件直接分 3 条通道（3 个 RX 队列）
        |       |        |
      快递员A  快递员B  快递员C      ← 每个 CPU 只处理自己通道的包，零额外开销
```

**RPS — 只有一个分拣口，人工按单号哈希二次分拣**：

```
          快递站（网卡单队列）
               |
            分拣口                   ← 只有 1 个 RX 队列，全部涌入 CPU-0
               |
         按单号哈希二次分拣           ← RPS: skb_get_hash → enqueue_to_backlog
        /      |       \
      快递员A  快递员B  快递员C      ← 软件跨 CPU 转发，有 IPI 开销
```

**RFS — 二次分拣时还看收件人在哪个区域**：

```
          快递站（网卡单队列）
               |
            分拣口
               |
         智能分拣                    ← RFS: 不只看哈希，还查 rps_sock_flow_table
               |                        "这个流的 recvmsg 最后在 CPU-B 上调用"
            快递员B                  ← 包直接送到应用线程所在 CPU，减少 cache miss
```

#### 技术对比

| 维度 | IRQ Affinity（硬中断绑核） | RPS（软件分发） | RFS（流亲和） |
|------|---------------------------|----------------|--------------|
| **层级** | 硬件/驱动层 | 软中断层（`netif_receive_skb_internal`） | 软中断层 + 应用层 |
| **配置** | `/proc/irq/<N>/smp_affinity` | `rps_cpus` | `rps_sock_flow_entries` + `rps_flow_cnt` |
| **原理** | 网卡 RX 队列中断绑到指定 CPU | 按五元组哈希选目标 CPU，`enqueue_to_backlog` 转发 | 在 RPS 基础上查 `recvmsg` 最后执行的 CPU，将包路由到应用所在 CPU |
| **适用** | 多队列网卡（RSS） | 单队列/虚拟网卡（virtio） | 延迟敏感，减少 CPU 间 cache 迁移 |
| **开销** | 零额外开销 | 有 IPI + 跨 CPU 入队 | 同 RPS，但 cache 局部性更好 |

#### 选择策略（一句话）

| 场景 | 方案 |
|------|------|
| 多队列网卡 | **IRQ Affinity**（或网卡自带 RSS）即可，不需要 RPS/RFS |
| 单队列/虚拟网卡 | 启用 **RPS**，将负载分散到多个 CPU |
| 单队列 + 延迟敏感 | **RPS + RFS**，让包在应用所在 CPU 处理 |

RPS 将不同流的包分散到多个 CPU 处理，打破单 CPU 瓶颈：

```c
// net/core/dev.c:6013-6027  (netif_receive_skb_internal 中)
#ifdef CONFIG_RPS
    if (static_branch_unlikely(&rps_needed)) {
        int cpu = get_rps_cpu(skb->dev, skb, &rflow);  // 按流哈希选CPU
        if (cpu >= 0) {
            ret = enqueue_to_backlog(skb, cpu, &rflow->last_qtail);
            return ret;
        }
    }
#endif
    ret = __netif_receive_skb(skb);  // 无RPS，本CPU直接处理
```

#### RPS 的两次软中断机制

开启 RPS 后，一个包的接收会经历**两次 `NET_RX_SOFTIRQ` 软中断**，分别在不同 CPU 上执行，各自负责不同阶段的工作：

```
CPU-0（网卡中断所在CPU）                       CPU-2（RPS选中的目标CPU）
─────────────────────────                     ─────────────────────────
① 硬中断 → napi_schedule()
         ↓
② 第1次软中断: net_rx_action()
   → 驱动 poll() 从 Ring Buffer 取包
   → napi_gro_receive() → GRO 合并
   → netif_receive_skb_internal()
   → get_rps_cpu() → 选中 CPU-2
   → enqueue_to_backlog(skb, cpu=2)
     ├─ skb 入 CPU-2 的 input_pkt_queue
     └─ rps_ipi_queued() → 挂到 rps_ipi_list
         ↓
③ net_rx_action 结尾:
   net_rps_action_and_irq_enable()
   → net_rps_send_ipi()
     → smp_call_function_single_async()       ──→ IPI 中断到达 CPU-2
                                                        ↓
                                              ④ rps_trigger_softirq() [硬中断上下文]
                                                 → ____napi_schedule(sd, &sd->backlog)
                                                 → raise NET_RX_SOFTIRQ
                                                        ↓
                                              ⑤ 第2次软中断: net_rx_action()
                                                 → process_backlog()
                                                   → __netif_receive_skb(skb)
                                                     → ip_rcv → tcp_v4_rcv → ...
```

**两次软中断的分工**：

| | CPU-0 的软中断（第 1 次） | CPU-2 的软中断（第 2 次） |
|---|---|---|
| **poll 函数** | 驱动的 `napi_poll`（如 `e1000_clean_rx_irq`） | `process_backlog` |
| **数据来源** | 网卡 Ring Buffer（DMA 硬件地址） | `softnet_data.input_pkt_queue`（内存队列） |
| **做什么** | 从网卡取包 → GRO → 按哈希选 CPU → 入队 | 从队列取 skb → 协议栈处理（`ip_rcv` → `tcp_v4_rcv`） |
| **NAPI 对象** | 驱动注册的 `napi_struct` | 每个 CPU 内置的 `sd->backlog`（伪 NAPI） |

**源码调用链**：

`enqueue_to_backlog` 将 skb 放入目标 CPU 的 `softnet_data.input_pkt_queue`，并通过 IPI 间接触发目标 CPU 的 backlog NAPI：

```c
// net/core/dev.c:4986-5003  enqueue_to_backlog 核心逻辑
if (qlen <= netdev_max_backlog && !skb_flow_limit(skb, qlen)) {
    if (qlen) {                          // 队列非空，直接追加
        __skb_queue_tail(&sd->input_pkt_queue, skb);
        return NET_RX_SUCCESS;
    }
    // 队列为空 → 需要调度 backlog NAPI
    if (!__test_and_set_bit(NAPI_STATE_SCHED, &sd->backlog.state)) {
        if (!rps_ipi_queued(sd))         // 跨 CPU？加入 IPI 列表
            ____napi_schedule(sd, &sd->backlog);  // 同 CPU？直接调度
    }
    goto enqueue;
}
```

```c
// net/core/dev.c:4914-4928  rps_ipi_queued：跨 CPU 时不直接调度，而是记录到 IPI 列表
static int rps_ipi_queued(struct softnet_data *sd)
{
    struct softnet_data *mysd = this_cpu_ptr(&softnet_data);
    if (sd != mysd) {                    // 目标 CPU 不是当前 CPU
        sd->rps_ipi_next = mysd->rps_ipi_list;
        mysd->rps_ipi_list = sd;         // 挂到本 CPU 的 IPI 待发列表
        __raise_softirq_irqoff(NET_RX_SOFTIRQ);  // 确保本 CPU 软中断会执行
        return 1;                        // 返回 1 → 跳过 ____napi_schedule
    }
    return 0;
}
```

```c
// net/core/dev.c:4898-4903  IPI 到达目标 CPU 后的回调
static void rps_trigger_softirq(void *data)  // 在 IPI 硬中断上下文执行
{
    struct softnet_data *sd = data;
    ____napi_schedule(sd, &sd->backlog);     // 调度 backlog NAPI → 触发第二次软中断
    sd->received_rps++;
}
```

目标 CPU 上的 `process_backlog`（`net/core/dev.c:6923`）作为 `sd->backlog` 的 poll 函数，从 `process_queue` 取出 skb，调用 `__netif_receive_skb` 进入正常协议分发。

**为什么要两次软中断？** CPU-0 只负责"收包 + 分发"（从 Ring Buffer 取数据，做 GRO，按哈希转发），将 IP/TCP 协议栈处理的 CPU 开销转移到 CPU-2，打破单 CPU 瓶颈。不开 RPS 时只有一次软中断，CPU-0 从头做到尾。

```c
// net/ipv4/ip_input.c:504
static struct sk_buff *ip_rcv_core(struct sk_buff *skb, struct net *net)
{
    const struct iphdr *iph;

    // 丢弃非本机包（混杂模式收到的）
    if (skb->pkt_type == PACKET_OTHERHOST)
        goto drop;

    // 确保skb线性区包含完整IP头
    if (!pskb_may_pull(skb, sizeof(struct iphdr)))
        goto inhdr_error;

    if (iph->ihl < 5 || iph->version != 4)
        goto inhdr_error;
    if (unlikely(ip_fast_csum((u8 *)iph, iph->ihl)))
        goto csum_error;
    // ... 长度检查、设置transport_header ...
}
```

### 3.3 `ip_rcv_finish_core`：Early Demux + 路由查找

```c
// net/ipv4/ip_input.c:370
static int ip_rcv_finish_core(struct net *net, struct sock *sk,
                  struct sk_buff *skb, struct net_device *dev,
                  const struct sk_buff *hint)
{
    const struct iphdr *iph = ip_hdr(skb);

    // ★ Early Demux优化：在路由查找前先尝试匹配socket
    if (READ_ONCE(net->ipv4.sysctl_ip_early_demux) &&
        !skb_dst(skb) && !skb->sk && !ip_is_fragment(iph)) {
        switch (iph->protocol) {
        case IPPROTO_TCP:
            if (READ_ONCE(net->ipv4.sysctl_tcp_early_demux))
                tcp_v4_early_demux(skb);     // 提前匹配TCP socket
            break;
        case IPPROTO_UDP:
            if (READ_ONCE(net->ipv4.sysctl_udp_early_demux))
                udp_v4_early_demux(skb);     // 提前匹配UDP socket
            break;
        }
    }

    // 若early demux未命中，执行完整路由查找
    if (!skb_valid_dst(skb)) {
        err = ip_route_input_noref(skb, iph->daddr, iph->saddr,
                       iph->tos, dev);
    }
}
```

**Early Demux（提前解复用）的全称与含义**：

- **Demux** = **Demultiplexing**（解复用），即根据目标地址/端口从众多 socket 中找到正确的那一个
- **Early** 指在正常流程（路由查找）之前**提前**做这件事
- 合起来就是：**Early Demultiplexing — 在 IP 路由查找之前提前进行传输层 socket 匹配**

**价值**：如果提前找到目标 socket，可以复用该 socket 缓存的路由信息（`sk->sk_rx_dst`），跳过昂贵的 FIB 查找。对已建立连接的大量数据包（如 TCP ESTABLISHED 状态），路由结果几乎不变，Early Demux 可以显著减少每包的路由查找开销。

**控制开关**（sysctl）：
- `net.ipv4.ip_early_demux`：总开关（默认 1 开启）
- `net.ipv4.tcp_early_demux`：TCP Early Demux（默认 1）
- `net.ipv4.udp_early_demux`：UDP Early Demux（默认 1）

### 4.1 `tcp_v4_rcv`：TCP 总入口

```c
// net/ipv4/tcp_ipv4.c:2661
int tcp_v4_rcv(struct sk_buff *skb)
{
    // 1. 检查包类型
    if (skb->pkt_type != PACKET_HOST)
        goto discard_it;

    // 2. 确保skb包含完整TCP头
    if (!pskb_may_pull(skb, sizeof(struct tcphdr)))
        goto discard_it;
    th = (const struct tcphdr *)skb->data;
    if (unlikely(th->doff < sizeof(struct tcphdr) / 4))
        goto bad_packet;

    // 3. 校验和初始化
    if (skb_checksum_init(skb, IPPROTO_TCP, inet_compute_pseudo))
        goto csum_error;

    // 4. ★ 在全局哈希表中查找目标socket
    //    根据 <源IP, 源端口, 目的IP, 目的端口> 四元组查找
    //    先查ehash（已建立连接），再查监听表
    sk = __inet_lookup_skb(&tcp_hashinfo, skb, __tcp_hdrlen(th),
                   th->source, th->dest, sdif, &refcounted);
    if (!sk)
        goto no_tcp_socket;

    // 5. 按socket状态分发
    if (sk->sk_state == TCP_TIME_WAIT)
        goto do_time_wait;
    if (sk->sk_state == TCP_NEW_SYN_RECV)
        /* ... 三次握手最后一个ACK ... */

    // 6. ★ 关键分发逻辑
    bh_lock_sock_nested(sk);
    tcp_segs_in(tcp_sk(sk), skb);
    if (!sock_owned_by_user(sk)) {
        // socket未被用户锁定：直接处理
        ret = tcp_v4_do_rcv(sk, skb);
    } else {
        // socket被用户锁定（如正在recv）：放入backlog
        if (tcp_add_backlog(sk, skb))
            goto discard_and_relse;
    }
    bh_unlock_sock(sk);
}
```

**并发设计**：`bh_lock_sock` / `sock_owned_by_user` 是 TCP 接收路径最关键的并发机制。软中断处理包时若用户进程正持有 socket 锁（比如 `recv()` 中），数据包暂存到 `sk_backlog` 队列；用户进程 `release_sock()` 时统一处理积压包。

### 4.3 `tcp_rcv_established`：Header Prediction 快速路径

```c
// net/ipv4/tcp_input.c:6934
void tcp_rcv_established(struct sock *sk, struct sk_buff *skb)
{
    // Header Prediction 快速路径条件：
    // 1. 标志位匹配预期（纯ACK+数据，无SYN/FIN/RST/URG）
    // 2. 序列号 == 期望的下一个序列号（按序到达）
    // 3. ACK序列号合法
    if ((tcp_flag_word(th) & TCP_HP_BITS) == tp->pred_flags &&
        TCP_SKB_CB(skb)->seq == tp->rcv_nxt &&
        !after(TCP_SKB_CB(skb)->ack_seq, tp->snd_nxt)) {

        // ★ 快速路径：跳过大量校验，直接入队
        if (len <= tcp_header_len) {
            // 纯ACK → tcp_ack处理
        } else {
            // 数据包 → 直接入队
            __skb_pull(skb, tcp_header_len);
            eaten = tcp_queue_rcv(sk, skb, &fragstolen);
            tcp_event_data_recv(sk, skb);
            tcp_data_ready(sk);   // 唤醒等待的用户进程
            return;
        }
    }

slow_path:

    tcp_data_queue(sk, skb);
    tcp_data_snd_check(sk);
    tcp_ack_snd_check(sk);
}
```

### 4.5 `tcp_queue_rcv`：实际入队操作

```c
// net/ipv4/tcp_input.c:5928
static int __must_check tcp_queue_rcv(struct sock *sk, struct sk_buff *skb,
                      bool *fragstolen)
{
    int eaten;
    struct sk_buff *tail = skb_peek_tail(&sk->sk_receive_queue);

    // 尝试合并到队列尾部（减少skb数量，提升性能）
    eaten = (tail &&
         tcp_try_coalesce(sk, tail, skb, fragstolen)) ? 1 : 0;

    // 更新期望接收的下一个序列号
    tcp_rcv_nxt_update(tcp_sk(sk), TCP_SKB_CB(skb)->end_seq);

    if (!eaten) {
        // 无法合并，添加到接收队列尾部
        __skb_queue_tail(&sk->sk_receive_queue, skb);
        skb_set_owner_r(skb, sk);   // 记入socket内存计费
    }
    return eaten;
}
```

```c
{
    // 1. 校验UDP头、长度、checksum

    // 2. 尝试使用early demux已找到的socket
    sk = skb_steal_sock(skb, &refcounted);
    if (sk) {
        ret = udp_unicast_rcv_skb(sk, skb, uh);
        return ret;
    }

    // 3. 广播/组播 → __udp4_lib_mcast_deliver
    if (rt->rt_flags & (RTCF_BROADCAST|RTCF_MULTICAST))
        return __udp4_lib_mcast_deliver(net, skb, uh, ...);

    // 4. 单播查找socket
    sk = __udp4_lib_lookup_skb(skb, uh->source, uh->dest, udptable);
    if (sk)
        return udp_unicast_rcv_skb(sk, skb, uh);

    icmp_send(skb, ICMP_DEST_UNREACH, ICMP_PORT_UNREACH, 0);
```

### 6.1 `sock_def_readable`：唤醒阻塞进程

```c
{
```

### 6.3 TCP 读取：`tcp_recvmsg`

```c
// net/ipv4/tcp.c:3057
int tcp_recvmsg(struct sock *sk, struct msghdr *msg, size_t len, ...)
{
    // busy poll 优化
    if (sk_can_busy_loop(sk) &&
        skb_queue_empty_lockless(&sk->sk_receive_queue) &&
        sk->sk_state == TCP_ESTABLISHED)
        sk_busy_loop(sk, nonblock);

    lock_sock(sk);
    ret = tcp_recvmsg_locked(sk, msg, len, ...);
    release_sock(sk);  // ★ 释放锁时处理 backlog 中的积压包
}
```

`tcp_recvmsg_locked` 核心循环：

```c
// net/ipv4/tcp.c:2822-3015
do {
    // 遍历 sk_receive_queue 中的 skb
    skb_queue_walk(&sk->sk_receive_queue, skb) {
        // ... 计算offset ...
        if (offset < skb->len)
            goto found_ok_skb;
    }

    // 队列为空且阻塞模式 → 睡眠等待
    sk_wait_data(sk, &timeo, last);
    continue;

found_ok_skb:
    used = min(skb->len - offset, len);

    // ★ 将数据从内核skb拷贝到用户空间
    err = skb_copy_datagram_msg(skb, offset, msg, used);

    *seq += used;   // 更新已读序列号 (tp->copied_seq)
    copied += used;
    len -= used;

    // 释放已读完的skb
    if (!(flags & MSG_PEEK))
        sk_eat_skb(sk, skb);
} while (len > 0);

tcp_cleanup_rbuf(sk, copied);  // 可能触发窗口更新ACK
```

## 九、关键设计思想总结

| 设计 | 机制 | 目的 |
|------|------|------|
| **硬中断+NAPI分离** | 硬中断只做 `napi_schedule`，poll 在软中断批量处理 | 避免长时间关中断，高负载时自动轮询 |
| **GRO** | `napi_gro_receive` 合并同流小包 | 减少协议栈穿越次数 |
| **RPS/RFS** (Receive Packet/Flow Steering) | `get_rps_cpu` 按流哈希分发到多CPU；RFS 进一步导向应用所在 CPU | 打破单CPU接收瓶颈，减少 cache miss |
| **Early Demux** (提前解复用) | `tcp_v4_early_demux` 在路由前匹配socket | 跳过FIB查找，复用缓存路由 |
| **Header Prediction** | `tcp_rcv_established` 快速路径 | 常见按序数据包跳过大量校验 |
| **Socket Lock+Backlog** | `sock_owned_by_user` + `tcp_add_backlog` | 软中断与用户进程并发安全 |
| **sk_data_ready回调** | `sock_def_readable` → `wake_up` | 数据到达时唤醒阻塞的recv/epoll |
| **零拷贝优化** | `skb_copy_datagram_msg` / mmap / splice | 减少内核→用户空间数据拷贝 |

| ⑦ 用户态 | epoll/select/poll | 多路复用，高效等待多 socket |
| ⑦ 用户态 | copy_to_user | 内核→用户空间数据拷贝 |
| 贯穿所有层 | sk_buff | 统一包缓冲区，零拷贝传递 |
| 贯穿所有层 | per-CPU + RCU | 高并发无锁读取 |
| 贯穿所有层 | Network Namespace | 容器级完整网络栈隔离 |

## 十一、VRF 与 Network Namespace 在收包路径中的位置

### 11.1 Network Namespace — 贯穿所有层的隔离容器

Network Namespace 不属于协议栈的某一层，它是**贯穿所有层**的隔离容器。每个 namespace 拥有独立的设备列表、路由表、netfilter 规则、socket 集合。

```c
// include/net/net_namespace.h:56
struct net {
    struct list_head    dev_base_head;     // 该 ns 的所有网络设备
    struct net_device   *loopback_dev;     // 独立的 loopback
    struct netns_ipv4   ipv4;              // 独立的 IPv4 配置（路由、sysctl等）
    struct netns_ipv6   ipv6;              // 独立的 IPv6 配置
    struct netns_nf     nf;                // 独立的 netfilter 规则
    // ...
};
```

每个网络设备通过 `dev->nd_net` 绑定到一个 namespace：

```c
// include/linux/netdevice.h:2463
struct net *dev_net(const struct net_device *dev) {
    return read_pnet(&dev->nd_net);
}
```

**创建方式**：

| 方式 | 系统调用 | 说明 |
|------|---------|------|
| `clone(CLONE_NEWNET)` | `clone` | 创建子进程时带新 namespace（Docker 用此方式） |
| `unshare(CLONE_NEWNET)` | `unshare` | 当前进程脱离原 namespace |
| `ip netns add <name>` | 用户态 | 底层调用 `unshare` + bind mount 到 `/var/run/netns/` |

底层统一走 `copy_net_ns()`（`net/core/net_namespace.c:446`），当 `flags & CLONE_NEWNET` 时分配新 `struct net` 并调用 `setup_net()` 初始化所有子系统。系统启动时内核创建全局 `init_net`，所有进程默认使用它。

### 11.2 VRF — IP 层（L3）的路由隔离

VRF（Virtual Routing and Forwarding）是一个虚拟网络设备（`drivers/net/vrf.c`），工作在 **IP 层**。它在**同一个 namespace 内**提供多个独立的路由表域。

VRF 通过 `l3mdev_ops` 接口介入 IP 收发路径：

```c
// include/net/l3mdev.h:35
struct l3mdev_ops {
    u32  (*l3mdev_fib_table)(const struct net_device *dev);  // 指定使用哪个路由表
    struct sk_buff *(*l3mdev_l3_rcv)(...);   // L3 接收钩子
    struct sk_buff *(*l3mdev_l3_out)(...);   // L3 发送钩子
};
```

在接收路径中的介入点在 `ip_rcv_finish`：

```c
// net/ipv4/ip_input.c:489
skb = l3mdev_ip_rcv(skb);  // VRF 接管：路由查找指向 VRF 绑定的路由表
```

**创建方式**：通过 Netlink `RTM_NEWLINK` 创建，与创建 bridge、veth 一样：

```bash
ip link add vrf-red type vrf table 100    # 创建 VRF，绑定路由表 100
ip link set eth0 master vrf-red           # 将物理设备加入 VRF
```

底层走 `vrf_newlink()`（`drivers/net/vrf.c:1740`），VRF 模块在 `vrf_init_module()` 中注册 `rtnl_link_ops`，但不自动创建任何设备。

### 11.3 两者层级对比

```
┌──────────────────────────────────────────────────────┐
│               Network Namespace (struct net)          │  ← 隔离一切
│                                                       │
│  ┌───────────────────────────────────────────────┐   │
│  │            L2: 网络设备（eth0, veth...）         │   │
│  └───────────────────────────────────────────────┘   │
│  ┌───────────────────────────────────────────────┐   │
│  │            L3: IP 层                            │   │
│  │   ┌─────────────┐  ┌─────────────┐            │   │
│  │   │  VRF-red    │  │  VRF-blue   │ ← 同ns内   │   │
│  │   │ 路由表 100   │  │ 路由表 200   │   不同路由域│   │
│  │   └─────────────┘  └─────────────┘            │   │
│  └───────────────────────────────────────────────┘   │
│  ┌───────────────────────────────────────────────┐   │
│  │            L4: TCP/UDP                          │   │
│  └───────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────┘
```

| 维度 | Network Namespace | VRF |
|------|-------------------|-----|
| **隔离粒度** | 整个网络栈（设备/路由/iptables/socket） | 仅路由表（L3） |
| **层级** | 贯穿所有层的容器 | IP 层的路由域 |
| **类比** | 整栋独立的大楼 | 同一大楼内不同楼层的路由表 |
| **创建方式** | `clone`/`unshare` + `CLONE_NEWNET` | `ip link add type vrf table N` |
| **典型场景** | 容器隔离（Docker/K8s） | 多租户路由、运营商 VPN |
