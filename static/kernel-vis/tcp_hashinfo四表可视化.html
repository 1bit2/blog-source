① ehash — 已建立连接查找表

按 四元组（本地 IP:端口 + 远端 IP:端口）哈希。数据包到达时，O(1) 找到对应的 socket。存 struct sock（活跃连接）和 inet_timewait_sock（TIME_WAIT 轻量 socket），两者都通过 sock_common 的 skc_nulls_node 链入。
桶数组：*ehash 动态分配
桶数：2 的幂，由 ehash_mask+1 决定
桶类型：inet_ehash_bucket
桶字段：仅 nulls_head
哈希函数：inet_ehashfn()（Jenkins）
锁：*ehash_locks 分离锁数组，多桶共享一锁
为什么用 nulls 而不是普通 NULL 结尾？ RCU 无锁遍历时，socket 可能在中途被迁移到另一个桶。nulls 尾值编码了桶号，遍历完检查"尾桶号 == 当前桶号"可发现这种迁移并触发重查。源码：include/net/inet_hashtables.h:191。
源码定位：桶类型 inet_ehash_bucket（inet_hashtables.h:38-40），数组字段 *ehash（inet_hashtables.h:301），分离锁 *ehash_locks（inet_hashtables.h:303）。

② bhash — 端口绑定哈希表

按 本地端口哈希。双层链表：外层桶挂 inet_bind_bucket（每端口一 bucket），bucket 内的 owners 挂所有使用该端口的 socket。用于 bind() / connect() 时的端口冲突检查。
桶数组：*bhash 动态分配
桶数：2 的幂，由 bhash_size 决定
桶类型：inet_bind_hashbucket
桶字段：spinlock + chain
哈希函数：inet_bhashfn(net, port, size)
中间层：inet_bind_bucket slab 分配
为什么需要中间层 inet_bind_bucket？ 同一端口可能被多个 socket 共用（REUSEADDR / REUSEPORT / TIME_WAIT 复用）。如果直接把 sock 链在桶里，端口冲突检测要遍历全表；用 bucket 做"端口代理"后，查端口 = O(1)，桶内只挂使用该端口的 sock。
源码定位：桶类型 inet_bind_hashbucket（inet_hashtables.h:181-184），中间层 inet_bind_bucket（inet_hashtables.h:156-171），bucket 内 owners 链（inet_hashtables.h:170），sock 用 sk_bind_node 链入。

③ lhash2 — 监听 socket 二级哈希表（IP + 端口）

按 本地 IP + 端口 哈希，是 listening_hash 的升级版。解决"大量虚拟主机同端口"时旧表查找慢的问题。收 SYN 时先查精确 IP 桶，再查 INADDR_ANY 桶。
桶数组：*lhash2 动态分配
桶数：2 的幂，由 lhash2_mask+1 决定
桶类型：inet_listen_hashbucket
桶字段：lock + count + head
链入字段：icsk_listen_portaddr_node
初始化：inet_hashinfo2_init()
lhash2 vs listening_hash：同端口不同 IP 的 socket 在 listening_hash 里挤在同一桶，需要遍历比对 IP；在 lhash2 里分散在不同桶，O(1) 直达。虚拟主机场景性能差 10x+。
源码定位：桶类型 inet_listen_hashbucket（inet_hashtables.h:192-199），链入字段 icsk_listen_portaddr_node（inet_connection_sock.h:106），查找函数 inet_lhash2_lookup()（inet_hashtables.c:278）。

④ listening_hash — 监听 socket 一级哈希表（仅端口）

按 本地端口 哈希，固定 32 桶内联在 inet_hashinfo 结构体中。旧机制，与 lhash2 共存（兼容 + 双保险）。同一端口不同 IP 的监听 socket 都链在同一桶，靠遍历区分 IP。
桶数组：listening_hash[32] 内联
桶数：固定 INET_LHTABLE_SIZE=32
桶类型：inet_listen_hashbucket
桶字段：lock + count + nulls_head
链入字段：sk_node（sock_common.skc_node）
尾标记：NULLS(LISTENING_NULLS_BASE + 桶号)
为什么固定 32 桶就够了？ 源码注释 "Yes, really, this is all you need."（inet_hashtables.h:202）。监听 socket 数量远少于已建立连接，32 桶 + 遍历已经足够。lhash2 是为虚拟主机 / 高密度 listen 场景引入的优化。
源码定位：数组声明 listening_hash[INET_LHTABLE_SIZE]（inet_hashtables.h:335-339），INET_LHTABLE_SIZE = 32（inet_hashtables.h:202），链入字段 sk_node（sock.h），尾标记 LISTENING_NULLS_BASE = 1 << 29（inet_hashtables.h:191）。

回答你的判断：
  • ✓ ehash 是数组 + 链表，精确地说是"数组 + nulls 链表"（尾节点是编码了桶号的哨兵，不是 NULL，用于 RCU 无锁遍历检测节点迁移）。
  • ✓ bhash 是数组 + 链表 + 链表节点里还有链表，精确地说：外层数组的每个桶含 spinlock + chain；chain 挂 inet_bind_bucket 链（按 tb->node）；每个 bucket 内还有 owners hlist，挂所有使用该端口的 sock（按 sk_bind_node）。双层链表是 bhash 区别于其他 3 张表的关键特征。
